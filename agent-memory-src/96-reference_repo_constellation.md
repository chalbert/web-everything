The web-platform work spans four sibling repos under `~/workspace/`, layered standard → impl → platform → product.

**All of them are local working dirs I can build, test, and gate from ANY session** — `webeverything`, `frontierui`, and `plateau-app` are in the additional-working-directories list with deps installed; each has its own `build` / `test:unit` / `test:e2e` / `check:standards` scripts. So **"I can't build/verify the other repo from here" is never a valid caveat** — probe first (`cd ../frontierui && npm run …`). Cross-repo work like the #170/#449 plugs dedup (porting WE fixes into FUI, backfilling FUI tests) is fully verifiable end-to-end. Don't assert a capability limit untested — same rule as [[feedback_test_before_asserting_cause]], applied to *what I can do*, not just *what's wrong*.

The four repos:

- **Web Everything** (`webeverything`) — the **canonical standard**. The repo IS the spec (`src/_data/*.json` + `.njk`). Owns terminology, protocols, intents. Contracts only, no shipping impl. See [[feedback_catalog_auto_render]], [[feedback_protocol_first_class]].
- **Frontier UI** (`frontierui`) — the **reference implementation** of WE standards. Ships concrete code (e.g. the actual `FullCommitmentPolicy` that WE only contracts). Points back to WE as canonical.
- **Plateau** (`plateau`) — **LEGACY, ABANDONED. New implementation work NEVER lands here (it goes to Frontier UI).** Per the author: plateau was the *initial single-repo version* of the whole system, since **migrated/split into WE + Frontier UI + plateau-app**. **Nuance (clarified 2026-06-08):** you *may* read plateau as a **non-authoritative reference** when nothing more recent exists — a behavioral sketch to adapt — but it is **not a model**: it predates current decisions and may contradict them, so anything taken from it must be **re-derived against the live spec/conventions and explicitly called out as plateau-sourced**, never copied wholesale (esp. its architecture). Codified in `webeverything/AGENTS.md` → "Mental model" (2026-06-08). Its `definitions/*.md` are mostly empty stubs. Last commit 2026-01-31; it also has a pile of *uncommitted* 2026-06 experiments (Anchor.ts, `custom-elements/*`, auto-complete-demo, an `e2e/` dir) that were done in the wrong repo — **none of it is to be migrated** (author's standing instruction). The canonical platform layer is **`webeverything/plugs/`** (plugs = the platform primitives, owned by WE; AGENTS.md/docs never say "plateau"). **Goal: this repo is never opened again.** Its uncovered concepts were harvested once, on 2026-06-07, into backlog **#173 menu/menubar, #174 tooltip, #175 slider/range, #176 segmented-control+compound-children, #177 form block, #178 access-control/authorization** — all fresh design-first candidates, not migrations. ⚠️ Backlog items that say "plateau's DOM patches" or `graduatedTo: "plateau: …"` point at this legacy repo, not `plugs/`. **Corrected 2026-06-07:** the trait/block surfaces whose *spec* exists in WE but had **no reference impl** in Frontier UI/`plugs/` were **reopened** as fresh builds (#122 #136 #137 #138 #145 #148 #149); #167 was **retargeted** plateau→`plugs/` and reframed as a verify task (the live plugs already implement the autonomous-CE lifecycle). **Still resolved & untouched:** #156 #160 #162 #168 — their concerns (autonomous-CE lifecycle, insertAdjacentElement patch, e2e harness) are **already covered in the live `plugs/`**, so they're left resolved, but their `graduatedTo` still names plateau (cosmetic rot — fix only if revisited).
- **plateau-app** (`plateau-app`) — the **active SaaS product UI** ("Enterprise Front-End Platform Management"), the new Plateau UI. Tiny (~10KB glue). **Cardinal rule (its CLAUDE.md): NO new UI components — compose WE/FrontierUI blocks only.** Imports the runtime via path aliases `@we/plugs/*`, `@we/blocks/*` — **does NOT duplicate plugs**. Tracks gaps in `MISSING_BLOCKS.md`.

**Plugs duplication hazard (2026-06-07):** `frontierui/plugs/` is a **vendored copy** of `webeverything/plugs/` (not an `@we/plugs` import). 53/53 WE plug files have a frontierui counterpart; **40 byte-identical, 13 already drifted** — a fix in one repo silently rots the other. Tracked in backlog #170 (consolidate to single-source; recommend frontierui import `@we/plugs/*` like plateau-app does). The narrower Spec-Explorer dev-panel Vite plugin is duplicated the same way ([[project_dev_panel_plugin_duplicated]]).

Chain (current, post-migration): **WE (contract) → Frontier UI (reference build) → plateau-app (product)**. The legacy `plateau` repo is **not** a node in this chain anymore — it's the abandoned ancestor they all came from.

⚠️ **CORRECTED 2026-06-14 by [[#606]] (ratified):** the plugs runtime is **implementation**, not a standard artifact, so it is **owned by Frontier UI as `@frontierui/plugs`** (a granular sub-package in the FUI impl umbrella — the `@lit/reactive-element` precedent), **not** "inside WE (`webeverything/plugs/`)" as older notes in this file (and AGENTS.md/architecture.md) say. WE keeps **contracts only** and *consumes* `@frontierui/plugs` as a client in its demos (the #604 WE→FUI seam). Invariants from the ruling: every plug must have a non-invasive **unplugged** form (the real-app surface); the global-patching **plugged** mode is POC/demo only; every plug ships **dual-mode tests**. The #170/#449 consolidation direction is **reversed** accordingly (FUI canonical, WE deletes its `plugs/`). The "platform layer inside WE" / "#170 recommend frontierui import `@we/plugs`" / "plateau-app imports `@we/plugs/*`" statements above are the pre-#606 model and are being migrated — treat them as stale.

Key fact for intent work: plateau-app already CONSUMES an intent today — it hardcodes a `loaderIntent` and injects it via `documentInjector.set('customContexts:loaderIntent', loaderIntent)` in `plateau-app/src/main.ts`. That injector key (`customContexts:{id}Intent`) is the consumption point any intent-configuration tool would write to.
