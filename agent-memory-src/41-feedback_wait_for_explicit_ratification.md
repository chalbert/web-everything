A governing principle the user states **mid-discussion** ("WE should be only protocol, contracts — cannot hold code") is **input to the decision, not the ratification of it.** Do NOT collapse "here's the principle" into "ratify and ship it." On #779 (2026-06-17) the user stated the contracts-only invariant; I immediately wrote the ruling, flipped `resolved`, committed, scaffolded a follow-up (#893), and updated downstream items — the user's next message was "I did not say ratify."

**Why:** decision-mode is present → discuss → red-team → **and only on an explicit "ratify / go / make the call" do I write the ruling + resolve.** The user is often still working through the substance (e.g. "still trying to make sense — explain exactly what X does") even after handing me a principle. A principle narrows the option space; it doesn't close the item. Closing prematurely forces a noisy rollback (reopen item, revert downstream edits, delete spawned follow-ups, undo the commit) and erodes the discussion-not-multiple-choice contract.

**How to apply:** keep the decision item `active` and record the stance as a **"Proposed ruling — PENDING RATIFICATION"** section (not a `## Ruling`, not `resolved`). Do NOT flip `status: resolved`, stamp `dateResolved`, commit, or spawn consequence items until the user explicitly says to ratify. When unsure whether a message is the go signal, ask "ratify this as the call?" rather than assuming. The rollback itself is cheap only if nothing was committed — so hold the commit until ratification too. **Reinforced 2026-06-19 (#1135) — a non-go reply is not a go, and external-service choices are a hard-approval class.** On #1135 (pick the public-site host + gate) the user said "go" to *select* the item from the shortlist, then "ready" to have me *present* it — I read "ready" as ratification and resolved + committed in one turn. The user pushed: "wait did I ratify?" then "choosing a service seems like something I should approve." Two lessons: (1) selecting an item and being ready to *hear* the decision are **not** ratification utterances — only an unambiguous "ratify/approved/go ahead and make the call" is; (2) decisions that **commit to an external service/vendor, incur recurring cost, or create accounts/credentials** are a class that must **never** be agent-auto-resolved no matter how strong the recommendation — always stage as pending and wait for the owner's explicit approval. Reverting was: `git reset --soft HEAD~1`, flip `resolved → active`, drop `dateResolved`/`codifiedIn`, re-header to PENDING.

Hold the commit, the resolve, AND the spin-off scaffolding behind the gate.
