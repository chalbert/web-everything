User-confirmed stance (2026-06-21): upcoming/in-flight web platform standards efforts — **Open UI CG, WHATWG, W3C/CSSWG** — are upstream collaborators WE wants to *align with and consume*, NEVER competitors.

**Why:** WE is native-first ([[feedback_native_first_default]]); its deliverable is a contracts/conformance/adapter layer *above* native primitives, not a rival primitive. As these bodies land native features (Popover, customizable `<select>`, anchor positioning, invokers), they shrink the surface WE must standardize — that's the goal, not a threat. WE's altitude (intents, protocols, capability matrices, adapters, polyglot reach) sits above where Open UI operates (single-control anatomy → native HTML/CSS), so they're complementary by construction.

**How to apply:** When comparing WE to a standards body, frame it as upstream/complementary and cite their vocabulary (the aria-sort / Intl.Collator pattern) rather than re-inventing anatomy they've canonicalized. The only real risk is WE duplicating their work. Already wired in: #1401 (OpenUI proposal-diff discovery lens under program #1399), #541 (verify 96 benchmark capabilities vs open-ui.org), #1257 (platform-standards-watch). Distinguish from W&B's "OpenUI" (wandb/openui, an AI UI-generator) — unrelated to the standards group.
