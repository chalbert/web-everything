#1467 ruling (ratified 2026-06-21, → b, codified via [[project_platform_decisions_statute_layer]] #constellation-placement): when deciding whether a runtime symbol stays in WE, split the conformance role precisely.

- **AMENDED 2026-06-22 by #1566** (ratified): WE keeps only the **declarative contract** — the verifier *interface* + the vector/golden **corpus** + the golden **schema**. The verifier **implementation** (e.g. `auditDataTable`) + the conformance **run** move to **Plateau** (a neutral non-implementer; home of #427 dashboard / #1577 explorer). Judging is executable and WE holds zero executable (#1282); the cross-impl-neutrality reason the verifier couldn't be FUI is satisfied by Plateau, not WE. The original "verifier reads output as DATA → stays WE" reading is overturned.
- The **renderer / runnable backend** (config→DOM impl) is the **subject under test** → FUI — one *target* among many, reached via a per-target **binding** it owns. (Unchanged.)
- **Still-WE carve-out:** conformance tooling a WE-side `check.ts` gate runs over WE's **own declarative artifacts** (manifests; corpus completeness/schema-validity) stays WE — it checks WE-owned *data*, not an implementation's runtime output.

**Why:** #817's carve-out ("runtime stays WE iff a WE-side conformance gate consumes it") is easy to over-read — "a WE test imports it" is NOT the test, else every FUI impl WE exercises would migrate into WE. #1566 tightened it further: even the verifier IMPL is executable, so WE keeps only the contract (interface + vectors + schema); the impl + run are a neutral Plateau product, and the impl-under-test is a FUI target via a binding.

**INTENT-conformance framing (#947, 2026-06-23, codified [[project_platform_decisions_statute_layer]] #intent-conformance-is-block-compliance):** an **intent is an interface; blocks implement it** (indirectly, via the build-time `intentProfileResolver`). "Conformance of an intent" = does the implementing block satisfy the intent *contract* — **NOT** a runtime DOM gate, and faking a tie is forbidden. Splits: **static/contract** (declares required `intentDimension`s / bundles required traits) → checkable in WE build-time (resolver / `requirementValidator` shape); **behavioral** (rendered component behaves per intent) → contract→WE, runner/verifier→Plateau, subject→FUI (same split as above). Never smuggle a conformance run into a trait *consumer* (e.g. docs chrome). Surfacing an intent for inspection (the profile already lives in the DOM as `data-intent-*`) is read-only dev-tooling, distinct from *checking* conformance.

**DELETE-TAIL COROLLARY (2026-06-24, #1772/#1776):** a "FUI-host demo + swap WE page to #701 iframe + delete the WE runtime copy" tail is NOT a trivial delete. Before deleting any `we:blocks/.../<runtime>.ts`, grep its importers — the WE **unit tests** usually still **execute** that runtime to audit each fixture. `check:standards` doesn't run them but `npm run verify` does (`vitest run`), so the delete breaks verify. Ratified fix is verifier-golden-FIRST: redesign the audit into a **stored-golden data-reader** (freeze DOM/source-shape into a `*-goldens.{ts,json}` under the block's `__fixtures__/`, validated by `we:blocks/renderers/golden-schema.ts`) so WE verifies vectors as DATA — precedent #1356 (pagination), #1494 (data-table). THEN delete. Caveat: goldens fit DOM/source-shape audits; **behavioral** WE tests (events/timers/retry, e.g. #1774 handoff) can't be goldened → just delete the WE test (FUI runs the vectors now). Decide per-tail at pickup.

**How to apply:** when placing conformance code, sort by *what it is*: declarative contract (interface + vector/golden corpus + schema) → WE; verifier **implementation** + the **run** → Plateau (neutral); the impl-under-test + its **binding** → the implementer (FUI ships one). Only conformance tooling that checks WE's own *declarative artifacts* via a `check.ts` gate stays WE. The prepared #1467 default got the renderer wrong (kept it in WE); #1566 then corrected #1467 itself (kept the verifier impl in WE).
