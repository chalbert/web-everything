# Serial-batch в†’ drain coordination: overlap-stack lanes, proof-of-land gate, push the handoff

A serial `/batch` today works each item in an isolated lane clone that `reset --hard`s to `origin/main`, so N serial items become **N siblings off the same base**. The deferred drain then lands them one-by-one: it merges #1, main moves, and #2..#N each need a rebase вЂ” every overlapping file becomes a **real conflict the drain resolves blind**, re-discovering conflicts the single authoring session already understood and threw away. The user's complaint: *"no point delivering 10 serial PRs that then need to be drained one by one with conflicts merged between each"* вЂ” plus a want for **push, not poll**, between sessions.

**The design (red-team-converged):** a serial batch authors an **overlap-stacked** lane chain вЂ” only items that actually share files stack (lane N+1 based on its predecessor's tip); provably-disjoint items stay siblings off `origin/main`. It lands through the **unchanged** deferred drain, made safe by a **positive, identity-based proof-of-land gate**. Basing a lane on its predecessor's tip converts blind drain-time conflicts into authoring-time three-way-clean merges: the existing rebase-drop-manifest step already commits the lane tip as the merge's 2nd parent, so a landed parent is an ancestor of main and git auto-computes it as the merge-base вЂ” the child lands clean, no `--merge-base` override. Sessions never message live; the drain stays a separate process (a genuine non-author) and the sole serial writer to main. "Push" honestly reduces to *write a durable marker + wake a single lander*.

## Fork decisions

**F1 вЂ” Stacked chain vs. siblings в†’ SELECTIVE overlap-driven stacking with *authoritative* (not predicted) file sets.** At acquire, derive each item's **declared** file-set from its backlog locus, union-find over declared sets per repo: an overlapping item bases on the open chain's frontier tip (extending it); a disjoint item stays a sibling; a *bridging* item that touches two previously-disjoint chains merges their tips in-session and records both `stackParents`. **Critical:** the base is chosen pre-work but files are known post-work, so at **push** the producer recomputes the *actual* touched set (`git diff --name-only base...head`) and asserts `actual вЉ† declared`; on any violation it does **not** ship a certified-disjoint sibling вЂ” it rebases onto the overlapping chain's frontier **in-session** (context hot), re-resolves, records `stackParents`, then pushes. A post-hoc overlap discovery can therefore never reach the deferred drain as a mislabelled sibling. Chain depth is capped (fall back to siblings past the cap) to bound cumulative CI cost. *Rejected:* full linearization (a broken middle strands the whole tail incl. independent items); pure siblings (the status quo); predicted-set stacking without the push-time re-check (reproduces today's blind conflict in the drain).

**F2 вЂ” In-session land vs. deferred в†’ STAY DEFERRED; serial does not fork the engine.** The producer never runs the lander. Serial and parallel share the one drain engine; the only serial delta is the lane *base* (overlap chain) and the manifest's `stackParents`/`base` fields. *Why:* both in-session-land failure modes are verified in-tree вЂ” the drain has **no lock** and computes `max+1` from a local view (so firing a drain per closing session в†’ concurrent numbering writers в†’ duplicate NNN), and the merge gate only string-checks the caller (so an author's own in-session drain becomes the merger for every non-escalated item = literal self-approval). Keeping the drain a separate process preserves both invariants with zero new self-approval surface.

**F3 вЂ” Manifest representation в†’ extend the existing per-item WE manifest; prove land by item identity.** Add `stackParents` (an `asItemId` list вЂ” the frontier tip item(s) the lane was cut from / merged onto) and a per-repo `base` SHA; keep `blockedBy` as the hard semantic edge. Record proof-of-land as **`bornAs: <hash>`** on the *landed* item's frontmatter вЂ” a durable, cross-clone, renumber-immune hashв†’landed record. **Critical:** `bornAs` is stamped inside the numbering routine and **excluded** from the ledger's blind whole-token hash-rewrite (a one-line guard), so `bornAs: xPARENT` survives on main as the birth hash instead of being clobbered to `bornAs: 500` вЂ” the adversary proved that *not* excluding it is a permanent-strand deadlock. **Single source of truth:** local `we:id-ledger.json` = numbering bookkeeping only; `bornAs`-on-main = the sole cross-clone landed proof; `bornAs` is derived one-directionally from the ledger assignment so they cannot diverge. All *other* hash cross-refs (`blockedBy`, body, `parent`) are still rewritten hashв†’NNN. *Rejected:* a batch-couple blob (loses per-item review/CI/numbering); ref-absence as the land signal (unreliable both directions); base-SHA-ancestry as the gate (broken by rebuild+renumber вЂ” proof must be by identity).

**F4 вЂ” Push transport в†’ detached lease-checked drain + two distinct locks + durable capability marker.** Happy path: the batch close fires **one detached** self-terminating drain watch (`--label=ready-to-merge --watch --until-batches-idle`), a separate process, never the authoring loop. **Two distinct locks:** (1) a fine **numbering-critical-section mutex** wrapping number+publish at every land call site (enforces "sole serial writer"); (2) a distinct **whole-process drain lease** (`O_EXCL` + TTL heartbeat) held for a drain's lifetime. Push-at-close checks the **lease** (not the ms-held mutex): free в‡’ fire the detached drain and take the lease; held в‡’ just enqueue (label) and the running drain's watch collects it вЂ” so exactly one drain runs regardless of how many sessions close. Stacking is gated on a **durable capability marker committed to `origin/main`**, read by the producer at authoring time, **defaulting hard to siblings** on any read failure / version mismatch вЂ” never on probing a not-yet-running drain. Exceptions (red CI, parked review) ride the existing durable `ready-to-merge`/`review:*` labels, swept by `/finish` or a periodic sweep. No new always-on daemon. *Rejected:* one lock for both roles (can't deliver "exactly one drain"); probing the deployed drain for capability (it runs later against a possibly-different main).

**F5 вЂ” Partial batch (item 5 of 10 red / review:changes) в†’ never blind-salvage; hard-defer only true overlap-descendants.** The positive proof-of-land gate defers **only** a broken link's true overlap-descendants (their tips literally contain the broken code, and CI red-poisons them anyway); disjoint siblings land normally. Recovery: repair in the **live** session while context is hot (preferred), else `/finish` rebuilds the descendant tail onto the repaired tip **guided by `stackParents`/`base`** вЂ” a fast-forward when the fix touched no shared file, exactly **one guided conflict** per directly-overlapping descendant (resolved *with* the manifest topology) when it did. A descendant is **never** landed past an unlanded parent: readiness requires every `stackParent` be landed-this-pass (in-memory set) **or** `bornAs`-proven on main вЂ” never ref-absence. *This is stowaway-proof:* the rebuild does a 3-way merge of the descendant's whole ancestry against main, and a descendant never conflicts with its ancestor's *absence* вЂ” so "salvage the tail without the broken parent" would silently drag the parent's unreviewed code onto main under the child's number. The gate makes that structurally impossible, at both the impl and WE level (the gate is couple-granular).

## Invariants preserved

- **#2219/#104 read-only primary** вЂ” `--base` still resets a *pool* clone (to a predecessor lane tip); the producer never splices to primary; main advances only when the drain merges a PR.
- **#2288/#2290 JIT numbering / drain is the sole serial writer** вЂ” *enforced* (not conventional) by the TTL'd numbering mutex at every land call site; the producer only enqueues+wakes; `stackParents` are `asItemId` so no session hand-assigns a number; proof-of-land is by identity (`bornAs`); the distinct whole-process lease guarantees exactly one drain.
- **#1869 conflict-free WE-lane merge / one carrier = WE** вЂ” `stackParents`+`base` are added fields on the existing WE-only manifest (still a one-sided add); the stacked WE manifest is a modify/delete the disposition already sheds as "manifest-only"; `bornAs` lives on the landed item's own file (no new carrier).
- **Non-author review** вЂ” the drain stays a *separate detached* process under the whole-process lease, never the authoring loop; the per-item review diff (`base...head`) keeps the reviewer seeing only this item's delta, not ancestors' code; `review:human` stays human-only.
- **Impl-first/WE-last couple order** вЂ” the `stackParents` gate is evaluated at couple granularity via the impl-PRв†’WE-manifest `laneRef` join, so a manifest-less impl PR inherits its couple's `stackParents` and is never independently "ready" ahead of its couple (closes the impl-orphan-always-ready hole).

## Migration path (this is the ship order)

Stacking **cannot** ship first: producer stacking without the proof gate lets the drain land a stacked child before its parent (merge-base collapses to pre-batch main, dragging the parent's code onto main under the child's number). The safe atomic unit is **prereqs + gate + manifest + proof, then stacking**, sequenced and enforced by the durable capability marker.

1. **Both locks** (numbering mutex + whole-process lease) вЂ” pure hardening; fixes a latent concurrent-numbering race that exists *today*.
2. **Hash-aware cascade** (`Number()`в†’`asItemId` in the plan-label-drain path) вЂ” pure bug fix; the cross-item `blockedBy` cascade is already broken for hash-keyed items under JIT numbering.
3. **Manifest `stackParents`+`base` fields** вЂ” backward-compatible (absent в‡’ sibling behavior).
4. **`bornAs` proof-of-land** вЂ” stamp at land, exclude from the rewrite, add the reader. Depends on step 1.
5. **Proof-gated stacked drain** вЂ” teach the sweep the impl-PRв†’WE-manifest join and the gate; **commit the durable capability marker** to main. Drain is now stacking-safe though nothing stacks yet. *Confirm 1вЂ“5 are on `origin/main` before any producer stacks.*
6. **Producer overlap-stacking** вЂ” `--base`, union-find on declared locus, push-time `actualвЉ†declared` + in-session rebase, depth cap. Stacks **only** when the marker off main advertises gate support.
7. **Per-item review diff** вЂ” review quality on stacked PRs; no correctness dependency.
8. **Push-at-close** (detached, lease-checked, bounded lifetime) **+ `/finish` stack-repair** + docs. Each is additive; the system stays correct with none of them (deferred drain + durable labels are the backstop).

## Open risks (honest residuals)

- A `review:changes` non-ff repush of a stacked link still yields **exactly one guided conflict per directly-overlapping descendant** (resolved with topology + context). The irreducible residual вЂ” strictly better than today's blind conflict on every overlapping pair, but not zero.
- Red CI on a stacked parent red-poisons its true overlap-descendants; that sub-chain's throughput waits for the fix (correct, but a real latency cost). Disjoint items are unaffected.
- Stacking CI cost is real: each stacked child runs CI over the cumulative stack; a parent repush re-triggers descendants' CI. Mitigated by the depth cap.
- The detached push-at-close process must be **truly detached** to outlive the closing session; if the harness kills session children it silently degrades to the deferred sweep (still correct, not "push"). Needs a platform-verified detached/scheduled launch + bounded lifetime.
- Both locks must be TTL'd/reclaimable вЂ” too-short races, too-long wedges the queue on a crash. Deliberate lock-lifetime + stale-reclaim choice needed.
- The capability marker must be live on main before any producer stacks, and the read must default hard to siblings on any doubt (conservative by construction).
- The impl-PRв†’WE-manifest join assumes each impl PR's headRef appears in exactly one WE manifest's `repos[]`; a cheap validation guard (every stacked couple's impl refs resolve) is worth adding.
