# Conformance-lit MVP — how does the extension detect WE-conformance of a live page?

The buried fork at the heart of [#1656](/backlog/1656-conformance-lit-extension-funnel-mvp-chrome-devtools-panel-t/). **No design exists yet** — this brings the call to Definition of Ready. How does the dev-browser extension decide a running page is "WE-conformant" so its tooling lights up? Greenfield: no live-page probe exists today. The forks are grounded in the [live-page WE-conformance detection](/research/live-page-we-conformance-detection/) research topic (report via `relatedReport`): it surveyed how framework devtools detect a stack on a live page and mapped the declared-vs-probe-vs-verify trade space onto WE's machinery. Recommended default in **bold**: **probe-first, tiered** — probe to light up, declared manifest the opt-in upgrade, verify on-demand only.

## Axis-framing

The concern decomposes into one detection-mechanism axis with three candidate mechanisms, plus a sub-fork the skeptic surfaced about *what the tiers gate*. Each mechanism maps onto concrete WE code that fixes what it can read on a live page today:

- **(a) DECLARED** — read the app's shipped `CapabilityManifest`. The interface is [we:capability-manifest/provider.ts:139](../capability-manifest/provider.ts#L139) (`specVersion` / `conformanceLevel` L0–L2 / `features`), shipped as a static `export const manifest` ([we:capability-manifest/provider.ts:129](../capability-manifest/provider.ts#L129), `MANIFEST_EXPORT_NAME`). It is a **build-time convention, not runtime-queryable** — and no implementation ships one yet ([we:capability-manifest/check.ts:107](../capability-manifest/check.ts#L107), `IMPLEMENTATION_MANIFESTS` is empty). Decisively, the declared `conformanceLevel` is **self-asserted and merely trusted** — `assertCapabilityManifest` schema-validates the shape and never verifies behaviour ([we:capability-manifest/provider.ts:225](../capability-manifest/provider.ts#L225)); the only integrity check is that an L1+ claim lists the Core feature ids ([we:capability-manifest/provider.ts:232](../capability-manifest/provider.ts#L232)).
- **(b) PROBED** — inspect the live runtime for WE signals with no app cooperation. Real, non-mutating signals exist today: [we:plugs/webregistries/declarativeRegistry.ts:39](../plugs/webregistries/declarativeRegistry.ts#L39) (`REGISTRY_SCRIPT_TYPE` → `document.querySelectorAll('script[type="registry"]')`), [we:plugs/webregistries/declarativeRegistry.ts:279](../plugs/webregistries/declarativeRegistry.ts#L279) (`SCOPED_REGISTRY_KEY` symbol on host elements), [we:plugs/webregistries/declarativeRegistry.ts:292](../plugs/webregistries/declarativeRegistry.ts#L292) (`getActiveRegistryResult()`). The webexpressions binding layer ([we:plugs/webexpressions/index.ts:75](../plugs/webexpressions/index.ts#L75), `applyPatches()`) is **injector-internal with no public global marker** — a known limit on probe richness.
- **(c) VERIFIED** — run the conformance vector corpus ([we:conformance-vectors/schema.ts:53](../conformance-vectors/schema.ts#L53), `ConformanceVector`) via a runner ([we:wrapper-conformance/runner.ts:62](../wrapper-conformance/runner.ts#L62), `runVector()`). It is authoritative but **mutates the live page** — the runner creates DOM containers, renders the subject, and dispatches events ([we:wrapper-conformance/runner.ts:64](../wrapper-conformance/runner.ts#L64)).

## Recommended path at a glance

| Fork | Recommended default | Main alternative | Confidence |
|---|---|---|---|
| Fork 1 — detection mechanism | **(b) PROBE to light up, tiered: DECLARED as opt-in upgrade, VERIFY on-demand** | (a) DECLARED-first gate | High |
| Fork 2 — what the tiers escalate | **The conformance *assertion* (present → self-declared level → verified level), not the toolset** | Tiers gate which tools appear | Med-high |

## Fork 1 — the live-page detection mechanism

*Fork-existence justification:* a real either/or — the three mechanisms cannot coexist as *the default passive activation gate*, because two are structurally disqualified from that seat: VERIFY (c) **mutates the inspected page** ([we:wrapper-conformance/runner.ts:64](../wrapper-conformance/runner.ts#L64)), so it can never silently decide whether to light up a stranger's running app; and a DECLARED-only (a) gate reads a manifest no app ships at runtime today ([we:capability-manifest/check.ts:107](../capability-manifest/check.ts#L107)) and is itself only build-time inspectable. Exactly one mechanism can be the always-on light-up gate; the other two compose around it.

- **(b) PROBE to light up — recommended default.** The extension reads the WE runtime signals that exist with no app cooperation ([we:plugs/webregistries/declarativeRegistry.ts:39](../plugs/webregistries/declarativeRegistry.ts#L39) / [:279](../plugs/webregistries/declarativeRegistry.ts#L279) / [:292](../plugs/webregistries/declarativeRegistry.ts#L292); injector-chain providers). **Merit:** widest reach (zero cooperation) — the universal pattern every shipping framework devtool uses (React `__REACT_DEVTOOLS_GLOBAL_HOOK__` + `__reactFiber$`, Vue `__VUE__`/`el.__vue_app__`, Angular `ng-version`); non-mutating, so safe on any live page; and the only mechanism that can light up on real pages at all today. Composes the other two as tiers: DECLARED as an authoritative opt-in upgrade, VERIFY behind an explicit inspector action.
- **(a) DECLARED-first gate** — *Rejected (as the default gate).* **Merit:** a manifest carries an explicit `conformanceLevel` the probe can only infer — genuinely richer *when present*. But the level is **self-asserted and merely trusted**, never behaviour-verified ([we:capability-manifest/provider.ts:225](../capability-manifest/provider.ts#L225), [:232](../capability-manifest/provider.ts#L232)) — so on the honesty axis it is no stronger than a probe's presence signal, just a claim the *app* authored about itself rather than one the *tool* observed. For a stranger-facing first impression, a tool-observed inference is the more trustworthy of two unverified signals. Retained as the **opt-in upgrade tier**, not discarded.
- **(c) VERIFY-to-light** — *Rejected (as the default gate).* **Merit:** the only mechanism that makes the behavioural conformance claim *truthfully* (it exercises the contract via [we:conformance-vectors/schema.ts:53](../conformance-vectors/schema.ts#L53)). But it **mutates the inspected page** ([we:wrapper-conformance/runner.ts:64](../wrapper-conformance/runner.ts#L64)) — disqualifying for a passive always-on gate — and is slow. Retained as the **on-demand inspector action**, the only tier permitted to display a *verified* conformance result.

*Skeptic:* SURVIVES-WITH-AMENDMENT (dedicated sub-agent). Its hardest attack — *"only DECLARED honestly reports a conformance level, so probe-first makes the funnel's 'WE-conformant' headline a false-positive lie"* — was traced to the tree and **collapsed**: DECLARED's level is self-asserted-and-trusted ([we:capability-manifest/provider.ts:225](../capability-manifest/provider.ts#L225)/[:232](../capability-manifest/provider.ts#L232)), no more verified than a probe signal, and VERIFY is structurally barred from the passive-gate seat by page mutation — so the attack argues *against* the branch it wanted to flip to. The amendment it forced is real and is folded into Fork 2: **probe must light up under a "Web Everything detected" (presence) label, never a "WE-conformant" claim** — conformance is a verified word, presence is what probe knows; this dissolves the false-positive objection and the React-devtool false-analogy (probe then claims exactly what React-devtools claims: present). Accepted residual the default beats by *bounding*: webexpressions-only / non-scoped-registry apps are probe-invisible today (a false-negative class), an accepted MVP gap whose unblock is a separate public-probe-marker item, not a reason to flip the default.

## Fork 2 — what the tiers escalate (the skeptic's amendment)

*Fork-existence justification:* a genuine either/or the skeptic exposed inside the tiering — if all three tiers light up the *same* tooling, "probe-first" either lies about certainty (probe asserts what only verify earns) or, if probe lights up *less*, ships a deliberately degraded first impression. The two branches cannot both hold; the tiering is only coherent under one of them.

- **(a) Tiers escalate the conformance *assertion*, not the toolset — recommended default.** All tiers may surface the same inspector tooling; what escalates across probe → declared → verify is the **strength of the conformance statement the panel is allowed to print**: probe ⇒ "Web Everything detected" (presence only); declared ⇒ the `conformanceLevel`, badged **self-declared / unverified**; verify (on-demand) ⇒ the **only** tier that prints a *verified* conformance result (it alone runs [we:conformance-vectors/schema.ts:53](../conformance-vectors/schema.ts#L53)). **Merit:** keeps every printed claim backed by its evidence — the honest mapping of word to mechanism — while preserving widest reach.
- **(b) Tiers gate which tools appear (probe = degraded toolset)** — *Rejected.* **Merit:** simple to reason about. But it makes "probe-first" a degraded first impression (the funnel's whole job is to *show off* on any conformant-looking app), and it still doesn't fix the honesty gap — a lit tool implies a claim the probe can't back. Gating the *assertion* rather than the *toolset* is what makes probe-first truthful and impressive at once.

*Skeptic:* this fork *is* the skeptic's surfaced amendment (same dedicated-agent pass), so it arrives already attacked — the SURVIVES-WITH-AMENDMENT verdict on Fork 1 is its origin. The default here (gate the assertion, not the toolset) is precisely the amendment that neutralizes the false-positive / false-analogy / cosmetic-tiering attacks; no separate refutation remained once it was adopted.
