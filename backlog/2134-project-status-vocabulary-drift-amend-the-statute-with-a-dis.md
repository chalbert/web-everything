# Project-status vocabulary drift: amend the statute with a distinct PROJECT_LIFECYCLE

The project `status` axis is named by three different vocabularies — the data convention, the public five-stage ladder, and the descriptor `LIFECYCLE` enum — and the statute *deliberately* keeps project status outside `LIFECYCLE`, so closing the drift is a statute amendment, not a plain drift fix. Surfaced by the #2088 grounding digest and explicitly reserved by the `#portfolio-project-tiering` lineage as "a separate statute-amendment decision". Two forks: which vocabulary is canonical for the project axis, and whether the gate enforces it. Research topic: [`project-lifecycle-vocabulary`](/research/project-lifecycle-vocabulary/).

## Grounding digest

- **Three vocabularies name one axis.** (1) *Data:* every `we:src/_data/projects/<id>.json` carries `status`: **concept 17 / poc 24 / draft 4** across the 45 entries — all values drawn from {concept, poc, draft}. (2) *Public ladder:* [we:src/project-lifecycle.njk](src/project-lifecycle.njk):11-69 promises five stages — concept (10%) → poc (30%) → draft (50%) → candidate (80%) → stable (100%) — each with consumer-facing semantics ("The API is frozen", "Safe for mission-critical use"). (3) *Descriptor enum:* `LIFECYCLE = {concept, draft, experimental, active}` at [we:scripts/check-standards-rules.mjs](scripts/check-standards-rules.mjs):789 — no `poc`, no `candidate`; and `STATUS_SYNONYMS` (line 790) deprecates `stable` → `active`, so the public ladder's terminal stage is a *deprecated spelling* on the descriptor axis.
- **The separation is statute-deliberate, so closing it is an amendment.** The placement-test rule 6 at `we:docs/agent/platform-decisions.md:136-143`: "`poc` is the convention — project `status` is not enum-validated; the `LIFECYCLE` set governs *descriptors*, not projects." Reaffirmed by the `#portfolio-project-tiering` lineage (`we:docs/agent/platform-decisions.md:470-472`): the rule "scopes the *status* axis and is untouched — the status-vocabulary drift is a separate statute-amendment decision" (this item). Echoed in code comments at [we:scripts/check-standards.mjs](scripts/check-standards.mjs):184.
- **The data is clean; the drift is definitional.** All 45 stamped values already sit inside the public ladder, so the drift lives between the three *definitions*, not in the entries — the ladder-matching amendment has a zero-migration path (enum + statute text move; no data file changes).
- **The literal `concept` is load-bearing beyond render.** D3-readiness demotes open items whose `relatedProject` is `status: 'concept'` with no shipped surface ([we:src/_data/backlog.js](src/_data/backlog.js):114-120; surfaced at [we:scripts/check-readiness.mjs](scripts/check-readiness.mjs):235), and statute rule 6's own prose instructs "bump it **off** `concept` (`poc` is the convention)". Any vocabulary that renames `concept`/`poc` breaks the backlog engine and the statute text, not just labels.
- **Render is axis-agnostic and shared across both vocabularies.** Home tiles render `status` → header `we-badge` via the SSR `weProjectGrid` ([we:.eleventy.js](.eleventy.js):307-315); detail pages render a `status-{{ status }}` meter + "Status:" label via the `projectStatus` macro ([we:src/_includes/project-status.njk](src/_includes/project-status.njk):2-3) — and that *same macro* is reused for descriptor statuses across the descriptor pages — [we:src/plug-pages.njk](src/plug-pages.njk):31, [we:src/state-pages.njk](src/state-pages.njk):25, plus block/adapter/intent/demo pages and the conformance/demos listings. Hence [we:src/css/style.css](src/css/style.css):962-970 defines meter fills for the union of both vocabularies (concept/poc/draft/candidate/stable *and* experimental/active) — plus two junk classes, `status-implemented` (:967, a deprecated-synonym spelling) and `status-open` (:970, in neither vocabulary). A distinct project enum needs **zero render changes**; the union is correct-by-design for the shared macro, with `candidate`/`stable` fills already provisioned (:966, :968); the junk classes go to the follow-up CSS audit.
- **A fourth spelling exists in a dead file.** The orphaned pre-#1157 root spec `we:src/_data/webhandlers.json:5` says `"status": "speculative"` — outside every vocabulary; no template consumes the root `webhandlers`/`webportals` globals (they survive only in the lane-attribution list, `we:scripts/readiness/lane-partition.mjs:45`, and root `we:src/_data/webportals.json` is a stale duplicate of the split `we:src/_data/projects/webportals.json`, missing `tier`). Cleanup is a follow-up build, not a fork — but it is exactly what an unvalidated convention produces.
- **In-repo precedent: the sibling project axis validates day one.** Project `tier` is enum-validated (`PROJECT_TIERS` + `validateProjectTier`, [we:scripts/check-standards-rules.mjs](scripts/check-standards-rules.mjs):856-884; gate wired at [we:scripts/check-standards.mjs](scripts/check-standards.mjs):183-188) — #2088 Fork 3 chose "validated from day one (unlike `status` — grounding-digest drift)". `checkStatus` is per-kind and pure ([we:scripts/check-standards-rules.mjs](scripts/check-standards-rules.mjs):822-850), so a project-axis check composes beside it without touching the descriptor axis. Descriptor semantics are implementation-flavored: the enum's own header names it the "Implementation-lifecycle vocabulary" ([we:scripts/check-standards-rules.mjs](scripts/check-standards-rules.mjs):784), `implemented` is a synonym of `active` (rules:790), and a `status: active` Block without `implementedBy` warns ([we:scripts/check-standards.mjs](scripts/check-standards.mjs):172 — a Block-scoped warn, the coupling's only enforcement) — "active" reads as *implemented*, a meaning a zero-impl WE project (#1282) cannot take.
- **External prior art — one normative home per label set, per track.** W3C maturity levels (WD → CR → PR → REC) are defined once in the Process Document; TC39 stages 0–4 live in its process doc with a validated proposals table; the Node.js stability index (0 deprecated / 1 experimental / 2 stable / 3 legacy) is machine-readable doc metadata. Distinct tracks keep **distinct ladders** (W3C does not reuse TC39's stages; Node's index is not npm's). The counter-model is npm dist-tags — publisher-declared free-form labels, the one surveyed scheme with no closed set — which is exactly where ecosystem drift lives. Design systems (USWDS, Polaris, Carbon) each keep a small closed component-lifecycle enum enforced by their docs tooling. Pattern: **declared value + one closed normative definition + mechanical validation**; drift arises where the label set lives in prose, data, and tooling separately — WE's current shape.

## The axes + classification

Two live choices: **(1) vocabulary** — which label set is canonical for the project axis; **(2) enforcement** — whether `check:standards` validates it. Classification pass: this is **website-app / build-infra + portfolio-governance** turf (the "WE = standard AND website-app" disambiguation — catalog-app side); no block/intent/protocol is being shaped, so Q1–Q3/Q5/Q7 don't fire. Q4: one catalog + one gate can carry only one canonical vocabulary per axis — a rival label set is not a per-consumer knob, so no config dimension fires (same verdict as #2088 Fork 1). Q6's spirit survives as: the enum constrains *our own data files* only — an internal editorial vocabulary, never a standard imposed on consumers (mandate-nothing untouched). Statute-amendment mechanics (rewriting rule 6's parenthetical, adding the anchor) are codification at ratification, not forks — the statute is **not** edited during prep.

|---|---|---|---|

## Fork 1 — canonical vocabulary: which label set owns the project axis

*Fork-existence:* one data field validated by one gate and rendered on one public catalog can have exactly one canonical label set — a value is valid or it isn't, so rival vocabularies cannot coexist (composability probe fails: there is no shared kernel two enums could both be facades over; the label set *is* the contract). The excluded-broken branch is the status quo — three definitions for one axis is the defect this item exists to close.

- **(a) a distinct `PROJECT_LIFECYCLE = {concept, poc, draft, candidate, stable}` — the public ladder, exported beside `LIFECYCLE` in `we:scripts/check-standards-rules.mjs`** *(default)* — zero migration (45/45 entries already conform); the load-bearing `concept` literal (backlog D3 at `we:src/_data/backlog.js:118`, statute rule 6 prose) untouched; the ladder is already the *published contract* (five stages with consumer-facing semantics at `we:src/project-lifecycle.njk:11-69`); and the descriptor semantics genuinely don't fit projects — the descriptor enum is self-described as the "Implementation-lifecycle vocabulary" (`we:scripts/check-standards-rules.mjs:784`; `implemented` ⇒ `active` synonym, with a Block-scoped warn coupling `active` to `implementedBy` at `we:scripts/check-standards.mjs:172`), a meaning a zero-impl WE project (#1282) cannot take: a project matures toward *spec stability* (`candidate` = API frozen, `stable` = SemVer), not toward "implemented". Two ordered enums for two different maturation processes is the surveyed-prior-art shape (W3C ladder ≠ TC39 stages ≠ Node stability index — separate tracks, separate ladders). *Sub-point — word reuse across axes:* `stable` becomes canonical-for-Project while staying a deprecated synonym for Block/Plug; acceptable because the checks are kind-keyed and the axes never share a field, but the amendment text must state the reuse explicitly so it reads as designed, not as fresh drift. **The split must also be mechanical, not comment-only (skeptic A1):** `checkStatus('Project', …)` runs without complaint today (`fileFor('Project')` resolves via `PER_ID_SPEC_DIR`, `we:scripts/check-standards-rules.mjs:810`) and carries a *deterministic reference autofix* `stable` → `active` (`we:scripts/check-standards-rules.mjs:835-840`) — one mis-wired loop away from silently rewriting a future `stable` project. The ratified contract therefore includes the invariant: **the descriptor axis's reference autofix must never be reachable for project data — enforced mechanically, not by comment.** (How — an assert/throw in `checkStatus` vs a pinned unit test — is an impl choice below the contract; see Supported-by-default.)
- (b) merge projects into the descriptor `LIFECYCLE` {concept, draft, experimental, active} — *Rejected:* forces a 24-file `poc` → `experimental` migration plus a rewrite/retirement of the public ladder page; dissolves the deliberate statutory separation (descriptors mature toward implementation, projects toward spec maturity — the axes measure different processes); and imports the `stable` → `active` synonym, so the catalog could never say the very word its public page promises as the terminal stage.
- (c) coin a fresh third label set (e.g. W3C-style WD/CR/REC spellings) — *Rejected (broken):* creates a fourth vocabulary; every existing surface (45 data files, the public page, the CSS classes, the D3 `concept` literal) drifts against it on day one — the disease offered as the cure.

Skeptic: SURVIVES-WITH-AMENDMENT → no classification re-route (composability probe honestly fails within the axis; #2088's close-out *sketch* of a distinct enum is a capture, not a ratified shape — naming-fork-precedent discipline); the `poc`-is-nonstandard and dual-enum-confusion attacks failed (internal editorial axis, the published page's own heading, 24/45 files + rule-6 prose; the overlap exists today — 1a documents it, rejecting 1a leaves it undocumented); the `active`⇒implemented evidence was corrected from "the `implementedBy` coupling" (a Block-scoped warn only) to the enum's own "Implementation-lifecycle vocabulary" self-description (rules:784) — non-load-bearing either way; amended to add **A1** — a mechanical kind-guard so the descriptor axis's deterministic `stable`→`active` autofix (`we:scripts/check-standards-rules.mjs:835-840`) can never touch a project file (`checkStatus('Project', …)` is silently well-formed today via `PER_ID_SPEC_DIR`, rules:810) — and **A4** — codification must also update the `#portfolio-project-tiering` lineage forward-reference (`we:docs/agent/platform-decisions.md:470-472`), which otherwise dangles as "…is untouched" the moment rule 6 is amended.
Screen: flagged(impl) → fix applied — the core fork (which label set is canonical) is consumer-observable and its merit survives with zero-migration stripped, but A1's original phrasing ratified the *mechanism* (`checkStatus` assert/throw vs pinned unit test — internal function behavior, the #2030 layer the item's own non-fork section disclaims); re-layered so the fork keeps only the invariant (the descriptor autofix is mechanically unreachable for project data) and the mechanism choice moved to Supported-by-default; the duplicated kind-guard comment in the Fork 2 snippet marked non-normative.

## Fork 2 — enforcement: does the gate validate the project axis

*Fork-existence:* the current statute text ("project `status` is **not** enum-validated") and an enum-validated field are mutually exclusive statute end-states — the amendment lands on exactly one. The broken branch is defining a canon (Fork 1) and then leaving it unenforced: a defined-but-unchecked label set is precisely what produced the stray `speculative` and three drifted definitions.

- **(a) enum-validate in `check:standards` — a `checkProjectStatus` beside `validateProjectTier`, error severity, day one, no synonym map, and missing/empty `status` is an ERROR** *(default)* — the data is already clean (verified: 45 files, concept 17 / poc 24 / draft 4), so the gate goes green with zero fixes; the on-point precedent over the *same files* sits in the same rules module (#2088 Fork 3: tier "validated from day one — unlike `status`", and `validateProjectTier` errors day one); no `STATUS_SYNONYMS`-style map ships because the project axis has no live deprecated spellings — a synonym entry is added only when a real drifted value is caught and canonicalized (the descriptor pattern, "flagged so drift can't return") — and with no synonym map there is no reference-autofix path, so the check can never rewrite data. **Missing-status semantics (skeptic A2):** `checkStatus` and `validateProjectTier` *disagree* here — `checkStatus` early-returns on `!status` (`we:scripts/check-standards-rules.mjs:831`) while `validateProjectTier` errors on a missing tier (rules:870-875); `checkProjectStatus` takes the tier side (required field), otherwise a new project file omitting `status` slips the very gate this fork adds and D3-readiness reads `undefined`.
- (b) keep convention-only — amend the statute merely to *name* the ladder as the convention — *Rejected:* leaves the exact enforcement gap that produced three drifted definitions; and yields an incoherent statute — "here is the project enum, deliberately unchecked" one anchor away from tier's day-one-validation rationale over the same JSON files. (Scope honesty, per skeptic: the orphaned `speculative` exhibit is *outside* the gate's reach either way — `loadDataRegistry('projects')` reads only `we:src/_data/projects/*.json` (`we:scripts/lib/registry-loader.cjs:31-37`), so that instance is closed by the follow-up deletion, not by this fork; what (a) prevents is the *next* stray inside `projects/`.)
- (c) warn-only — *Rejected:* the on-point mirror is `validateProjectTier` over the same project files, which errors day one, and clean data makes error severity free. (Precision note, per skeptic: "status rules never warn" is `checkStatus`'s own docstring (`we:scripts/check-standards-rules.mjs:824-825`), not a universal — the gate's `RESEARCH_STATUSES` check *warns* (`we:scripts/check-standards.mjs:224`) — so (c) is rejected on the tier mirror, not a house law.)

The concrete shape (the code-level contract this fork ratifies):

```js
// we:scripts/check-standards-rules.mjs — beside LIFECYCLE (line 789)
// Ordered concept → poc → draft → candidate → stable (the public ladder,
// we:src/project-lifecycle.njk). Distinct from the descriptor LIFECYCLE by statute:
// descriptors mature toward implementation ("active" ⇒ implementedBy), projects
// toward spec stability. NOTE: "stable" is canonical here while a deprecated
// synonym on the descriptor axis — deliberate per-axis vocabularies (kind-keyed).
export const PROJECT_LIFECYCLE = new Set(['concept', 'poc', 'draft', 'candidate', 'stable']);

/** Error entries, never warns. Missing/empty status is an ERROR (the validateProjectTier
 *  required-field shape, NOT checkStatus's `!status` early-return — rules:831 vs rules:870-875).
 *  No synonym map: the project axis has no live deprecated spellings (add one only on caught
 *  drift), so there is no reference-autofix path — this check can never rewrite data. */
export function checkProjectStatus(id, status) { /* … */ }
// (A1 invariant: the descriptor stable→active autofix is unreachable for project data —
//  mechanism per Supported-by-default, non-normative here.)
```

```js
// we:scripts/check-standards.mjs — wired beside the §3a tier check (line 183)
for (const p of projects)
  for (const e of checkProjectStatus(p.id, p.status)) err(e.message, e.descriptor);
```

Skeptic: SURVIVES-WITH-AMENDMENT → no re-route (validated-vs-not are mutually exclusive statute end-states; #2088 validated *tier* only); data-clean and gate-scope claims verified by command (the orphaned `speculative` is invisible to the gate — `loadDataRegistry('projects')` reads only `we:src/_data/projects/*.json`, `we:scripts/lib/registry-loader.cjs:30-36`); amended to **A2** (missing/empty `status` errors — the tier shape, not `checkStatus`'s early-return) and to soften the "never warn" rationale to the tier mirror (`RESEARCH_STATUSES` warns at `we:scripts/check-standards.mjs:224`, so never-warn is not a house law); **A3** folded into the codification checklist (stale gate comments).
Screen: clear — enforcement-vs-convention is an observable contract difference at every consumer seam (a drifted or missing status now *fails* the gate); everything the snippet decides is gate-observable behavior (error severity, no synonym map, required field) with placement/naming already demoted to the non-fork section; merit survives stripping "data is clean so it's free" (a defined-but-unchecked convention is what produced the drift). Codifier caution, not a flag: "day one" is timing language encoding "no grace period in the contract".

## Supported by default / not forks

- **Statute amendment mechanics** — rewrite rule 6's parenthetical (`we:docs/agent/platform-decisions.md:141`) to cite `PROJECT_LIFECYCLE`, add the anchor + `#portfolio-project-tiering` cross-ref, set `codifiedIn`: codification at ratification, never a prep-time edit (#1886). **Codification checklist also touches what the amendment falsifies (skeptic A3 + A4):** `validateProjectTier`'s "unlike the deliberately-loose project `status`" docstring (`we:scripts/check-standards-rules.mjs:859-861`), the §3a "stays deliberately outside LIFECYCLE" comment (`we:scripts/check-standards.mjs:184` — literally still true, but re-point it at `PROJECT_LIFECYCLE`), and the `#portfolio-project-tiering` lineage forward-reference "…is untouched — a separate statute-amendment decision" (`we:docs/agent/platform-decisions.md:470-472`), which must gain the #2134 anchor once rule 6 is amended.
- **Check placement / function naming / A1 mechanism** — impl detail below the observable gate contract (#2030); the snippet above is the shape, not a ratified file layout. The A1 invariant's *mechanism* lives here too (screen re-layer): `checkStatus` asserting/throwing on `kind === 'Project'` vs a unit test pinning that `Project` never reaches it are both acceptable — the fork ratifies only that the autofix is mechanically unreachable for project data.
- **The page↔enum relationship** — the rules-module enum is the single normative home; `we:src/project-lifecycle.njk` stays its editorial rendering. Auto-deriving the page from the enum is *not* ratified here (declared-over-auto-derived, #2074) — the gate guarantees data ⊆ enum, which is the drift that matters.
