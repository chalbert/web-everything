# Parallel orchestrator: exclude self-modifying items from concurrent lanes (or move lane tooling to an editable locus)

> **Related but orthogonal to epic #2289 (2026-07-05).** That epic concerns NNN *id allocation* (the drain-only + JIT-numbering prevention). This item concerns concurrent edits to the *running tooling* — a different failure mode. Kept independent; neither blocks the other.

> **RATIFIED 2026-07-10 — both forks resolved to their bold defaults.** Fork 1 → **(a)** declared `RUN_TOOLING` pathspec in the tested partition module, probe-time match pre-claim, three detection seats + a coverage-parity test (amended to walk *transitive* shell-outs). Fork 2 → **today (a)** probe-drop + solo/last serial `/batch`; **steady-state (c)** concurrent lane + drain landing (entailment of the #2138 rider), sunset capability-keyed on the #2153 drain. In-run serial lane (b) rejected as the observed wedge; frontmatter flag (b) and engine `batchable:false` (c) rejected in Fork 1. Codified as a rider under [#pr-flow-rollout-mechanism](../docs/agent/platform-decisions.md). Build arm filed as a successor item. Red-team at ratification: "defer entirely to #2138" fails (#2153 blocked on #2160, #2149 is an open self-modifier now — real interim window); coverage-parity test must walk transitive shell-outs (amendment folded).

No standing rule exists yet for **self-modifying items** — items whose touch-set includes the `/workflow` run's *own* executing tooling. The two forks below are grounded in a prior-art survey (research topic `self-modifying-run-tooling-exclusion`; session report linked via `relatedReport`), each with a recommended default in **bold**. The driving incident is closed (#2071/#2072/#2073 all landed later, serially) — this item sets the rule for the *next* self-modifying item, and there is one open right now (#2149 edits `we:scripts/readiness/lane-partition.mjs` + the inline mirror in the orchestrator file; its own text carves the orchestrator-script case to this item).

**Which layer:** agent-workflow / session-tooling (same plane as #2123/#2138) — nothing crosses the WE↔FUI standard boundary or touches a registry. Codifies as session house rules, not standard vocabulary.

## Grounding digest

- **The observed failure (the item's premise, verified in git):** run `batch-2026-07-01-1947-2071` TaskStop'd after "#2073's lane wedged editing the live orchestrator file (sandbox-locked)" — 5 of 17 items landed, the other 12 sat pre-claimed `active` and were reopened by hand (commit `34a26a39`; pre-claim `0b66e918`). #2071/#2072/#2073 all edit [we:.claude/skills/batch-backlog-items/parallel-execute.workflow.js](../.claude/skills/batch-backlog-items/parallel-execute.workflow.js) — the file the Workflow tool executes by path (the `scriptPath` at [we:.claude/skills/batch-backlog-items/SKILL.md:238](../.claude/skills/batch-backlog-items/SKILL.md)).
- **Why every in-run route writes the executing file today:** the serial lane works items "IN PLACE in the PRIMARY checkouts" ([we:parallel-execute.workflow.js:793](../.claude/skills/batch-backlog-items/parallel-execute.workflow.js#L793)), and the integrator merges every concurrent lane ref into the primary WE checkout on main ([we:parallel-execute.workflow.js:820](../.claude/skills/batch-backlog-items/parallel-execute.workflow.js#L820), merge at `:838`). So *both* lanes for a self-modifying item end in a mid-run write to the file the run is executing — the wedge is structural, not bad luck.
- **The partition has no predicate for this hazard class:** `conflicts()` is strictly *pairwise* ([we:scripts/readiness/lane-partition.mjs:123-129](../scripts/readiness/lane-partition.mjs#L123)) and `RESERVED_MERGE_RISK_BY_REPO` ([we:scripts/readiness/lane-partition.mjs:42-54](../scripts/readiness/lane-partition.mjs#L42)) omits the orchestrator/tooling files — correctly, because the denylist guards a *different* failure (clean-but-wrong merges between two items), and even a denylist hit would only route the item to the in-run serial lane, which is exactly the broken path. A *lone* self-modifying item sails into a concurrent lane unflagged.
- **Recurrence is structural, not incidental:** the orchestrator file carries 20+ resolve commits (its whole feature history lands as edits to itself), and the ratified mirror pattern (#2086: canonical tested module in `we:scripts/readiness/` + inline sandbox mirror) means every orchestrator-behavior item touches this surface twice.
- **Two later rulings reshape the premise:** the #2138 rider ([we:docs/agent/platform-decisions.md:2397](../docs/agent/platform-decisions.md)) rules that *every* lane-producing session "stops at lane-push … and never touches `main`", with a human-drained queue landing later; #2123 rules every edit-action session runs in an isolated lane clone. In that ratified steady state **no write ever hits the primary checkout during a producing run** — a self-modifying item's clone edit never touches the executing file, and its landing happens in a drain session that is not executing the workflow script. The hazard is therefore **substrate-conditional**: real under today's in-run integrator, structurally dissolved once the #2138 substrate (#2151/#2152/#2153, all open) is built and the in-run serial lane/integrator retire onto the queue.
- **External prior art** (report, §prior-art): CI systems uniformly *pin the run's definition at run start and land tool changes for the next run* — GitHub Actions runs the workflow definition of the triggering commit (a mid-run push to the workflow file never re-configures a live run); Jenkins pins shared-library versions per run; merge-bots (bors-lineage) manage a repo from a deployment *outside* the tree they land. Nobody lets a job mutate the executing pipeline definition mid-run; everybody lets the *change* ride the normal queue and take effect next run.

## Axis-framing

Two live axes, one dissolved. **Axis 1 — detection:** where does the "this item edits the run's own tooling" predicate live (a declared pathspec in the tested partition module, a per-item frontmatter flag, or a loader/engine exclusion)? **Axis 2 — routing:** which lane/session may *work* and *land* a flagged item under the current vs the #2138 steady-state substrate? The item's original option (b) — "move the lane/orchestrator tooling to a locus the run does not execute from" — dissolves (below): the movable logic already moved (the #2086 canonical-module pattern), and relocating the executing file does not stop it being the executing file. The invariant both forks serve: **an item's edits to the run's own executing tooling must never be applied to a checkout a live run is executing that tooling from** — mid-run self-modification is the excluded behavior; the *edit itself* is ordinary work that lands like any other change and takes effect next run.

## Fork 1 — Detection: where does the "self-modifying" predicate live?

**Fork exists because** the three candidate seats are mutually exclusive as the *authoritative* classifier (double authority = the #2014 drift class: a rule live in one derivation isn't live in the other), and one branch is positively broken: a per-item frontmatter flag **fails open** — a new item touching the orchestrator without the flag gets zero protection, and the very population this rule targets (orchestrator-feature items) is where omission is likeliest.

- **(a) Declared pathspec in the canonical partition module (default).** Add a `RUN_TOOLING` pathspec + `isRunTooling()`/`selfModifying(entry)` to [we:scripts/readiness/lane-partition.mjs](../scripts/readiness/lane-partition.mjs) (unit-tested, per the #2086 mirror pattern), inline-mirrored in the workflow sandbox. Matched against the **probe's** predicted touch-set (frontmatter `declaredFiles` are a stated lower bound — [we:parallel-execute.workflow.js:466](../.claude/skills/batch-backlog-items/parallel-execute.workflow.js#L466)) in Phase 1, **after probes and before the Phase-2 pre-claim** — so a flagged item is turned away while nothing has been claimed or pushed. Consistent with how the repo already declares hazard classes as reviewed constants (`RESERVED_MERGE_RISK_BY_REPO`), and with the declared-over-auto-derived house style (#2074).

  ```js
  // In we:scripts/readiness/lane-partition.mjs — canonical + tested; inline-mirrored in the workflow sandbox file.
  // RUN_TOOLING = files a live /workflow run EXECUTES or SHELLS OUT TO mid-run. NOT the merge-risk denylist
  // (different failure class: mid-run self-modification, not clean-but-wrong merges). NOT build config (#2149's
  // axis) and NOT the batch state files (claims/reservations/capacity — already governed: lanes never touch them).
  export const RUN_TOOLING = [
    /^\.claude\/skills\/batch-backlog-items\//,        // the executing workflow script + skill + progress board
    /^scripts\/(backlog|lane-pool|push-if-green|backlog-renumber-collisions)\.mjs$/,
    /^scripts\/readiness\//,                           // file-locks*, lane-partition, carry-forward — shelled out mid-run
    /^scripts\/(dev|lib)\/render-check\.mjs$/,
  ];
  export function isRunTooling(repoQualified) {        // "<repo>:<path>" — WE-only surface
    const i = String(repoQualified).indexOf(':');
    if (i < 0) return false;
    const repo = repoQualified.slice(0, i), f = repoQualified.slice(i + 1);
    return repo === 'we' && RUN_TOOLING.some((re) => re.test(f));
  }
  export function selfModifying(entry) {               // probe-time, authoritative
    return [...filesOf(entry)].some(isRunTooling);
  }
  // In the workflow script's Phase 1, after probes, BEFORE the Phase-2 pre-claim (nothing mutated yet):
  //   const selfMod = probed.filter(selfModifying);
  //   for (const e of selfMod) ledger.push({ num: String(e.num), status: 'dropped',
  //     drop: 'self-modifying — edits the run\'s own tooling; work it via serial /batch or a solo lane' });
  ```

  **Scope sub-choice (folded into the default):** protect the **full run-tooling surface** — the skill directory *plus* the scripts the run shells out to mid-run (`we:scripts/backlog.mjs`, `we:scripts/lane-pool.mjs`, `we:scripts/readiness/*`, `we:scripts/push-if-green.mjs`, `we:scripts/backlog-renumber-collisions.mjs`, `we:scripts/dev/render-check.mjs` + `we:scripts/lib/render-check.mjs` — the verified shell-out set) — not only the executing workflow file. The wedge class is the executing file, but an in-run serial-lane edit to a script a *later phase* invokes produces a mixed-version run (CI prior art pins the whole definition, not one file). Explicitly excluded from the set: the batch state files (already governed by the "lanes never touch `we:claims.json`" hard rule, [we:parallel-execute.workflow.js:688](../.claude/skills/batch-backlog-items/parallel-execute.workflow.js#L688)) and build config (that is #2149's separate merge-risk question).
  **Guard completeness (skeptic-amended, three seats + a drift test):** (i) the probe-time match above is the authoritative front door; (ii) a **post-hoc** check over each lane's reported `changedFiles` at integrate time catches an item that *turned out* self-modifying (probes under-report) — such a lane is not merged in-run but carried (refs preserved) for a serial re-route; (iii) the serial-lane agent prompt gains a hard instruction — *if your edits would touch a `RUN_TOOLING` path, stop, revert, report `dropped: "self-modifying"`* — closing the probe-failed-and-undeclared residual (`mustSerialize` routes a probe-less item straight to the in-place serial lane, [we:scripts/readiness/lane-partition.mjs:117-119](../scripts/readiness/lane-partition.mjs#L117), which the integrate-time seat never sees). And because the declared list is *itself* fails-open on drift (the workflow file's shell-out set grows with every orchestrator feature), a **coverage-parity unit test** beside the existing partition tests extracts every `scripts/*.mjs` invocation from the workflow source — **walking transitive shell-outs** (a script invoked by another `RUN_TOOLING` script, not only those named directly in the workflow file — ratification amendment) — and asserts `RUN_TOOLING` covers it — fails-open-at-runtime becomes fails-closed-at-test-time, with the declared list staying authoritative (#2074: the test audits the declaration, it doesn't replace it).
- **(b) Per-item frontmatter flag (`selfModifying: true`).** *Rejected* — fails open on omission (the unflagged item is precisely the failure case), duplicates what the probe already predicts, and adds a second **untested, per-item** derivation surface to keep in parity (#2014) — unlike the default's canonical-module + mirror pair, which is test-specced (#2086).
- **(c) Loader/engine-level exclusion** (a `stopTheWorld`-style `batchable:false`, per [we:src/_data/backlog.js:367-377](../src/_data/backlog.js#L367)). *Rejected* — seat mismatch: the same `--select` pack feeds serial `/batch`, where a self-modifying item is perfectly safe (no Workflow run executes the file); an engine exclusion would block the safe path too. The hazard is `/workflow`-mode-specific, so the predicate belongs in the `/workflow`-only partition seat.

**Skeptic:** SURVIVES-WITH-AMENDMENT — two attacks landed as amendments (folded above), no flip. (1) The fails-open critique that kills (b) applies to (a) *at list level*: `RUN_TOOLING` drifts as the run's shell-out set grows → the coverage-parity test. (2) A probe-failed **and** undeclared self-modifier still reaches the in-place serial lane (`mustSerialize`) — the exact wedge — → the serial-lane prompt guard (third seat) + the post-hoc `changedFiles` seat. The pack-seat attack ("`--select` should never offer these to `/workflow`") *fails*: the pack sees only frontmatter/derived fields, so a pack-time exclusion is (b)/(c) again — detection genuinely needs the probe; the drop is pre-claim, so nothing is mutated. Classification re-routes failed (not a config dimension — no consumer-legitimate "both values" for mid-run self-modification; not merge-risk-denylist turf — pairwise-only, different failure class; not settled by #2123/#2138 — neither defines detection). Statute-overlap: none — composes with [#merge-risk-optimistic-with-targeted-lock](../docs/agent/platform-decisions.md) (different failure class, disambiguated in the code comment) and keeps #1952's optimistic build-config bucket out of the set. Citation-scope: #2014 reaches only as the drift *lesson* (its subject is loader/engine field parity) and is cited as such; #2074 as house style — clean. One statute nuance folded: the merge-risk statute's static set is "derived by category, not a fixed file list" — `RUN_TOOLING` honors that discipline via the coverage-parity test (the category is "what the run executes/shells out to", and the test *derives* it from the workflow source to audit the declared list); note the post-hoc `changedFiles` seat shares the pathspec, so only the test closes the drift window.
**Screen:** clear — the seat is observable across the session/tooling boundary (a rule live in one seat and not another changes which items enter a run, the #2014 lesson); merit difference (fails-open vs fails-closed, over-exclusion) survives with both branches free to build.

## Fork 2 — Routing: where does a flagged item run and land?

**Fork exists because** under the *current* substrate the three routes are mutually exclusive per item and one is positively broken: **(b) the in-run serial lane is the observed wedge** — it edits the primary checkout in place while the Workflow tool executes the file from that path (`34a26a39`: "#2073's lane wedged editing the live orchestrator file (sandbox-locked)"), so routing flagged items there re-creates the incident by design.

- **(a) Exclude from the `/workflow` run; work it solo, or as the LAST item of a serial `/batch` (default — current substrate).** The probe-time drop (Fork 1) reports the item `dropped: "self-modifying"` before anything is claimed; the packer routes it to a solo session (post-#2123, itself a lane clone) or a serial `/batch` where no Workflow run is executing the tooling. **Last-position/solo, not anywhere in the batch (skeptic-amended):** a mid-batch edit to shelled run-tooling (`we:scripts/backlog.mjs`, `we:scripts/readiness/*`) changes what *later batch items' seams* execute — the same mixed-version condition Fork 1's scope rationale pins against, minus the sandbox lock; last-position/solo makes the change take effect next run there too. Costs nothing but scheduling: the item still lands the same day, just not inside a parallel run.
- **(b) In-run serial lane.** *Rejected* — the observed-broken branch (above). Also unfixable by ordering: even "serial lane last" still writes the executing file before the run's remaining phases (renumber/regen/reconcile/push all shell out to run tooling).
- **(c) Concurrent lane allowed; landing deferred to the drain (default — steady state; an *entailment of the #2138 rider*, cited here, not re-ruled).** Under the ratified rider a producing session never touches `main`; the clone edit never touches the executing file, and the drain (a separate session) lands it — the change takes effect *next* run, exactly the CI prior-art shape. Once the rider's substrate holds, a self-modifying item needs no special routing at all, so (c) is what the rider already grants — this fork adds only the interim rule (a) and the sunset trigger, avoiding duplicate authority over #2138's turf. **Sunset trigger for (a)→(c):** the #2153 PR drain is live **and** the in-run integrator + serial lane have been retired onto the queue (producers stop at lane-push). Until both hold, (c) is impossible (there is no queue; landing happens in-run), so this is a substrate-conditional route, not a timing preference. **Drain-era residual (skeptic-amended; recorded for the #2153 build and codified against [#pr-flow-rollout-mechanism](../docs/agent/platform-decisions.md), not ruled here):** the invariant transfers to the drain *as a live run* — the drain itself shells out to `RUN_TOOLING` (`we:scripts/push-if-green.mjs`, `we:scripts/readiness/*`), so within a drain pass, `RUN_TOOLING`-touching queued items land **last** (or as a dedicated final pass) and the drain invokes no `RUN_TOOLING` script after landing one; and the drain must not land a `RUN_TOOLING` change into the primary while a producing `/workflow` run is live (the sandbox lock makes that case fail loud). The same one-line ordering note applies to the *interim* re-route: a serial `/batch`/solo session landing a `RUN_TOOLING` change into the primary while a `/workflow` run is live wedges that run from outside — don't.

**The invariant (either route serves it):** the run's own executing tooling is never modified in the checkout a live run executes it from; tooling changes land like any other change and take effect on the *next* run.

**Skeptic:** SURVIVES-WITH-AMENDMENT — strongest attack **landed**: the sunset's premise "the drain is not executing the workflow script" was a contract narrowed to fit — the drain *does* execute `RUN_TOOLING` (`we:scripts/push-if-green.mjs` is the shared land helper; the drain reads `we:scripts/readiness/*` merge-risk logic), so a queued `RUN_TOOLING` change landed mid-drain makes the *drain itself* mixed-version. Fix folded: the drain-pass ordering residual above (land `RUN_TOOLING` items last / dedicated final pass; codify at [#pr-flow-rollout-mechanism](../docs/agent/platform-decisions.md)), plus the same ordering note extended to interim re-routes (don't land `RUN_TOOLING` into the primary while a `/workflow` run is live). With that folded, "permanent exclusion" *fails* (the drain handles its own hazard, so permanence buys nothing) and "never-pack instead of probe-drop" *fails* (pack-time detection is frontmatter-only → fails open; the drop is pre-claim, nothing mutated). Classification: (c) recast as an *entailment of the #2138 rider* (cited, not re-ruled) — the fork's own grant is the interim rule + sunset trigger. A second landed attack exposed an internal tension in (a): calling serial `/batch` "perfectly safe" while Fork 1 condemns mid-run edits to later-invoked scripts — fixed by the **solo-or-last-position** amendment folded into (a). Codification shape (statute-overlap): Fork 2 codifies as a **rider under [#pr-flow-rollout-mechanism](../docs/agent/platform-decisions.md)**, never a new anchor — (c) is the #2138 rider inherited, (b) is excluded on the `34a26a39` record, so only the today-route + sunset trigger are new statute. Citation-scope: the #2123 phase-1 capability-trigger citation reaches (same "when-the-capability-lands flip, not a permanent exemption" shape). Note: #2153 is itself currently blocked upstream (`blockedBy: 2160`), so the sunset is not imminent — capability-keyed, not date-keyed, so no verdict impact.
**Screen:** clear — routing is observable to the next session/run (which items a run accepts, where they land); with both branches free to build under the current substrate, (b) remains broken on correctness, so a merit difference remains. The (a)→(c) sunset was probed as possible prioritization-in-costume and cleared: (c) is not buildable on the current substrate, so "now vs later" is substrate-conditionality, not ordering.

## Dissolved / settled (not decisions)

- **The item's option (b) — "move the lane/orchestrator tooling to an editable locus":** dissolved. The movable logic *already moved* — the canonical-module pattern (#2086) keeps every testable piece in `we:scripts/readiness/` (lane-partition, file-locks, carry-forward) with the sandbox file as a thin mirror. The residual executing file cannot escape the hazard by relocation: the Workflow tool executes it *by path* (the `scriptPath` at [we:.claude/skills/batch-backlog-items/SKILL.md:238](../.claude/skills/batch-backlog-items/SKILL.md)), so wherever it lives, it is the executing file. The only variant that genuinely decouples — execute a run-start *snapshot copy* (the GitHub-Actions pin) — buys nothing once #2138 removes in-run landing, and adds a copy-drift surface meanwhile. If the sunset substrate slips indefinitely, snapshot-execute is the named fallback to re-open, not a fork today.
- **State files (claims / reservations / capacity JSON in the skill dir):** already governed — lanes never touch them; the orchestrator owns their writes (#1933 choice 2). Not part of this rule.
- **Build config (`we:package.json`, `we:.eleventy.js`):** #2149's open merge-risk question, deliberately not this item's surface (its own text carves the orchestrator-script case here and the build-config case there).
