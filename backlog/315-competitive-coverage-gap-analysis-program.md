# Competitive coverage gap analysis — benchmark the leading design systems & UI libraries, find what Web Everything is missing, and make the sweep repeatable

A recurring, systematic program that (1) maintains an exhaustive list of the **leading design systems and UI libraries**, (2) inventories each one's **components, documentation, and standards**, (3) maps every external capability against what the Web Everything platform already covers (blocks · intents · plugs · protocols · projects · semantics) to surface **gaps**, and (4) is **re-runnable on a cadence so a later pass produces a comparable, diffable result** rather than a one-off snapshot. The gaps it finds become candidate stories; this epic builds the *machine* that finds them, not the fixes themselves.

This productizes what was previously done **once, by hand**: the existing `gap #3–#13` items ([#006](backlog/006-gap-10-collection-ops-intent.md)…[#016](backlog/016-gap-9-webcommands-project.md)) are the output of an earlier, manual competitive sweep. That sweep had no defined corpus, no extraction method, and no way to re-run it; nothing today re-checks whether MD3, Carbon, Radix, Fluent, Spectrum, HIG (the `references.json` "Reference Design Systems" set) — or any library outside it — have shipped a pattern the platform still lacks. This epic turns that ad-hoc exercise into a standing, refreshable analysis.

## The pipeline (each phase becomes one or more child stories)

| Phase | What it produces | Notes |
|---|---|---|
| **1 · Corpus** | The exhaustive, maintainable list of design systems + UI libraries to benchmark, with selection criteria + per-source metadata | First child → **#316**. Extends the `references.json` "Reference Design Systems" set rather than forking a second list. |
| **2 · Extraction** | For each corpus source, a normalized enumeration of its **components, documented patterns, and standards/guidelines** into one capability schema | The lossy-normalization-hub pattern — ingest each incumbent bottom-up into an internal pivot the platform never ships. The *comparability* is the value. |
| **3 · Mapping & gap detection** | Each extracted capability mapped to WE's existing entities (block/intent/plug/protocol/project/semantics) and labelled **covered / partial / missing** | Reuses the existing capabilityMatrix + the triage fields the gap-# items already use (Native-first · Gap · Effort · Rank). |
| **4 · Gap → backlog** | Each `missing`/`partial` capability captured as a candidate backlog item (the `gap-#` shape), deduped against existing items | "Once the inventory is complete, we create stories to fill the gaps." These are the **child gap-fill stories** — generated by the process, not pre-written here. |
| **5 · Repeatability** | A defined re-run procedure (and any supporting tooling) that produces a **comparable, diffable** result on the next pass — same corpus → same shape, surfacing only what *changed* | The "run it again from time to time with a similar result" requirement. Strong relationship to **[#192](backlog/192-longitudinal-research-freshness-system.md)** (longitudinal research freshness) — this is its component-coverage instance; share its history/freshness model rather than reinventing it. |

## Design tensions — all resolved by the child stories (kept for rationale)

These were the open tensions when the epic opened; all are now settled (see *Status* below). None
remain a live fork — each links to where it was decided.

- **Where the capability matrix lives.** **Resolved (#346/#347):** a published `/research/` topic backed
  by a JSON capability file, each re-run a dated revision (the #192 chain model) — exposed at
  `/research/benchmark-corpus`, `/research/benchmark-capabilities`, `/research/benchmark-coverage`.
- **Corpus boundary — "leading" is a judgement.** **Settled in #316.** Explicit selection criteria
  (adoption, influence, breadth, distinct philosophy, native-alignment) spanning mature design systems,
  headless/primitive libs, and component kits — reproducible, not a taste list.
- **Granularity of a "capability."** **Resolved (#346 schema):** the matrix carries a `kind` so
  component-, pattern-, and token/standard-level capabilities coexist without collapsing — borrowing the
  intent/block/protocol/capability taxonomy.
- **Dedup against the live platform AND the existing backlog.** **Resolved (#348):** phase 4 reconciles
  against `blocks/intents/plugs/protocols/projects.json` *and* `backlog/*.md` so a re-run never re-opens
  `gap #3–#13` or duplicates live work (idempotent dedup).
- **Repeatability mechanism.** **Resolved (#349):** a manual `/`-skill first (#366; an agent re-runs the
  pipeline and reports the delta), graduating to a scheduled sweep (#367, blocked on #192) only once it
  proves stable. Shared with #192's axis-discovery trigger by design — one refresh mechanism, not two.
- **Native-first framing is mandatory.** **Recorded in the matrix (#347):** each capability carries its
  **native anchor** (`dialog`, `popover`, `<selectlist>`, anchor positioning…) so a library having a
  `<Card>` isn't automatically a platform gap.

## Why an epic (storied)

The phases above are independently valuable and separately deliverable (corpus, extraction method, mapping, the repeatability procedure), and the gap-fill work is an open-ended stream of child stories generated by phase 4 — so points live on the children, not here. Phase 1 is scaffolded as **#316**; phases 2–5 split into child stories once #316 pins the corpus and the capability schema's home is chosen.
