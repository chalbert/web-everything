# Solo-session lane isolation — prior-art survey for #2123

**Date**: 2026-07-02
**Point**: prep research for decision [#2123](../backlog/2123-should-a-solo-session-non-workflow-also-run-in-an-isolated-w.md) — should a solo (non-`/workflow`) session run in an isolated lane by default? The survey corrects the card's premise (the constellation's proven isolation primitive is a **lane clone**, not a git worktree — worktrees are guard-blocked and #1996 ratified "isolation = a clone"), finds the go/no half already directionally governed by the ratified statute [we:docs/agent/platform-decisions.md#pr-flow-rollout-mechanism](../docs/agent/platform-decisions.md) ("every automated writing session works in a clone", with serial-on-main named *the interim*), and grounds the residual live calls: the interactive-session verify loop, claim/status locus, and steady-state scope.
**Research page**: `/research/solo-session-lane-isolation/`

---

## Question

A single-item session (a lone `/next`, `/prepare`, a one-item `resolve`) mutates the shared primary checkout in place. The parallel `/workflow` orchestrator isolates each unit of work in its own lane; solo sessions get none. Should solo sessions also enter an isolated lane by default — and if so, how do the dev-server/verification loop, the claim semantics, and the landing path work for a *single interactive* session?

## Recommendation

**Adoption accepted on merit — the go/no gate dissolved** (after two skeptic passes + the fresh-context screen the ruling is a *confirm-and-sharpen* of #1996/#1998 statute: the screen flagged the go/no framing as prioritization — the go is ratified statute and every phase boundary is a mechanical capability edge, so the human turn on adoption compresses to a batch-confirm of the entry/landing mechanics; the ratify content is Forks 1–3). Concretely: (1) code-only solo builds default into a **lane clone** (the existing [we:scripts/lane-pool.mjs](../scripts/lane-pool.mjs) pool — not a worktree) now; visual + interactive-discussion sessions flip on the first green lane dev-pair probe; cross-repo solo lanes flip when the entry/closeout helper lands. (2) **Claim-class status transitions happen on the shared primary and publish to origin as `origin/main` + the splice only** (never primary HEAD — that would silently publish the human's unpushed commits); lanes never claim (the #1933 choice-2 pattern generalizes; a remote-locus claim is coherent, so this is a ruled fork, not a forced invariant); the resolve splice rides the lane's landing commit. (3) Verification for a lane session runs against the **lane's own dev pair** on its deterministic port band, with a **lane-local *before*** by default — the user's `:3000` (which keeps rendering the primary tree, untouched) serves as the before only when provably clean on the touched surfaces. Full fork shaping lives in the item.

## Key findings

### In-repo grounding (verified this session)

- **F1 — the isolation primitive is settled: a lane CLONE, not a worktree.** The user-global branch guard denies `git worktree add` / `checkout -b` in the shared checkout (proven live in the first parallel run — `we:.claude/agent-memory/parallel-workflow-blocked-by-git-guard.md:10-17`), and #1996 codified "branches are guard-blocked in the shared checkout (#1153), so **isolation = a clone**" (`we:docs/agent/platform-decisions.md:2306-2308`). The pool machinery is built and proven: persistent `--reference`-shared clones under `~/workspace/.lanes/<repo>/lane-<n>`, refreshed by fetch + hard-reset + clean, per-lane `npm ci` on lockfile change (`we:scripts/lane-pool.mjs:13-18,164-177`).
- **F2 — the go/no half is directionally governed by ratified statute.** #1996 (ratified 2026-06-30): "**Isolate-by-default for automation; the human writes `main` directly.** Every **automated** writing session (agent / `/workflow`) works in a clone … `main` is convergence-only for automation" (`we:docs/agent/platform-decisions.md:2306-2312`). Its enforcement ladder names today's practice explicitly as *the interim*: "the serial `/batch` still commits on the current branch per Rule 104 **as the interim**" (`we:docs/agent/platform-decisions.md:2334-2340`). So #2123 is not a cold go/no — it is the call that **ends the interim for solo sessions**. Citation-scope caveat: #1996 weighed *background automation* racing live lanes; the **interactive** solo session's live-observation loop (the human watching `:3000`, conversational verify turns) was not weighed there, so the statute is governing *direction*, and the interactive residual is re-derived on merit in the item.
- **F3 — per-lane dev-server ports are ratified and *generated*, with a verified dead link; the visual loop is not yet possible.** #1996/#1997 ratified pure deterministic per-lane ports with `strictPort` (lane *N* → band+100+10·N; WE `3000`/`8080` bands): `we:scripts/lane-pool.mjs:101-124` writes each lane's `.env.local`; `we:vite.config.mts:105-123` reads `WE_VITE_PORT`/`WE_ELEVENTY_PORT` **from `process.env`** with `strictPort: true`; `we:package.json:8-10` uses **shell** expansion of the same vars. **Skeptic-verified defect: nothing sources `.env.local` into the shell/process environment** (repo-wide, its only consumer is its writer plus a vite.config comment) — so a lane `npm run dev` today boots on 3000/8080 and `strictPort` collides loudly with the user's live server. The env-load link is a named missing build, and a lane booting its pair for a Playwright before/after has **never happened** — #1996's visual render-check harness (Gate 5, bounded by reproducing the #1895 `.fui-card` regression from the CLI) is a v1 deliverable, not built (`we:docs/agent/platform-decisions.md:2325-2329`). That is the honest not-yet on visual work.
- **F4 — claims are structurally centralized; a lane cannot claim.** The claim command guards concurrency via the `open → active` status transition on the *visible shared state* (`we:scripts/backlog.mjs:133-137`), and the claims ledger lives on the primary (`we:scripts/backlog.mjs:41`). The parallel orchestrator therefore **pre-claims all items centrally in the primary WE checkout and lanes never claim** (#1933 choice 2 — `we:.claude/skills/batch-backlog-items/parallel-execute.workflow.js:554-580`), pushing a `lane/_base` ref so lane clones reset to the post-claim state. A claim performed only inside a lane clone would be invisible to every concurrent session until landing — a double-claim race by construction. This forces the solo-lane claim locus: status splice on the primary first, then work in the lane.
- **F5 — the landing path is already composed, not open.** The authoritative gate runs once on the **merged tree** (`we:docs/agent/platform-decisions.md:2236-2255`, the gate-on-merged-tree anchor); the lane gate is at most the scoped fast-fail. Rule 104 (commit on the checked-out branch) composes cleanly: a lane clone's own HEAD *is* its local `main` (`we:scripts/lane-pool.mjs:167-169`), so a lane session commits on `main` in the lane, pushes the durable `lane/*` ref (guard carve-out + #1995 bounded push-retry), merges `origin/main`, runs the full gate on that merged tree, and lands. Cross-repo caveat carried from the runs: sync impl-repo origins first, or the lane resets to a stale tree and false-drops (`we:.claude/agent-memory/workflow-crossrepo-lanes-falsedrop.md:10-23`).
- **F6 — the lane model is proven at scale, with two solo-relevant lessons.** 2026-07-01 mixed run: 16/18 items landed green including cross-repo couples, 0 stranded (`we:.claude/agent-memory/parallel-workflow-blocked-by-git-guard.md:74-97`). Lessons that transfer to a solo lane: (a) carried/abandoned items strand `status: active` unless closeout releases them — a solo lane session that dies mid-item leaves the same stranded claim on the primary; (b) lane agents **over-drop on stale-tree misjudgments** — a solo session in a lane must re-verify any "absent from the tree" conclusion against lane freshness before dropping.
- **F7 — the non-destructive-closeout incident is the merit case FOR solo lanes.** The forced invariant `we:docs/agent/platform-decisions.md:2267-2278` (non-destructive-closeout-prflow) exists because a closeout on the shared dirty checkout destroyed a concurrent `/prepare`'s in-progress files (`batch-2026-06-29e`). Rule 1 removes the destructive reaction, but only isolation removes the *collision class* — two sessions editing the same file on one tree is last-write-wins silent loss either way.

### External prior art

- **F8 — industry norm for parallel coding agents is worktrees + per-instance ports; the delta here is repo-local and already ruled.** Git worktrees (shared object store, per-worktree HEAD/index, branch-exclusivity) are the dominant 2025–26 isolation primitive for parallel AI agents, with the same two caveats this repo already codified: worktrees isolate *code, not runtime* (shared ports/services — each instance needs its own port range), and clones trade disk for full independence ([Augment Code guide](https://www.augmentcode.com/guides/git-worktrees-parallel-ai-agent-execution), [Towards Data Science](https://towardsdatascience.com/ai-agents-need-their-own-desk-and-git-worktrees-give-it-one/), [Upsun](https://devcenter.upsun.com/posts/git-worktrees-for-parallel-ai-coding-agents/)). WE deviates from the worktree norm for a *local* reason (the branch guard protects the shared HEAD), and #1996 already ratified the clone answer — so worktree-vs-clone is **not re-opened** by #2123; the harness's own `EnterWorktree`/`isolation:"worktree"` remains fine for throwaway read-mostly sub-agents but is not the session lane primitive.
- **F9 — first-class VCS workspaces validate the "one repo, N working copies, boring convergence" shape.** Jujutsu `jj workspace add` gives multiple working copies backed by one repo, each with its own working-copy commit, expressly for "run a long build/test in one workspace while editing in another" ([jj docs — working copy](https://docs.jj-vcs.dev/latest/working-copy/), [jj workspaces write-up](https://www.joshualyman.com/2026/02/demystifying-jujutsu-jj-workspaces/)); Sapling/Mercurial share the lineage. The takeaway for #2123 is directional, not mechanical: modern VCS design treats "every unit of work in its own working copy" as the default posture, with the shared checkout as the special case — supporting go, not no.
- **F10 — hosted agent harnesses isolate per task as the baseline.** Devin-class agents, Copilot Workspace, and Codex-style cloud tasks each run in an ephemeral per-task workspace/VM converging via branch/PR — i.e. the "solo session in an isolated checkout, land through the remote" model is the *only* model hosted harnesses offer; working directly in the user's live checkout is the local-CLI anomaly, tolerated because the human co-supervises. This is the same asymmetry #1996 encoded (human direct, automation isolated).

## Trigger (for the phased visual carve-out)

Concrete and testable, aligned with #1996 Gate 5: a lane clone boots its WE dev pair on its band ports from its generated `.env.local` (requires the env-load link — F3's missing build — first), and a CLI Playwright before/after (before = a lane-local pre-edit snapshot by default; the primary's `:3000` only as a provably-clean shortcut; after = the lane band) captures a real UI diff — bounded "proven" by reproducing the #1895 transparent-`.fui-card` class of regression from the CLI. The first green probe flips both gated session classes (visual work *and* interactive-discussion sessions, whose live-board mitigation rides the same boot capability) from primary-only to lane-default.
