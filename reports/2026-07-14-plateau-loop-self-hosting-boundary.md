# Plateau Loop self-hosting boundary — prior-art survey and fork grounding

*2026-07-14 · grounds decision [/backlog/xeccleu](/backlog/xeccleu-plateau-loop-self-hosting-boundary-supervisor-clone-location/) (under epic #2445, supervisor slice #2468).*

## The question

The phase-1 resident drain daemon (#2449) runs itself under launchd for weeks, self-updating the WE **drain** code every pass — but it never self-updates its **own** source. Its own code (plateau:tools/drain-daemon/daemon.mjs, plateau:tools/drain-daemon/cli.mjs, plateau:tools/drain-daemon/lib.mjs) runs out of the user's PRIMARY plateau-app checkout, so every daemon change needs a manual `git pull` + `restart`. Closing that (the #2468 supervisor slice's "self-update-then-reload") is gated on three coupled design calls. This survey grounds them in the real code and in outside prior art so the decision is a fast ratification.

## Grounding in the actual daemon (verified against source)

- **Own source runs from primary.** `PLATEAU_ROOT = resolve(HERE, '..', '..')`; `install()` bakes the daemon entrypoint + `workingDir: PLATEAU_ROOT` into the launchd plist (plateau:tools/drain-daemon/cli.mjs). So the process launchd supervises is the user's working-tree copy.
- **The WE clone is already isolated and already self-refreshes.** The merge sweep runs from a dedicated single-lane WE clone (pool `we-drain-daemon`), provisioned by `ensureWeClone()` (`node we:scripts/lane-pool.mjs provision --name=we-drain-daemon --count=1`) and refreshed each pass by `refreshClone()` — `git fetch` + `git reset --hard origin/main` + `git clean -fdq`, with `npm ci --ignore-scripts` only on a lockfile-hash change (plateau:tools/drain-daemon/daemon.mjs, plateau:tools/drain-daemon/lib.mjs `resolveConfig`). This is the exact pattern Fork A would extend to the daemon's OWN source.
- **A clean shutdown path already exists.** `releaseAndExit(code)` kills the in-flight child with SIGTERM *before* releasing the drain lease (the double-drain guard), and both signal handlers call it (plateau:tools/drain-daemon/daemon.mjs). The plist carries `RunAtLoad + KeepAlive` (plateau:tools/drain-daemon/lib.mjs `renderLaunchdPlist`), so an exit is auto-relaunched.
- **The operator `restart` is bootout+bootstrap.** The `restart` command (plateau:tools/drain-daemon/cli.mjs) = `launchctl bootout` then `bootstrap`; `start` falls back to `launchctl kickstart -k`. Both are *operator* primitives run from a separate process.
- **#2480 registered the daemon's own files in the WE trust chain — as ENGINE tier.** [we:scripts/lib/gate-config.mjs](../scripts/lib/gate-config.mjs) lists all three with `tier: 'engine'`. `humanRequired` in [we:scripts/lib/review-escalation.mjs](../scripts/lib/review-escalation.mjs) keys ONLY on the policy tier (`isGateSelfPath`/`isPolicyCorePath`) + the statute layer — so an engine-tier edit ESCALATES (independent adversarial panel) but does NOT force `review:human`. The daemon also never clears its own parked reviews (it surfaces them for sessions), so its own source lands only after a separate session's converged panel applies `review:accepted`. NB (code-verified): the `redteam:accepted` label is *earned* in [we:scripts/lib/review-core.mjs](../scripts/lib/review-core.mjs) but is NOT itself checked as a merge precondition today (enforcement deferred to #2412, unwired) — do not treat it as a land-gate. This is the correction that reshapes Fork C.

## Prior art — self-updating resident daemons and reload primitives

- **launchd is the supervisor; a daemon self-reloads by exiting.** Apple's `launchd`/`launchctl` design treats `KeepAlive` as the relaunch contract: a managed job that exits is restarted from its (now-updated) on-disk program. The idiomatic macOS self-update is therefore "swap the files on disk, then exit cleanly" — not "ask launchd to kill you." `kickstart -k` exists for the *operator* to force-restart a job; issuing it from inside the job races the job's own teardown. `bootout`+`bootstrap` (the `restart` command) is inherently two external steps — the first unloads the running job, so a job can't run both against itself.
- **Blue-green / atomic-swap deploy.** The robust pattern for a long-lived worker adopting new code is: stage the new version in an isolated location, health-check, then cut over — never mutate the live working copy in place. A dedicated clone that the updater `reset --hard`s (Fork A) is exactly the "stage in isolation" half; `exit + supervised relaunch` (Fork B) is the atomic cut-over. Systemd's `Restart=` and process managers like pm2's reload follow the same "let the supervisor relaunch" contract rather than in-process `exec`.
- **In-process `exec` self-replacement is the fragile option.** `exec`-in-place (re-running the interpreter over the new entrypoint) keeps the pid but re-inherits open fds, timers, and — here — a *held* lease and a possibly-live child, and it sidesteps the supervisor's health/backoff. It is used where there is no supervisor; with launchd present it discards the safety net.
- **Self-modifying automation excludes its own source from its automated path.** The constellation already carries this instinct: self-modifying run-tooling is excluded from the drain's auto-land path (#2422 run-tooling-last; the `self-modifying-run-tooling-exclusion` research topic), and #2077 excludes an item from being parallel-edited by the process acting on it. The trust chain (#2285/#2448/#2480) is the review-gate expression of the same idea. The open residual is that a *review* gate does not cover *scheduling* (a daemon concurrently lane-editing itself), and that engine-tier escalation is agent-clearable, not human-gated.

## Recommendation (forks for xeccleu)

1. **Fork A — dedicated single-lane plateau-app clone** (mirror `we-drain-daemon`). Forced invariant: safe self-update must `reset --hard` the checkout the daemon runs from, which can't be the user's primary tree.
2. **Fork B — `process.exit(0)` + launchd `KeepAlive`.** The "reuse the proven `restart`" instinct fails: `restart`=bootout+bootstrap is external-only (self-bootout is suicide before relaunch). `exit(0)` reuses the proven `releaseAndExit` clean-release path; launchd relaunches on the new code. `kickstart -k` is redundant with KeepAlive and races the teardown; `exec` discards supervision.
3. **Fork C — rely on #2480's engine-tier escalation + a light in-daemon guard.** #2480 guarantees an independent adversarial panel + no daemon self-clear (a separate session's converged panel must accept it) — but NOT `review:human` (engine tier is agent-clearable), and it does NOT today gate on `redteam:accepted` (earned, not enforced — #2412). Add a light `isSelfSourceChange` deferral keyed on the daemon's source DIRECTORY (a basename guard would share the rename-fragility it backstops) as an escalation tripwire; leave promotion to policy tier (human-gated) as the ratifier's override. The concurrent-self-lane-edit exclusion is deferred with the runner/fanout (#2444/#2418) — the phase-1 daemon lane-edits nothing.

## Out of scope

Durable agent/lane-queue persistence and "the coordinator resumes WITH its agents after a reload" are entangled with the agent-runner (#2444) and orchestrator-as-Node-fanout (#2418); they stay deferred with phase-2 until the daemon's operating evidence (#2456) lands. This decision covers only the drain-daemon self-update subset.
