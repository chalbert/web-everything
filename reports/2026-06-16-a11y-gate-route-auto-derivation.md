# Auto-deriving the rendered-site a11y gate's route set — prior-art survey & decision prep

**Date:** 2026-06-16
**For:** decision [#774](/backlog/774-auto-derive-the-a11y-gate-route-set-from-the-11ty-collection/) (prep, not the call)
**Parent context:** [#763](/backlog/763-gate-ui-best-practices-on-website-changes-rendered-site-a11y/) (ratified the gate), [#770](/backlog/770-rendered-site-a11y-gate-axe-lane-route-allowlist-we-docs/) (built it, hand-list), [#793](/backlog/793-remediate-we-docs-a11y-violations-flip-the-rendered-site-gat/) (flipped 9/10 routes to `enforce`)

## The question

#770 shipped a **hand-maintained** route allowlist — [`we:tests/a11y/route-allowlist.ts`](../tests/a11y/route-allowlist.ts) lists 10 stable WE-docs catalog index surfaces (`/`, `/intents/`, `/blocks/`, … `/backlog/`). A new published surface is gated only when someone edits that file. #774 asks: **replace the hand-list with auto-derivation from the 11ty collection/sitemap, so every published page is gated automatically.** #763 deferred this as its Fork-3 alternative ("self-maintaining but scans churn-y pages and couples the gate to the build graph"); #793's per-route `enforce` ratchet sharpened it (auto-derivation must not silently un-earn those 9 enforced routes).

Two coupled, non-mechanical questions block any code: **(1) scope** — what does "every published page" resolve to? — and **(2) mechanism** — there is no sitemap today (`/sitemap.xml` 404s), so what artifact does the test read?

## Grounding in the real tree

- **The gate is a Playwright lane, not part of `check:standards`.** [`we:tests/a11y/rendered-site-a11y.spec.ts`](../tests/a11y/rendered-site-a11y.spec.ts) loops `GATED_ROUTES`, runs `AxeBuilder(...).withTags(WCAG_TAGS).analyze()` per route, pinned to `baseURL: http://localhost:8080` (the 11ty docs origin). It reuses the running dev server via `we:playwright.config.ts`.
- **Pages come from pagination over `_data` collections — one template per collection.** Every detail page is generated by a single `*-pages.njk` template paginating a `_data/*.json` collection with a `permalink:` pattern: [`we:src/intent-pages.njk:6`](../src/intent-pages.njk#L6) `permalink: "intents/{{ intent.id }}/"`, [`we:src/block-pages.njk:6`](../src/block-pages.njk#L6) `blocks/{{ block.id }}/`, [`we:src/backlog-pages.njk:6`](../src/backlog-pages.njk#L6) `backlog/{{ item.id }}/`, [`we:src/research-topic-pages.njk:6`](../src/research-topic-pages.njk#L6) `research/{{ topic.id }}/`, etc. There are **26 paginating/permalinked templates**. This is load-bearing: **a violation on an intent detail page is almost always a `we:intent-pages.njk` *template* bug shared by all ~N intents, not a one-off** — so one sampled detail page per template catches it.
- **The index surfaces are the standalone (non-paginating) templates.** `we:src/intents.njk` → `/intents/` (file-based routing), `we:src/blocks.njk` → `/blocks/`, etc. — exactly the class the current 10-route hand-list holds.
- **The output is deeply nested and large.** `dir.output: "_site"` ([`we:.eleventy.js:207`](../.eleventy.js#L207)); a full build emits **2,129 `we:index.html` files** (most under `/backlog/{n}/` — ~790 — and the other catalogs). Scope-A ("literally every route") runs axe over all of them.
- **`eleventy --serve` serves from memory — it does not populate `_site`.** `dev:docs = eleventy --serve --port=8080` ([`we:package.json`](../package.json)); only `build:docs = eleventy` writes `_site` to disk. The Playwright lane boots the *dev server*, so a `_site/**` glob mechanism would need an **extra full build step** the lane doesn't currently run.
- **No sitemap exists.** No `src/*sitemap*` template; `/sitemap.xml` 404s. Authoring one is a net-new (and independently useful — SEO) artifact.
- **The enforce posture is earned, per-route.** #793 flipped 9 routes to `enforce: true`; `/backlog/` is warn-only pending [#805](/backlog/805-flip-backlog-rendered-site-a11y-route-to-enforce-once-backl/). Auto-derivation must preserve this.

## Prior art

### Mechanism — how real a11y CI derives a route set

- **pa11y-ci `--sitemap` is the canonical auto-derivation.** Pa11y CI natively consumes a `sitemap.xml`: `pa11y-ci --sitemap https://site/sitemap.xml` parses it for the URL list, with `--sitemap-find`/`--sitemap-replace` to rewrite the origin (prod→localhost) and `--sitemap-exclude` to drop pages (e.g. PDFs). The sitemap **is** the industry-standard "every published page" registry that an a11y runner reads. ([pa11y-ci README](https://github.com/pa11y/pa11y-ci/blob/main/README.md), [Matthias Ott: whole-site pa11y](https://matthiasott.com/notes/generating-accessibility-test-results-with-pa11y-ci))
- **Lighthouse CI `staticDistDir` globs a build dir — but only the root.** LHCI auto-discovers HTML files in `staticDistDir` and serves them, with `--autodiscoverUrlBlocklist` to exclude. **Documented limitation: only files in the directory *root* are collected — nested subdirectories are skipped** ([LHCI issue #95](https://github.com/GoogleChrome/lighthouse-ci/issues/95), [LHCI config docs](https://googlechrome.github.io/lighthouse-ci/docs/configuration.html)). Our output is entirely nested (`/intents/{id}we:/index.html`), so a naive build-dir glob is exactly the fiddly case this tool warns about.
- **Sitemaps are a published-page registry by design** ([sitemaps.org protocol](https://www.sitemaps.org/protocol.html)); the standard 11ty pattern is a `we:sitemap.njk`/`we:sitemap.xml.njk` template iterating `collections.all` (eleventy-base-blog ships exactly this).

### Scope — how a11y practice handles large sites

- **The field explicitly recommends *against* testing every page, and *against* testing only the homepage.** Guidance: "don't try to test every single page on a large site; focus on **templates** and critical paths"; "running automated tests only on the homepage or a few key pages misses issues throughout your site — configure tools to test representative samples **across all page types and templates**." A representative sample "includes components and **template pieces** that are reused throughout your website." ([Pope Tech: sampling](https://blog.pope.tech/2024/02/26/improve-effectiveness-and-accuracy-of-web-accessibility-testing-with-sampling/), [TestParty: automated a11y](https://testparty.ai/blog/automated-accessibility-testing-guide))
- This is the load-bearing reshaping: it makes **scope-A (all 2,129 pages) the discouraged extreme** (slow, and redundant because detail pages share templates) and **scope-B (index surfaces only) the "homepage-only misses issues" anti-pattern** — and points at a middle the item didn't name: **index surfaces + one representative detail page per template/collection.**

## What the survey changed (vs the item's framing)

1. **Added a third scope option (now the recommended default): per-template representative sample.** The item framed scope as A (all) vs B (index-only). The sampling prior art + the one-template-per-collection tree fact dissolve that binary: gate every index surface **plus one detail page per `*-pages.njk` collection** (~26 extra page loads, bounded and self-maintaining as new collections appear). Catches template-level regressions — the dominant class — at ~1% of scope-A's cost, without scope-B's blind spot.
2. **Elevated the sitemap mechanism from "one of three" to the prior-art-blessed default.** pa11y-ci's `--sitemap` is *the* standard auto-derivation; authoring a `we:sitemap.njk` over `collections.all` also fills a real SEO gap. The build-dir glob carries LHCI's documented nested-dir limitation; the `_data`-reconstruction path re-implements permalink logic that already lives in 26 templates (drift risk). Both lose to the sitemap on merit.
3. **Confirmed enforce-preservation is a forced invariant, not a fork.** "Auto-derive then reset all routes to warn-only" silently regresses #793's 9 earned enforced routes — a broken branch by the fork-existence test. So it's not a `## Fork`; it's an invariant: keep an explicit `ENFORCED_ROUTES` set (seeded with #793's 9), and newly-derived routes enter **warn-only** (most-permissive default).

## Classification

Per the 7-question pass: this is a **devtools quality gate** (kin to `check:standards` and the existing a11y lane) — same classification #763 already recorded. Zero consumer lock-in; **no protocol minted** (a sitemap is the W3C/sitemaps.org-standard artifact, consumed not re-specified; axe *is* the WCAG engine). Mirrored **per-repo** (FUI's gate, [#771](/backlog/771-mirror-the-rendered-site-a11y-gate-into-frontier-ui-3001/), would auto-derive from its *own* sitemap) — the duplicated-dev-panel pattern. `graduatedTo: none` (a test + a sitemap template + a derivation helper, no standard artifact).

## Recommended path at a glance

| Fork | Recommended default | Main alternative | Confidence |
| --- | --- | --- | --- |
| 1 — route-set scope | **Index surfaces + one representative detail page per collection-template** | all-pages · index-only | Med-high |
| 2 — derivation mechanism | **Author `we:sitemap.njk` over `collections.all`; the gate parses it** | glob `_site/**` · reconstruct from `_data` | High |

**Forced invariant (not a fork):** preserve #793's earned `enforce` posture — an explicit `ENFORCED_ROUTES` set seeded with the 9 routes; newly auto-derived routes enter **warn-only** (most-permissive default), promoted per-route as each goes green.
