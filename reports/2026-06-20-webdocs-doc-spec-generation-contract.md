# Web Docs — Doc Spec: the generation-layout + options contract (prep for #1163)

> Prior-art survey + grounding digest preparing decision **#1163** (Tier-C child of the #1038 webdocs
> spec-surface epic). Subject: the **WE-layer Doc Spec** — the generation-layout + options *contract*
> the FUI generator (`fui:webdocs/generator.ts`, graduated from #424) conforms to. Constrained by the
> **#091 docs-as-code ruling** (the customer's `webmanifest`+`webcases` are the source of truth; Plateau
> never owns content; "cancel and self-host always holds"). No web judgment is made here — this prep
> brings the forks to Definition of Ready; `/next decision` makes the call.

## 1. What exists today (the ground truth)

The webdocs *product* is built (#424/#425/#427) but its **WE-layer spec is greenfield**: webdocs is a
`status: poc` project stub on the WE side (`we:src/_data/projects/webdocs.json`) with **no WE code
surface** — the only running implementation is FUI's generator. So the Doc Spec is a *contract authored
over a shipped impl*, not a design drawn cold.

The shipped FUI generator (`fui:webdocs/generator.ts`) already pins most of the contract, implicitly:

- **The webcases pivot** — `WebCase { id, title, description, code }` and `WebCases = Record<blockId,
  WebCase[]>` (`fui:webdocs/generator.ts:27-37`). The normalized case shape the ingestion adapters
  target; mirrors this repo's build-time loader `we:src/_data/cases.js`.
- **The webmanifest** — `WebManifest { id, name, description?, blocks? }`
  (`fui:webdocs/generator.ts:83-89`). `blocks?` is the *only* layout-bearing field: an optional ordered
  list of block ids this site documents.
- **The generated model** — `DocsSite { id, name, description?, pages: DocsPage[] }`,
  `DocsPage { blockId, cases }` (`fui:webdocs/generator.ts:92-103`). A **flat**, block-keyed page list.
  The generator emits a *data model*, not rendered HTML — rendering is the renderer's concern.
- **The layout rules**, all inside `generateDocsSite` (`fui:webdocs/generator.ts:112-118`):
  1. **Order** — pages follow `manifest.blocks` order when given (author controls the TOC); else every
     block present in the cases in **stable id order** (`Object.keys(cases).sort()`).
  2. **Scope** — a non-empty `manifest.blocks` is the **source of truth for scope**: a case block not
     named is omitted; a manifest block with no cases still produces an (empty) page so the documented
     surface is complete.
  3. **Options** — there are **none**. The signature is the pure `(manifest, cases) → DocsSite`; no
     options/presentation argument exists.
- **The resolver** — `resolveWebcasesInput` over the three input kinds `bundle | git | registry`
  behind one contract (`fui:webdocs/generator.ts:122-151`). This is **already settled** by #091 Fork 1's
  "all three behind one resolver contract" sub-decision — *not re-opened here*.

So the absent surface the Doc Spec must decide: **(a)** whether/how WE *normatively owns* the
manifest→site mapping (vs leaving it impl-private), **(b)** the navigation model the contract encodes,
**(c)** where presentation *options* live, and **(d)** whether the v1 contract is flat or hierarchical.

## 2. Prior-art survey — how the leaders specify a generation-layout + options contract

Surveyed the navigation/config contracts of the six leading docs generators (the docs-as-code corpus
behind #091/#426/#429/#552). Three archetypes emerge on a **derived ↔ authored** spectrum:

| Archetype | Tools | Where the TOC comes from |
|---|---|---|
| **Pure declarative manifest** | **Mintlify** (a single `docs`/`mint` JSON config → `navigation.tabs[].groups[].pages[]`), **VitePress** (`themeConfig.sidebar`) | One config file lists every page/group explicitly; array order = ordering, membership = scope |
| **Hybrid (autogenerate + overrides)** | **Docusaurus** (a `sidebars` JS module `{type:'autogenerated', dirName}` + `sidebar_position` frontmatter + `_category_` JSON) , **Astro Starlight** (`sidebar` array with `autogenerate:{directory}` + `sidebar.order` frontmatter) | Config declares the skeleton; `autogenerate` entries derive sub-trees from the filesystem; frontmatter overrides the alphabetic default |
| **Fully derived from source** | **Storybook** (the `Meta.title` `"A/B/C"` slash-path builds the tree; `storySort` only re-orders), **TypeDoc** (nav is a projection of the type graph; `sort`/`groupOrder`/visibility filters tune it) | No nav manifest at all — structure is emergent from content; a config controls only ordering/options |

Findings that bear on the forks:

- **F1 — Hierarchy is universal at the sidebar level; flat survives only in the top nav.** *Every*
  surveyed system models a tree (groups/categories/folders); the only flat layer is the top navbar
  (Mintlify `tabs`, VitePress `nav` ≤2 levels). The webdocs generator's current flat, block-keyed page
  list is the *immature* shape — the mature end-state is hierarchical. (Implication for the flat-vs-
  hierarchical question: it is *now-vs-later*, not *which-is-correct*.)
- **F2 — Ordering vocabulary converges.** Explicit array index (Mintlify/VitePress/Starlight manual),
  a numeric `position`/`order` frontmatter field (Docusaurus `sidebar_position`, Starlight
  `sidebar.order`, `_category_` JSON `position`), or a sort strategy (`storySort` `order[]` + `'*'`
  wildcard, TypeDoc `sort[]`). The webdocs generator's "manifest array order, else alphabetic" is exactly
  the Mintlify-explicit-with-autogenerate-alphabetic-fallback idiom.
- **F3 — Scope and presentation: the more *derived* a system, the more they split.** The declarative
  tools (Mintlify/VitePress) *fold* scope and presentation into one ordered array — being listed is both
  inclusion *and* placement. The derived tools (Storybook/TypeDoc) cleanly separate them: scope = a glob
  / entry-set + visibility filters, presentation = a separate sort/group config. The webdocs generator
  today folds (the manifest's `blocks` array is both scope and order).
- **F4 — Mintlify (the commercial docs-as-code leader) is the single-manifest archetype.** Its single
  `docs`/`mint` JSON config *alone* fully determines structure, scope, order, and per-node options (`icon`, `tag`, `expanded`,
  `directory: none|accordion|card`) — reviewable in one diff. It bets on total declarative control over
  the autogenerate convenience the OSS hybrids chose. This is the closest prior art to a "one manifest →
  whole site" Doc Spec and the natural fit for #091's single-source-of-truth requirement.
- **F5 — Options are open in the leaders that scale.** Docusaurus (`sidebar_custom_props`, a
  `sidebarItemsGenerator` escape hatch) and Starlight (`badge`, custom `attrs`) keep the per-node option
  surface *extensible*; Mintlify's is closed but rich. The WE house style ("standardize the meta-schema,
  not the list" — Intents Open Design) maps onto the open end.

## 3. The WE-layer placement — settled by the statute layer, not re-litigated

By the **#817 placement test** ("only types→WE; ALL runtime→FUI; a runtime symbol stays WE only if a
WE-side conformance gate consumes it") and the **#855 ruling** ("WE = contracts/protocols/conformance
ONLY; a framework-specific codegen is impl, never `@webeverything`; WE ships the contract + behavioral
conformance vectors, FUI owns the generator"), the cut is mechanical and is **ratified-by-statute, not a
fork**:

- **WE owns** — the *type contracts* (`WebManifest`, `WebCase`/`WebCases`, `DocsSite`/`DocsPage`) plus a
  **golden conformance-vector suite** (`(manifest, cases) → expected DocsSite`). The distribution end-
  state is a per-subpath export `@webeverything/contracts/webdocs` on the existing package
  (`we:contracts/package.json`, e.g. the one-line type re-export pattern of `we:contracts/lifecycle.ts`),
  the #239/#700/#872 ratified mechanism. The vector suite rides the established
  `we:conformance-vectors/*.vectors.ts` kit (`we:conformance-vectors/schema.ts`, #899/#1016).
- **FUI owns** — `generateDocsSite`, `parseWebCase`, `resolveWebcasesInput`, every renderer — all
  runtime (`fui:webdocs/generator.ts`).

The genuine forks below are *about what the contract says*, not *where it lives*.

## 4. The forks (Definition-of-Ready summary)

Three genuine forks survive the standing test; the rest are supported-by-default. (Full options +
tradeoffs + bold defaults are authored into the backlog item.)

| Fork | Recommended default | Excluded branch | Confidence |
|---|---|---|---|
| **1 · Contract strength** | **Deterministic, conformance-locked** — WE owns the manifest→DocsSite *mapping rules* + golden vectors; any conforming generator is byte-equivalent on the same input | thin type-only contract (mapping is impl freedom) — *coherent but breaks #091 no-drift* | **High** |
| **2 · Navigation model** | **Declarative-manifest authority + derived fallback** — the manifest is the authoritative TOC (scope+order); blocks omitted ⇒ derive from cases, alphabetic | pure-derived (Storybook model), no manifest authority — *contradicts #091 manifest-is-scope-SoT* | **High** |
| **3 · Options locus** | **Fold into the manifest as an open declarative `docs` extension** (`manifest.docs.*`, meta-schema standardized not the list) | a separate imperative `DocsOptions` argument — *re-introduces a second source that can drift from the manifest* | **Med-High** |

**Supported by default** (no fork — invariant, prior ruling, or prioritization, not a merit either/or):

- **WE/FUI placement** — types+vectors→WE, runtime→FUI (#817/#855). Ratify the statute.
- **Distribution** — `@webeverything/contracts/webdocs` subpath, type-only (#239/#700/#872).
- **Resolver / input shape** — `git | bundle | registry` behind one contract; already settled by #091
  Fork 1's sub-decision and implemented (`fui:webdocs/generator.ts:122-151`). Not re-opened.
- **Flat vs hierarchical (v1)** — **flat now, with the contract reserving a forward-compat hierarchy
  extension point.** Survey F1 says hierarchy is the mature end-state, so "which is correct" is settled
  (hierarchical, eventually); flat-now-vs-hierarchical-now is therefore *prioritization* (now vs later),
  not a merit fork — and prioritization is decided at what-to-build-next time, not in the fork. v1 ships
  the flat shape that matches the shipped generator + the #1161 sibling theme ("carry richer fields, not
  drop them" → reserve, don't preclude). Build hierarchy when a consumer demands it.
- **The webcases/webmanifests seam** — the Doc Spec *references* the base manifest envelope owned by
  webmanifests (#1161) and the case pivot owned by webcases (#1162); it owns only the *docs-layout
  interpretation* (the `docs` extension namespace + the mapping rules). No schema is duplicated across
  the three sibling specs.

## 5. Per-fork classification (the 7-question pass, recorded)

- **Which layer?** Contract (types + vectors) → WE; mapping runtime → FUI. (#817/#855.)
- **Protocol or intent dimension?** A *protocol/contract* (the generator conformance contract), not a UX
  intent.
- **Expose the whole axis?** Yes for options (Fork 3) — open meta-schema, default-less, project-config
  extends (Config-Extends-Platform-Default).
- **Fixed mechanic or dimension?** Navigation model (Fork 2) is a *fixed mechanic* (manifest authority is
  baked, not configurable per-site) because the alternative (pure-derived) is excluded by #091, not a
  legitimate co-equal end-state. Hierarchy depth is a *deferred dimension*.
- **DI-injectable?** The resolver's transport (`fetchSource`) is injected (already); the mapping rules
  are not (they are the normative contract).
- **Most-permissive default?** Yes — omitting `manifest.blocks` auto-includes every case block
  (permissive); the restriction (an explicit `blocks` list) is the author's opt-in.
- **Seam between intents?** The webdocs/webmanifests/webcases seam is the spec boundary (each spec owns
  its own envelope; webdocs owns only the orchestration/layout interpretation).
