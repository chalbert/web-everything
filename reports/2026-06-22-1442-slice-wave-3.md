## Could split — wave 3 (3 slices)

All three are factory blocks that already produce styled DOM (a styled noun, `is-a`), are
**stateful/reactive** (disclosure open-state, active-section, slot layout), and **do not transform a
hostile host** (so the #1349-S2 shadow opt-in stays a non-default, no B-vs-C fork). The codified
guideline (`we:docs/agent/block-standard.md` §7) therefore maps each to **B — persistent light-DOM**,
mirroring the shipping reference at `fui:blocks/wizard/WizardElement.ts:25`. Mechanism is settled — no
buried fork. This is the coherent "nav/structural wave" the prior draft wanted designed together.

| # | Block → element | Mechanism | Reference pattern | Size |
|---|---|---|---|---|
| 1 | disclosure-nav → `we-disclosure-nav` | persistent B | `fui:blocks/wizard/WizardElement.ts` | 3 |
| 2 | sectioned-nav → `we-sectioned-nav` | persistent B | `fui:blocks/wizard/WizardElement.ts` | 3 |
| 3 | app-shell → `we-app-shell` | persistent B | `fui:blocks/wizard/WizardElement.ts` | 3 |

DAG: none — `{1,2,3}` are mutually independent and batchable in one FUI lane. Each leaves a valid
demoable state (the existing `mountInDocument` contract provides the demo path).

## Could not split (this wave) — and the unblocking action

Not carved now; they remain incremental-on-pickup under the open epic (which now reads `tracking` once
the above land, so it never again looks falsely "done"):

- **pan-zoom-surface** — its `fui:blocks/pan-zoom-surface/PanZoomSurface.ts` transforms host content, so
  **B-vs-C (shadow, #1349 S2)** is a genuine per-block call. Carving it flat would bury that fork →
  file a `kind: decision` on pickup if contested.
- **Is-a/can-do blocks** (navigation, master-detail, marquee-select, edit-in-place) — each is a
  `CustomAttribute`/behavior-shaped block where "does it earn a `we-` element facade (support-both,
  #1457) or stay a pure behavior?" is a real per-block call, not a flat application → file the #1457-style
  classification decision per block on pickup.
- **temporal** — a presentational date/time `Component`; carve in a display-element wave once its
  transient(A)-vs-persistent(B) mechanism is confirmed.
- **Headless behaviors** (selection, for-each, guard, lifecycle, stores, view, resource-loader,
  draft-persistence, workflow-engine, traits, parsers, renderers, text-nodes, audit, bulk-action,
  annotation) — a "can-do" verb stays a `CustomAttribute` behavior, never a `we-` element. Out of scope
  for this epic by design.

## Recommendation (confidence ~80%)

Carve **wave 3 = slices 1–3** now (the clean factory-block → persistent-B nav/structural conversions),
leaving the fork-carrying tail for per-block decisions rather than buried-fork conversions. This regains
open children for #1442 so it reads `tracking` honestly. *Residual ~20%:* whether app-shell is a visual
block worth a `we-` wrapper vs. plain layout plumbing is the softest call in the wave — defensible as a
styled slot-host, droppable if challenged on pickup.
