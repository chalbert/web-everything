# #1442 Block-model conversion — slice wave 3 (2026-06-22)

Carves the next batchable wave off the [#1442](../backlog/1442-block-model-conversion-register-remaining-blocks-as-custom-e.md)
block-model-conversion epic. Wave 1 (button/badge/card) and wave 2 (meter/progress/checkbox/radio)
have all resolved, so the gate flags #1442 as "all slices done" — but #1442 is a **finite burndown
epic, not a program**: its remaining catalog blocks are real open scope, not a closeable umbrella.
Rather than resolve over the uncarved tail (the #1167/#1210 trap), this wave puts the next clearly
in-scope slices back into the pool so the epic reads `tracking` honestly and only resolves when the
catalog is genuinely drained.

## Mechanism is settled — no buried fork

The per-block forks are already ruled, so each slice is a flat application, not a decision:

- **#1457** (resolved) — behavior blocks → **support-both**: ship a `we-` element facade over the
  existing behavior kernel. Reference pattern shipping at `fui:blocks/stepper/StepperElement.ts` (over
  `fui:blocks/stepper/StepperBehavior.ts`) and `fui:blocks/tabs/TabsElement.ts` (over
  `fui:blocks/tabs/TabGroupBehavior.ts`).
- **#1381 / #1321** — mechanism per block per the codified guideline
  (`we:docs/agent/block-standard.md` §7): reactive/stateful light-DOM control → **persistent (B)**.

All three wave-3 candidates already ship a `*Behavior.ts` kernel and lack a `*Element.ts`, so the work
is the bounded element-facade addition the stepper/tabs precedent demonstrates — `size` 3, gate-verifiable,
each leaving a valid demoable state.

## Could not split (this wave) — deliberately deferred

Not carved now; they remain incremental-on-pickup under the open epic (which now reads `tracking` once
the above land, so it never again looks falsely "done"):

- **pan-zoom-surface** — its `fui:blocks/pan-zoom-surface/PanZoomSurface.ts` transforms host content, so
  the B-vs-**C (shadow, #1349 S2)** mechanism is a genuine per-block call. Carving it now would bury
  that fork → defer until the disposition is ruled (carve the fork to a `kind: decision` item if it's
  contested on pickup).
- **Navigation group** (navigation/nav-list, sectioned-nav, disclosure-nav, app-shell) — a coherent
  structural sub-family; carve as its own "nav wave" so the element semantics are designed together.
- **temporal** — a presentational date/time `Component`; carve in a display-element wave with its
  transient-vs-persistent mechanism confirmed.
- **Ambiguous is-a/can-do** (edit-in-place, marquee-select, bulk-action, master-detail) — each needs
  the behavior-vs-element call (#1457 is-a/can-do rule) before it's clearly a `we-` element; not a flat
  application, so deferred.
- **Headless behaviors** (selection, for-each, guard, lifecycle, stores, view, resource-loader,
  draft-persistence, workflow-engine, traits, parsers, renderers, text-nodes, audit) — a "can-do" verb
  stays a CustomAttribute behavior, never a `we-` element. Out of scope for this epic by design.
