# Merge-risk blacklist for we:package.json + we:.eleventy.js — prep research (#2149)

**Session report — 2026-07-02, `/prepare` lane for decision #2149.** Question: are `we:package.json` and `we:.eleventy.js` merge-risk (blacklist) files that force same-lane packing (rule 2c) + pre-lock + drain serial-replay, or does the #1952 "build config is line-structured → optimistic" principle cover them?

## In-repo grounding

**The machinery the blacklist plugs into.** `RESERVED_MERGE_RISK_BY_REPO` lives canonically in [we:scripts/readiness/lane-partition.mjs:42](../scripts/readiness/lane-partition.mjs#L42) (tested by [we:scripts/readiness/__tests__/lane-partition.test.mjs](../scripts/readiness/__tests__/lane-partition.test.mjs)) and is inline-mirrored in the Workflow sandbox at [we:.claude/skills/batch-backlog-items/parallel-execute.workflow.js:171](../.claude/skills/batch-backlog-items/parallel-execute.workflow.js#L171). Membership drives three behaviors: (1) partition rule 2c — two items sharing a listed file are forced same-lane ([we:scripts/readiness/lane-partition.mjs:125](../scripts/readiness/lane-partition.mjs#L125)); (2) the pre-lock layer — `reservedPathsFor` intersects a probe's `touchesMonolith` with the list to plan [we:scripts/readiness/file-locks.mjs](../scripts/readiness/file-locks.mjs) reservations ([we:.claude/skills/batch-backlog-items/parallel-execute.workflow.js:190](../.claude/skills/batch-backlog-items/parallel-execute.workflow.js#L190)); (3) since #2138 Fork 3 (ratified 2026-07-02), drain-time denylist-overlap detection with serial-replay as primary. A non-listed file rides the optimistic floor: rebase-retry → serial-replay of real conflicts → post-hoc `multiLaneFiles` notice.

**The codified rule the proposal touches.** #1952 (resolved 2026-06-29) demoted build config off the monolith trigger; its rationale is codified in the target file's own header ([we:scripts/readiness/lane-partition.mjs:33-37](../scripts/readiness/lane-partition.mjs#L33)): line-structured config merges clean on distinct lines, and a genuine same-line clash is a *real* git conflict the floor catches. Two prep findings sharpen the picture:

- **Neither disputed path was ever in the list.** `git log -S` over both homes shows no commit ever added/removed an entry for either path. #1952's ratified removals were `we:tsconfig.json` + `we:vite.config.mts`; `we:package.json` appears only in the header's generalized "BUILD CONFIG" parenthetical. A GO therefore narrows a codified *generalization*, not the ratified removals.
- **The clean-but-wrong case for keyed manifests is real but narrow.** Two lanes adding the *same* key at different offsets (both add a `wrangler` dep; both add the same script) git-merge clean into a duplicate-key JSON object; JSON parsing / npm resolve it silently last-wins, and no current gate parses `we:package.json` for duplicate keys. Distinct-key adds merge clean *and correct* — the #1952 argument holds for those.

**Incident history motivating the item.** batch-2026-07-01-wf (2nd real multi-lane run) flagged 7 `multiLaneFiles`; #2018 + #2020 both appended to `we:.eleventy.js` (component-render filters), #2024 + #2087 both appended to `we:package.json` (scripts entries). Related incident class (shared *checkout*, not lanes): the shared-index commit race (#2138/#2127, a peer commit sweeping staged files) is orthogonal — it is removed structurally by #2123's lane clones, and no blacklist entry affects it.

Also: under the drain, landing is serial anyway, so a ③ entry costs production-lane concurrency only.
- **#2148 (resolved 2026-07-02)**: split the FUI directive barrels per-entry ("strictly better than co-serializing") — the ①-exit precedent for `we:.eleventy.js`; #1938 did the same for `we:src/_data/adapters.json`, which was locked until its split landed.

## External prior art (short)

- **Path-scoped protections**: GitHub CODEOWNERS / protected paths and Zuul's declared-dependency serialization are the same shape as a static denylist — serialize only the shared scope, keep the rest optimistic (already the ratified #1935 hybrid).
- **npm-merge-driver** (npm's own historical answer): resolve manifest/lockfile merge conflicts by re-running the package manager — supports treating `we:package-lock.json` as regenerate-on-merge (category ②) rather than a second denylist entry.
- **Duplicate-key linting**: jsonc/ESLint-style parsers reject duplicate keys that plain JSON parsing silently last-wins — the basis of a deterministic post-merge guard considered as an alternative to (or belt-and-suspenders with) a blacklist entry.
- **ParallelChange (Fowler)**: additive hunks may land ahead of consumers only when *judged* additive — already folded into #2138's whitelist ruling.

## Synthesis — the prepared forks (post-skeptic)

The prep skeptic split the two files — they are **not symmetric**:

- **Fork 1 — `we:package.json`: (a) bare optimistic vs (b) declare ③ vs (c) optimistic + deterministic duplicate-key lint in the per-merge gate.** Default **(c)** — the skeptic REFUTED the draft (b) default: for a keyed manifest whose member order is semantically irrelevant, the *entire* clean-but-wrong class is the duplicate key (same key added at different offsets), which is script-decidable — a small parser rule in `we:scripts/check-standards-rules.mjs` (run per merge by the gate) converts the silent class into gate-red → serial-replay, restoring #1952's "a genuine clash is a caught clash" premise with strictly broader coverage than lane-only serialization (it also guards solo/human/drain merges). (b)'s residual benefit is avoiding a wasted replay — replay-speed, i.e. prioritization (#1961). The lint ships in the ruling's own changeset, so no unguarded interim opens. Sub-decision: `we:package-lock.json` gets no entry (derived; a lockfile clash is a real conflict that replays — the npm-merge-driver regenerate path is a possible later optimization, not implemented today).
- **Fork 2 — `we:.eleventy.js`: (a) optimistic vs (b) declare ③ now.** Default **(b)** — SURVIVES-WITH-AMENDMENT: a 382-line registration monolith (28 registration calls) whose failure class is *not* lintable — same-name registrations can be computed/wrapped (a name lint over imperative JS is fragile) and ordering-/side-effect-sensitivity is un-adjudicable by any script — so the pre-emptive entry is the only mechanism that closes the window. Amended out: the draft's "ratify the fragment split as end-state" branch (the statute's category-① rule already blesses split-and-exit generically — pre-blessing a future story is ordering); the delisting is conditional on a split *proving* order-insensitivity (or an explicit ordered index), settled in the split story.
