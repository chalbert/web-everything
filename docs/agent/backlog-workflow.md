# Backlog Workflow — the single source of truth for open work

> Tier-1 reference. Read when capturing an idea/issue/decision, or when a report ends with open questions.

The backlog is the website-as-source-of-truth tracker for ideas, issues, and decisions — so they don't rot in scattered markdown. **It feeds entirely off a directory of markdown files** — one file per item, and it's **completely dynamic**: the title, summary, and detail-page body are all derived from markdown, never hand-typed into frontmatter. Frontmatter holds only metadata (type, status, dates, links). It renders at `/backlog/` (a filterable, grid/list-toggleable index), and **every item gets a detail page at `/backlog/{id}/` automatically**. There is no `backlog.json`.

```
backlog/<id>.md            # one markdown file per item — metadata frontmatter + a markdown body (its title/summary/detail). The only file you create to add an item.
src/_data/backlog.js       # the loader — parses backlog/*.md into the `backlog` array the pages + validator all consume (single source)
src/backlog.njk            # the /backlog/ index (filter + grid/list toggle, status-ordered, live-collects project open questions)
src/backlog-pages.njk      # paginates the array → one /backlog/{id}/ detail page per item (renders the body)
reports/                   # deep thinking — a backlog item links OUT to a report; the report is not the tracker
```

**Tile ⊆ detail — the detail page is a superset of the tile.** Everything `src/backlog.njk` shows on a tile (badges, the epic `Contains` circles, `Blocked by` chips, the `unsliced·why` / `all slices done` / `⚠ open slice` epic-status badge, tags, references) **must also appear on the `/backlog/{id}/` detail page** (`src/backlog-pages.njk`). The detail page may show *more* (full body, leverage/unblocks counts, dates) but never *less*: a field worth surfacing at a glance is worth seeing on the page you click through to. **The badges/chips themselves can no longer drift:** every badge renders from ONE shared macro file (`src/_includes/backlog-badges.njk`) over ONE shared vocabulary (`src/_data/backlogMeta.js`), `{% import … as bk with context %}`-ed by both `backlog.njk` and `backlog-pages.njk` (detail passes `scale='md'`). `check:standards` **errors** if either template re-defines a shared badge macro locally or drops the import — so editing a badge in one place changes it everywhere by construction. This shared seam is also the #777 dogfood swap-point: its macro bodies become FUI `badge`/`filter-chip` component calls once those ship ([#1208](/backlog/1208-dogfood-the-backlog-badges-chips-onto-fui-badge-filter-chip-/)). New *page-level* surfaces (not badges) still follow the superset rule above.

## Three homes — nothing stays hidden

Every doc has exactly one home, and **nothing sits in the repo unreachable from the website**:

- **Research** (analysis, options, citations) → a report in `reports/`. Reports are **not** in the 11ty build, so a report is exposed only when it is either **promoted to a `/research/` topic** (`researchTopics.json` + `research-descriptions/{id}.njk`, where `id` = the report's de-dated slug) **or mirrored by a backlog pointer item** (`relatedReport`, no body — renders the report at `/backlog/{id}/`). A report that is neither is a **hidden doc**.
- **Spec** (the standard itself) → the website: `src/_data/*.json` + `src/_includes/*-descriptions/*.njk` (blocks, plugs, intents, adapters, projects, protocols, semantics). Exposed by construction.
- **Everything else** — ideas, issues, decisions, "do this later" → a **backlog item** at `/backlog/`.

The test for any new markdown: *is it research (→ report + a `/research/` topic or backlog mirror), spec (→ website), or a backlog item?* If it fits none, it doesn't belong in the repo. **`npm run check:standards` fails if any `reports/*.md` is hidden** — i.e. has neither a matching research topic nor a backlog reference.

## Where an open question goes

| Source of the question | What to do |
|---|---|
| In a `reports/*.md` report | **Convert** it to a `backlog/<id>.md` item with `relatedReport` pointing at the report. Shrink the report's "Open questions" prose to a pointer at `/backlog/`. |
| On a standards **page** (project/block/intent) | **Cross-reference** it: either a `backlog/<id>.md` item with a `crossRef`, or — preferred — declare it on the entity so it's collected live (below). |
| Ideally | **Live-collected from projects.** Add an `openQuestions` array to the entry in `projects.json`; `/backlog/` aggregates them automatically under "Live from the standards." Edit them on the project, never copy into the backlog. |

## Authoring an item

**Fastest path — scaffold it:** `node scripts/backlog.mjs scaffold --kind=story --size=3 --title="…" --digest="…" [--blocked-by=NNN|xNNNNNN,…] [--parent=NNN|xNNNNNN] [--session=<slug>]` (`--kind ∈ story|epic|task|decision`, the one merged axis — #466/#487) mints a collision-free **hash id** and writes a `check:standards`-shaped item — pass `--digest` to author it in one shot (omit it and a `TODO` digest line is left for you to fill). Or author the file by hand:

> **JIT numbering — a new item is born with a HASH id, not an `NNN` (#2288).** The birth-time `max+1` race (two parallel lanes grabbing the same next number, #2189) is structurally prevented: scaffold now files the item as `backlog/xNNNNNN-<slug>.md` (`x` + 6 base36, e.g. `x7k2q9a`), and all in-flight cross-refs (`blockedBy`/`parent`/`#xNNNNNN`) use that hash. The **drain — the sole serial writer to main (#2290) — assigns the real sequential `NNN` at land** (`max+1` against serialized main) and blind-rewrites the hash → number across the filename, the item's own frontmatter/body, and every other item's edges. So a hash-prefixed file is an in-flight item that hasn't landed yet; a numeric one has landed and its `NNN` is permanent. `check:standards` accepts both forms — a hash validates green pre-merge. **You never hand-assign a number** (the drain owns it); reference an unlanded sibling by its hash.

> **Spin-offs from a running session — scaffold `--session` so the item is born owned, not public (#670).** Without `--session` a scaffolded item is born `status: open` — instantly in every other session's `check:readiness --select` pool and **claimable before its digest / `blockedBy` / body are authored** (the born-public race; a concurrent batch can grab a half-authored spin-off). Pass `--session=<batch-slug>` and the item is born **`active` + `scaffoldedBy: <slug>`** (no `dateStarted` — that's the *claim* signal), so it is excluded from every other session's pool until you **publish it with `node scripts/backlog.mjs settle <NNN>`** (`active → open`, stamps cleared) once the digest + edges + body are written. Use `--session` for any agent spin-off filed mid-batch; hand-authored / ad-hoc items stay born-open. A born-active item left unsettled past its creating day is flagged **O1** by `check:health` (orphan recovery — `settle` it or revert to open).

Create `backlog/<id>-<slug>.md`. **The filename is the `id`**, and it has two parts: the **id token** — a `xNNNNNN` **hash** for a not-yet-landed item (the birth default, #2288 above) or a zero-padded **`NNN` number** once the drain has landed it (shown as `#042`, used in the URL `/backlog/<id>/` and for short refs) — plus a kebab-case **`slug`** (the human-readable text, which may be reworded). **Prefer `scaffold` (above) — it mints the hash for you.** A new item is born hash-keyed and the drain assigns its `NNN` at land; you never hand-pick a number. (The historic "allocate `max+1`, yield-don't-shuffle" birth rule is superseded by JIT numbering — it survives only as the demoted collision-heal backstop, #2291.)

> **The `NNN` is immutable — never renumber an existing item.** Once a file has a number, that number is its id for life. **Do not rename `backlog/<NNN>-…` to a different `NNN`** — not to tidy the sequence, not to close a gap, not to resolve a collision. Renumbering silently breaks every `#NNN` short-ref and the `/backlog/<NNN>/` URL, and under concurrent agents it cascades into id collisions (two sessions chasing the same next-number, each renumbering the other's work). **On any number collision the *newer* item yields:** give the item you are *adding* a different free number; never touch a number already on disk. You may reword the *slug* (see *Renaming an item* below) — you may never change the *number*. Reusing a deleted item's number is likewise forbidden (a never-allocated gap number is fine).

Frontmatter is metadata only — **no `title`, no `summary`, no `id`/`num`**; the title/summary derive from the body's `# H1` and first paragraph, the id/num from the filename. `check:standards` fails if the `NNN-` prefix is missing or a number collides. There are two shapes:

**1 — Content item** (its own write-up):

```markdown
---
kind: story | epic | task | decision   # the one merged nature+hierarchy axis (required) — #466/#487
status: open | active | preparing | parked | resolved   # `preparing` = a decision being /prepare-researched (in-flight like active, distinct on the board — #375)
size: 3                            # Fibonacci points — ONLY on stories + unstoried epics (optional on a decision)
parent: "049"                      # optional — NNN of the epic this rolls under (quote it: leading zeros)
blockedBy: ["079", "092"]          # optional — NNN(s) this item can't start until they're resolved (quote: leading zeros)
scope: ["src/backlog-view/", "docs/agent/"]   # optional — predicted touch-set (repo-relative path prefixes) a probe agent writes; the conveyor dispatcher reads it to hold overlapping items apart
dateOpened: "YYYY-MM-DD"          # quote it — keeps it a string, not a parsed date
dateResolved: "YYYY-MM-DD"        # required once status: resolved — the burndown plots this
tags: [tag-a, tag-b]
relatedReport: reports/YYYY-MM-DD-topic.md   # optional — deep dive (date is auto-exposed)
relatedProject: webvalidation                # optional — must resolve in projects.json
crossRef: { url: /blocks/droplist/, label: droplist block page }   # optional
graduatedTo: intent:droplist                 # set when status:resolved by becoming a real entity — or `none` if it resolved without spawning one
---

# Imperative title (this becomes the item title)

The first paragraph is the summary shown on the index card. The rest of the
body is the detail-page content. Keep the *deep* thinking in a report and link
to it via `relatedReport`, rather than pasting a whole report in here.
```

**Repo-locus on code-path references.** Every code-path reference in the body (and in reports) must carry a
`<repo>:` prefix so its constellation repo is unambiguous in chat / raw markdown: in-repo paths keep a
clickable relative link with the locus in the link text (`[we:src/_data/blocks.json](src/_data/blocks.json)`),
cross-repo paths are plain text with the prefix (`fui:blocks/…`, `plateau:src/domains/…`). Short aliases
(`we:`/`fui:`/`plateau:`) are the default; full names accepted. Full rule + carve-outs: *conventions.md →
Repo-locus code-path references* (gated by #884/#885).

*Enforced at write time so an in-work item never breaks the shared gate for a concurrent agent.* A
**`PreToolUse(Edit|Write)` hook** (`scripts/lint-locus-prefix.mjs --pre`) scans the *proposed* content
**before it is written to disk** and **denies the edit** if it would introduce a bare ref — so the working
tree never holds a gate-breaking ref mid-edit, and a concurrent agent's whole-repo `check:standards` can't
go red because of your unfinished work. (A `PostToolUse` lint + the `--staged` sweep / `npm run lint:locus`
remain as backstops for the rare paths the pre-write hook can't see — e.g. `cat >>`/heredoc body-appends
that fire no Edit/Write tool.) To unblock a denied edit, just add the `we:`/`fui:`/`plateau:` prefix and
retry.

**`scope` — the predicted touch-set, written once and read by the dispatcher forever.** `scope: ["src/backlog-view/", "docs/agent/"]` is an **optional** array of repo-relative **path prefixes** the item is predicted to touch. It exists so the deterministic conveyor dispatcher ([`we:scripts/readiness/dispatch-plan.mjs`](../../scripts/readiness/dispatch-plan.mjs)) can decide, *by script*, whether two queued items — or a queued item and a running lane — would collide, and hold the loser apart (`overlaps lane-N`) instead of double-booking a file. This is the **judgment/deterministic split** the statute [`#deterministic-core-thin-judgment`](platform-decisions.md#deterministic-core-thin-judgment) draws: **predicting** the scope is judgment work a `/workflow` probe agent does once and writes back here; **reading** it to plan dispatch is the deterministic half the script owns. An item with **no** `scope:` is `needs-probe` to the dispatcher (it can't predict overlap, so it won't launch it) — so the field is how a probe hands the item off to the machine. `check:standards` validates the shape (**an array of strings when present**); the loader (`src/_data/backlog.js`) sanitizes it to a clean `string[]` or `undefined`. Keep the prefixes coarse (a directory prefix, not every file) — the dispatcher's overlap check is prefix-aware.

**`blockedBy` — a directional prerequisite edge, not a "see also".** `blockedBy: ["NNN", …]` declares that this item *cannot start until each listed item is `resolved`*. It is a distinct relation from the two existing link fields: `crossRef` is a non-directional "see also" URL/label, and `parent` is epic grouping (which item rolls up under which) — neither carries the "blocks / is-blocked-by" direction. Lifting prerequisites out of prose ("blocked on…", "after #NNN") into this field turns the backlog into a real **DAG**, the prerequisite for any *deterministic* readiness scoring (#249/#250) that doesn't need an LLM to read sentences. Keep a short human note in the body too; the field is the machine-readable half. The loader resolves it to a `blockers` array (linked on the detail page), and `check:standards` **errors** if an edge is unresolvable, self-referential, or forms a cycle (the readiness function assumes acyclicity).

**2 — Pointer item** (mirrors a report): omit the body entirely. With a `relatedReport` and no body, the item's title, summary, and detail-page content are loaded *from the report md itself* — so the backlog shows the report, with nothing restated by hand.

```markdown
---
kind: story              # pointer items carry a kind too (required on every item)
status: open
size: 5                  # a story is always sized
dateOpened: "2026-06-02"
tags: [droplist, traits]
relatedReport: reports/2026-06-02-droplist-trait-language.md
relatedProject: webtraits
---
```

**Renaming an item (slug reword *only* — never the number) — keep old links alive.** Only the `slug`
half is reword-able; the `NNN` is immutable (see the immutability rule above — never renumber an
existing item). The `NNN` number is permanent, so a
cited `/backlog/<NNN>/` URL survives any reword (it redirects to canonical via `src/backlog-redirects.njk`).
But a previously-published link to the **old slug** (`/backlog/<NNN>-old-wording/`, or a pre-NNN
`/backlog/old-wording/`) would 404 after the rename. To preserve it, add the **full former URL segment**
(the whole old filename stem) to a `formerSlugs:` array on the renamed item:

```yaml
formerSlugs: [042-old-wording, even-older-wording]
```

A build-time template (`src/backlog-slug-redirects.njk`, fed by `src/_data/backlogAliases.js`) then
emits one tiny redirect page per entry at `/backlog/<former>/` → the item's canonical `/backlog/<id>/`.
`check:standards` rejects a `formerSlug` that collides with a live item id or with another item's alias.

Live-collected form, on a `projects.json` entry:
```json
"openQuestions": [
  { "question": "Should validity merge be last-writer-wins or strictest-wins?", "anchor": "merge-strategy" }
]
```

## Agile sizing — the burndown's fuel

Every item carries a `kind` (the one merged nature+hierarchy axis — #466/#487), and **points (`size`) live at exactly one level** so the `/backlog/` **Burndown** tab (a continuous-flow chart of remaining vs. scope vs. completed *points*, with burn-rate and a future projection) never double-counts. `npm run check:standards` enforces all of this.

| `kind` | What it is | `size`? |
|---|---|---|
| **story** | One coherent, independently valuable deliverable (a block/intent/trait/adapter feature). The default. | **Required** — Fibonacci `1 / 2 / 3 / 5 / 8 / 13`. |
| **task** | Sub-work with no standalone value — a regression guard, e2e/coverage, wiring, doc/runtime reconcile, a bug fix, follow-up hardening. Rolls up under a story/epic via `parent`. | **Never** — its points belong to the parent. |
| **epic** | An umbrella spanning multiple items (a program/initiative/vision). | **Storied** epic (its children are their own items) → **no `size`** (the child stories carry the points). **Unstoried** epic (a bucket not broken into child items) → **sized** as a whole. |
| **decision** | An open design fork to ratify — Tier B (one nod away), the buried-fork lint's home. | **Optional** — a decision may carry a size (the analysis effort) or none; never required. |

**The no-double-count rule, mechanically:** the burndown sums every item's `size`. A `task` has none; a *storied* epic has none; a `story` and an *unstoried* epic each have exactly one. So each unit of scope is counted once. The validator **errors** if: a story lacks a size, a task has one, a size isn't Fibonacci, a `parent` doesn't resolve, or an **unstoried (sized) epic has a sized child** (that would count its scope twice — make it storied or re-parent the child).

> **Strict rule — an epic is in exactly one of two states, never both:**
> - **Unsliced** — *no children*. It carries a **`size`** standing in for work not yet broken out (a build bucket, or a triage/planning artifact whose points score the analysis effort — e.g. an as-yet-unsliced strategy matrix).
> - **Sliced** — *has ≥1 child*. It carries **no `size`** and is a pure umbrella; **all** scope lives on the child cards (story children carry burndown points; task children carry none).
>
> **The flip is triggered by gaining the *first child of any kind* — story or task — not only a sized one.** "Containing a child" *is* "sliced." So the moment you scaffold a child under an epic, **drop its `size` in the same edit**. From then on, every new unit of work — a follow-on deliverable, a fork that became a build, scope that grew — is a **fresh child card** (`scaffold … --parent=<NNN>`), **never** a `size` re-added to the epic. There is no valid middle state: **"a sliced epic that still has a `size`" is a bug** — the fix is always to remove the epic's `size` (its scope now lives on its children), never to put points back on the umbrella. Equivalently: if an epic still has a `size`, it is unsliced and either (a) needs slicing, or (b) is an unbroken bucket/artifact you resolve *as a whole*; if it has children, it has no `size`, full stop.

**Sizing guide:** `1` trivial · `2` small · `3` moderate · `5` substantial · `8` large (batchable ceiling) · `13` very large (should-split). Size *relative effort/uncertainty*, not hours. A `story` of `size` > 8 (`13`) is a **split candidate** — too big to batch and at risk of bundling several deliverables in one item; see *Splitting a large story* for when (and when **not**) to slice it. `parent` (quote it — leading zeros) links a story/task to the epic it rolls under, for grouping; it does not affect the point sum.

## Programs — the strict bar for a perpetual `ongoing` epic {#program-definition}

`ongoing: true` (see *Closing out* step 4 below) marks an epic that never resolves. The flag is cheap to set, so it needs a **strict bar** or it degrades into a vanity label on any big theme. A **program** is the hardened form: a standing effort that guards a goal which is *never done because the world keeps moving*. "Evergreen" is the same idea as an adjective — a thing kept perpetually current is held that way **by** a program (the program is the noun; evergreen is the property).

**The Program Test — a line of work is a program iff all four hold. Fail any one and it is something else (see *What it is NOT*).**

1. **Standing goal, no Definition of Done.** Its goal is a *steady state to maintain*, not a deliverable to ship. Decisive check: **if you resolved every current child, the program would not end** — drift creates the next slice. "Done" is a state it *passes through* (see *watch mode*), never its terminus. An epic dies when its children close; a program persists.
2. **A conformance front (internal).** A *named* metric answering **"are we green against the currently-known target?"**, plus the mechanism that re-checks it (a gate/skill). If you cannot name the metric in one line, it is not a program — it is a wish.
3. **A currency front (external).** A *named* external-delta signal answering **"has the world moved so the target itself is now stale?"** — a competitor shipped, new tech landed, new research changed best practice — plus a **discovery method** that scans those signals and *files new items* to pull the goal back to current. The discovery method must be runnable **by hand**; automating it is a later maturity stage, not an entry bar. A program missing front B entirely is just an epic with a watch wish.
4. **A cadence.** A defined trigger for each front — internal drift, an external signal, or a schedule. Run-once is not a program.

**Naming — a program's title is a short bare name, no subtitle.** A program is a standing noun you cite for the life of the constellation (`/review-program <slug>`, "run the platform-standards watch"), so its H1 must read as a *name*, not a one-line pitch: a few words, **no `—`/`–`/`:` subtitle clause** and **no `(parenthetical)` aside** tacked on. Any elaboration — the lens shape, the front-A/front-B mechanics, the "cards only" discipline — belongs in the **body discussion**, never the title. When the title needs a disambiguator (which exercise app, which domain), fold it *into* the name rather than parenthesizing it: `Loan-origination exercise app`, not `Exercise app A (loan origination)`. Good: `Platform-standards watch`, `Framework-churn watch`, `Parity Loop`, `Latent-standard discovery program`. Bad: `Latent-standard discovery program — multi-axis lens catalogue (each lens only emits candidate cards)` (the trailing clause is body content); `Exercise app A (loan / mortgage origination)` (parenthetical aside). This is a **hard gate**: `check:standards` errors on any program (an epic with `ongoing: true` or `childlessReason: program`) whose title carries a subtitle separator *or* a parenthetical — fold the clause into the opening paragraph and keep the H1 a name. (User directive 2026-06-21; extended to parentheticals 2026-06-22.)

**Watch mode is a real lifecycle state, not "closed."** A program cycles `active → green / caught-up → watch → discovery finds a delta → spawns items → active …`. So a program legitimately sits **`open` with zero open children** while in watch mode — the *opposite* of the normal "resolved-looking epic" smell. This is exactly what `ongoing: true` encodes and why the gate must never nudge it to resolve.

**Maturity ladder — a program need not start automated:** **L0** manual discovery (a human runs the sweep) → **L1** skill-assisted (a `/gap-sweep`-style command) → **L2** scheduled / automatic. The conformance metric (front A) must be **definable at L0 and actually exist to graduate past L0** — a young program may have a clear goal and a discovery story before it has a clean metric, and that is allowed at L0 only. (This is the path #315 → #367 already walks: manual sweep skill built; scheduled version parked pending a track record.)

**What it is NOT.** A finite goal with a Definition of Done is an **epic**, however large or however many children (fails #1). A grouping that only *collects* existing items is a **theme** — use `relatedProject`, not a program; a program is a *source* of work, not a bag for it (fails #1/#2). A skill or gate by itself is a **tool** — the mechanism, not the standing goal it serves (fails #1).

**Worked examples.** The flagship exercise-app loop ([#314](/backlog/314-flagship-exercise-apps/), gated by `check:app-conformance`) is a real, operational program with a strong **front A** but **no front B** — it re-aligns apps to *standards* drift, not to the outside world. Treat it as the conformance-front reference, not the full shape. **[#099 "the evergreen app"](/backlog/099-evergreen-app-vision/)** is the full **two-front** archetype — re-aligns to standards (front A) *and* adopts new tech / competitor patterns / research (front B, "ever-improving") — but is currently **L0 / aspirational**: the goal and discovery story exist, the operational hooks do not yet. Cite it as the *target shape*, not a passing program.

**The currency portfolio.** Front B (currency) is shared infrastructure, not a per-program reinvention: the constellation runs a *set* of external watches, each guarding a different moving domain — platform standards ([#1257](/backlog/1257-platform-standards-watch-keep-we-current-as-the-web-platform/)), the design-system landscape (gap-sweep [#315](/backlog/315-competitive-coverage-gap-analysis/)), web/reference decay (reference-liveness [#192](/backlog/192-longitudinal-research-freshness-system/)), framework API churn ([#1258](/backlog/1258-framework-churn-watch-keep-we-s-forward-adapters-current-as-/)), and (Plateau) model capability ([#1259](/backlog/1259-model-capability-watch-plateau-keep-on-device-cost-linearity/)). The evergreen app ([#099](/backlog/099-evergreen-app-vision/)) is their primary *consumer* — its front B is "adopt what the watches surface." Treat a new watch proposal as an addition to this portfolio, each justified by the WE ruling or standard it keeps current. **Outward name (sellable):** the watch portfolio is the *monitor/upgrade autonomy* of the **Self-Driven Project** ([#666](/backlog/666-self-driven-project/)) — the project keeping itself current without a human driving each step. "Program"/"watch" stay the internal mechanism terms; "Self-Driven Project" is the brand they roll up into.

## Running a program review (the watch) {#program-review}

A program is only as alive as its last watch run. **Running a program review** is the manual operation (the L0/L1 form of the *discovery method* criterion #3 requires): it executes a program's **front-B discovery** against later data, refreshes the **front-A conformance** read, and turns the delta into work. It generalizes the gap-sweep (#315) to any program, and is **idempotent and dated** exactly like it. Operationalized by the `/review-program` skill; the *method* is here, the skill is a pointer + the commands.

The loop (per program item `NNN`):

1. **Orient — what's been done, *grounded in code not status*.** Load `NNN`; confirm it is a program (`childlessReason: program` or `ongoing: true`). Read its two fronts and DoD from the body, its **last review** (newest `reports/<date>-program-<slug>.md` and `## Review log` entry), and enumerate prior outputs — resolved children (`grep -l 'parent: "NNN"' backlog/`) and prior review reports. **A resolved child is a *claim* of progress, not proof of it — spot-verify the load-bearing ones against the actual code before counting them.** The backlog's `status: resolved` measures that an item was closed, not that its behaviour is live: a keystone can be marked resolved while the code is a stub (the 2026-07-01 audit found #1243 "reproduce shadcn" resolved with no shadcn artifact on disk, and the #777 dogfood epics "resolved" while the site was never actually SSR'd from its own components). So for each child the program leans on, open the file(s) it claims to have delivered and confirm the capability exists — **a "resolved-but-not-in-code" child is a *reopened gap*, and becomes a front-A candidate item in step 3, not a tick in the done column.** State current maturity (L0/L1/L2) and the front-A conformance read (the metric, if it exists yet) against *verified* state. **Then run the *goal-completeness* enumeration — the pass the currency sweep structurally cannot do.** A program's north star almost always implies a **finite, enumerable target-set**: the design systems #1226 must reproduce (shadcn / Material / Ant / Carbon / Fluent), the target kinds #1451 must ship an adapter for, the app-classes and issue-classes #1522's explorer must audit unattended, the axes #1399 must diff. Read that set out of the body — or reconstruct it from the goal when the body never listed it (an *unlisted* goal-set is itself a finding: the program has been running on whatever work happened to get filed, never against its full scope). Then map **each element** to a filed/resolved child *and* to live code. Every element with **no child**, or with a **resolved-but-stubbed** child, is *incomplete work toward the goal* — a front-A candidate for step 3, owed **independent of any external change**. This is the question front B cannot ask: *is the program actually finished against its own stated goal?* For a living program the honest answer is usually "no", so a review that files **0** completeness items should be the exception (a program genuinely at parity with its goal), not the default.
2. **Refresh — run the watch (front B).** Execute the program's discovery method against *later* data, scoped by its lenses:
   - a program with a discovery **script** (gap-sweep → `scripts/gap-sweep-status.mjs`; reference-liveness → `npm run sweep:references`) → run it;
   - a **research / spec** program (#1257 platform standards, #1258 framework churn) → web-search/fetch the domain for changes *since the last review*, one pass per lens.
   Capture the external delta: what is new, moved, or obsoleted since the last run.
3. **Review + brainstorm — what could be done.** Candidates come from **two independent sources, and the review owes both**: (a) the **front-B currency delta** — what front-A conformance just went stale, what gap opened, what is newly adoptable since the last run; and (b) the **front-A completeness residual** from step 1 — every element of the goal-set with no live coverage. A program can be **externally dry yet internally incomplete** (front B surfaces nothing new, yet 4 of 5 parity targets are unbuilt, or a framework feature #1258 tracks has no WE equivalent, or a target kind #1451 names has no adapter) — so filing the completeness residual is **not gated on anything having changed externally**. Draft candidate items (the *could-do*) from both sources, each tagged to the front it serves. Still idempotent (#315): once the residual is filed, a re-run over an unchanged goal-set + landscape opens **0** — completeness doesn't break idempotency, it adds a **second axis** along which the landscape can be incomplete (a new goal-set element, or a regression, re-opens it).
4. **Propose → file on `go` (idempotent).** Present the delta + candidate items; on approval, file **only the newly-appeared** ones under `NNN` (`scaffold … --parent=NNN`) — a re-run over an unchanged landscape opens **0** items (the gap-sweep idempotency rule). Then stamp the run: append a dated entry to the program's **`## Review log`**, and **append** the run detail to the program's single **living report** `reports/<first-run-date>-program-<slug>.md` (created on run 1, a new dated `##` section per run). Set the program's **`relatedReport`** to that file on the first run so it never reads as a *hidden report* — the standards gate requires every report be referenced and `relatedReport` is single-valued, so it is **one living report per program, not one per run**. **Discovery proposes, the human disposes — never auto-file.**
5. **Gate.** `npm run check:item -- <NNN>` per touched item, `npm run check:standards` before done.

The first skill-run of a program's watch is its **L0→L1 graduation** (defined → skill-assisted); scheduling it (L2) is a later, separately-prioritized move (the #315→#367 pattern). For **#315** itself the domain-specific corpus refresh stays owned by the *gap-sweep-rerun* skill — `/review-program` delegates to it rather than duplicating the corpus steps.

**Bare `/review-program` (no `NNN`) — pick the program most worth reviewing now, don't just list the stalest.** A watch is worth running when it will *generate valuable items*, which is more than "longest since last run". Split the pick deterministic-vs-judgment (rule #51): `node scripts/program-review-rank.mjs` computes the **objective** signals for every program and ranks them — `staleDays` (un-swept external delta accrues with time), `neverReviewed` (a first run always catches an accumulated backlog), `openChildren` (a **dry pool** — few open items — means the program is *starved* for work, so a review that files items is high-yield; a full pool is already fed), and `resolvedChildren` (an established, live program worth keeping fed). The composite `score` = `stale × first-run × dry-pool`. Then the skill applies the **judgment** half over the top few: **domain velocity** (a fast-moving watch — framework-churn, platform-standards, model-capability — surfaces more than a slow one since the last run) and **code-drift risk** (the step-1 grounding lesson: a program whose "resolved" children are most likely *not actually in code* is exactly where a review yields the most). Present the ranked top ~3 with a one-line "why now" each, **focus #1 and run the per-program loop above on it**, and keep the runners-up visible so the human can redirect to a different program in one word. The ranker is read-only and idempotent — it never files anything; it only chooses *what to review*.

## Closing out a completed item — mark it `resolved` (after a final-capture pass)

The backlog tracks **open** work; when an item — or a plan it mirrors — is **fully done**, flip its frontmatter `status` → **`resolved`** (don't delete the file). `resolved` items are dropped from selection (see *Gather*) and hidden by default on `/backlog/`, so they don't clutter the live view, but the file stays as an audit trail and keeps any `relatedReport` exposed. Never close on a hunch — run this gate first, **carefully**, because close-out is where leftover work silently disappears:

1. **Confirm it's actually done.** Re-check the item's own acceptance criteria against reality — relevant tests green, `npm run check:standards` green, build green. **If the item added or changed a standard's feature, the standard's conformance demo must reflect it** — a shared fixture/case exists for the new feature and the playground shows it (demos are fixture-driven; see the Definition of Done in AGENTS.md). If any criterion is unmet — including a feature with no demo case — it is **not** done; leave the item `active`.
2. **Take a careful last look for leftovers.** Did the work surface anything deferred, half-finished, or newly exposed (a follow-up, an edge case, a related cleanup)? Each such point gets its **own new backlog item now** — *review-before-adding / dedup first* (see Rules). Capturing leftovers before close-out is the whole point: nothing is lost when the item leaves the live view. **Verify each new item is registered, and don't claim it's live yet:** the `check:standards` you run in step 3 reports a backlog count — confirm it rose by the number of items you added (that proves the frontmatter parsed; a malformed item is silently skipped). New items **do** hot-reload on the running dev server: `backlog/` is an `addWatchTarget` in `.eleventy.js` and `src/_data/backlog.js` exports a function that re-globs `backlog/*.md` on every build, so a fresh item normally appears within seconds — **no restart per item.** The footgun is the *trigger*: a file created by a tool/`Write` (atomic write) can be missed by 11ty's chokidar watcher, so no rebuild fires and the new `/backlog/<id>/` `404`s — and a browser reload does **not** help (reload re-fetches 11ty; it doesn't rebuild it). So after creating an item, don't claim it's live until you've confirmed it, and don't "fix" a 404 by editing/renaming the file (it's stale, not wrong). To force the rebuild without restarting the user's server, **touch any watched file** — `touch src/_data/backlog.js` — then re-probe (give it a few seconds; rebuilds aren't instant and an early `curl` races them). Verify against 11ty directly (`:8080`) or the Vite proxy (`:3000`); a clean one-off build (`npx @11ty/eleventy --output=/tmp/11ty-verify`, grep the id) confirms the file itself is valid if the watch path is ever in doubt. Restarting 11ty is a last resort, not the routine fix.
3. **Mark it resolved.** Run **`node scripts/backlog.mjs resolve <NNN> [--graduated-to=<entity>|none]`** — it flips `active` → `resolved`, stamps `dateResolved` (today — the burndown plots this; `check:standards` errors on a resolved item without it), and sets `graduatedTo` if you pass it, in one splice. (This is the *edit* only — run the close-out gate above **first**; the command doesn't.) If the item became a real entity (block/intent/protocol/…), pass `--graduated-to=<entity>` so the graduation trail is recorded. If it resolved **without spawning a new entity** — a `story` that enhanced existing code, folded into a sibling, or shipped as a refinement — pass `--graduated-to=none` (the sentinel) (the body already says what it improved). This is the sanctioned way to acknowledge "no new entity, on purpose" so the close-out nudge below doesn't read as forgotten work. (`task`/`decision` items are exempt from the nudge and need no `graduatedTo` at all; the sentinel is for the resolved `story`/`epic` that genuinely produced no entity.) **Keep `graduatedTo` canonical (#614):** the value must be `none`, a resolving `<kind>:<id>` (`kind ∈ block·intent·protocol·project·plug·capability·adapter·demo`), or a repo path — **the entity reference leads**. A trailing annotation is tolerated but discouraged; put the narrative ("what shipped") in the body, not the field, or entity-graph joins and the G3 lineage walk can't read it. `check:standards` emits one aggregated nudge listing non-canonical values; `npm run normalize:graduated` auto-fixes the safe class (bare id → `<kind>:<id>`) and reports the rest. Re-run `npm run check:standards` to prove the build still passes. Because the file stays, a pointer item keeps its `relatedReport` exposed — there's no orphaned-report risk to guard against.

   **Resolving a `kind: decision`? Promote its rule, then set `codifiedIn` — this is now a hard gate.** A decision item is *case law* — it records one call and its lineage. If it establishes (or refines) a **reusable rule** (a placement test, a naming convention, a monetization line, a boundary), that rule must live in the *statute* layer so the next person cites it instead of re-deriving it: add/extend the named rule in [platform-decisions.md](platform-decisions.md) (or the topical `docs/agent/*.md` it belongs to), then pass it on resolve. **`backlog.mjs resolve` refuses a `kind: decision` that has no `codifiedIn`** — supply it inline with `--codified-to=<doc-path>#<anchor>` (the CLI stamps the field), or for a genuinely narrow one-off that yields no reusable rule pass `--codified-to=one-off` (the sanctioned sentinel, analogous to `graduatedTo: none`). The cheapest moment to capture the orientation is at resolve, with the deliberation still fresh — so the gate fires there, not in a later sweep. `check:health` flag **G6** remains the catch-up pool for the **legacy** decisions resolved before this gate existed (it should only ever shrink). This is the discipline that keeps the decision chain from being the *only* home of the platform's rules (it was 64% case-law-only at the 2026-06-18 sweep). Rules stay **reversible** — a later decision supersedes the statute entry *with lineage*, never erases it.

   **Cite the rule, not the case.** [platform-decisions.md](platform-decisions.md) is the **source of truth** for settled orientation. When a work item relies on an already-codified ruling, cite the **named anchor** (`platform-decisions.md#<anchor>`) — *not* the originating `#NNN`. The decision file carries the lineage for whoever needs the archaeology; everyday items should point at the statute so the *rule* propagates, not a pointer into a deliberation the reader then has to reconstruct. A bare "#NNN ruling" reference to a decision that is already codified is a smell — swap it for the anchor (keep the `#NNN` only when you genuinely mean "the historical call and its reasoning").

4. **If the resolved item has a `parent` epic, review the epic now.** Resolving a child slice changes the epic's state, so a child close-out is **not done until the parent is reconciled** — never leave an epic stranded with only resolved children. Look at the epic and take exactly one of three actions:
   - **More work remains → add the next stories.** Scaffold the follow-on child stories under the epic (`scaffold … --parent=<NNN>`) so the umbrella keeps live slices. The epic stays `open`.
   - **More work remains but can't proceed → mark the epic blocked, with a reason.** If the remaining slices are gated, set the epic's `blockedBy` (and, if it now has no open children, a `childlessReason: blocked|untriaged|program`) so the tile shows *why* it's stalled rather than looking abandoned. The epic stays `open`. **Pick the reason by what's actually gating it.** If the slices are gated on an **unresolved design fork**, that fork is a separate `kind: decision` item (see the conflation rule under *Rules*): the epic is `childlessReason: blocked` + `blockedBy` that decision — *never* an inline reason on the epic itself. **`childlessReason: undecided` is retired and `check:standards` errors on it** — an epic must not encode "a decision lives here" as its own state; the decision is its own item and the dependency is a `blockedBy` edge. And once the design *is* settled but the **slice breakdown isn't scoped yet** (e.g. a mechanical audit must run before the slices can be carved), that's just an **unsliced** epic: leave it with *no* `childlessReason` so it shows the **slice** badge (its first slice is the audit) — "decomposition pending" is the unsliced state, not a reason value. So the moment a gating decision resolves, drop the epic's `blockedBy`/reason and let it fall to unsliced.
   - **Scope is delivered → resolve the epic.** If that child was the last open slice and nothing more belongs under the umbrella, run the close-out gate on the **epic** too and `resolve` it (`--graduated-to=<entity>|none`). Don't let a finished epic linger `open`.
   - **It's a perpetual program, not a finite deliverable → mark it `ongoing: true`.** A few epics are *continuous programs* that are never "done": between slices they legitimately have every child resolved, yet resolving them would be wrong — the next slice arrives as the program advances (the flagship-exercise-apps program [#314](/backlog/314-flagship-exercise-apps/) — which keeps adding apps/loop turns; its *individual* app children like #317/#318 are finite builds that **resolve**, not programs themselves; or a recurring sweep/benchmark). **Don't set this flag on a hunch — apply the four-part *Program Test* ([Programs](#program-definition) above); `ongoing: true` is the cheap flag, the test is the bar that keeps it honest.** Set `ongoing: true` in the epic's frontmatter. It stays `open`/`active`, and **both** the gate and the app stop treating "all children resolved" as a resolve cue (see the badge note below). Distinct from `childlessReason: program`, which is for a *childless* continuous program — `ongoing: true` is for one that *has* (resolved) children between slices.

   The `/backlog/` tile surfaces the cue: a storied epic whose every child is resolved shows an **`all slices done`** badge (review it), and a `resolved` epic with an open child shows a red **`⚠ open slice`** badge (a contradiction to fix). Either badge means this step was skipped — **unless** the epic is `ongoing: true`, which instead shows a blue **`ongoing program`** badge and is never flagged as to-resolve (`check:standards` skips the nudge; `backlog.js` gives it `epicState: 'program'`, a non-actionable state excluded from the "to resolve" count).

   **Before resolving an *epic*, the umbrella must own no uncarved scope — not in children, AND not in its own body.** The `⚠ open slice` guard only catches a live *child*; the subtler failure is closing an epic whose **prose** still describes work that was never sliced (#777 resolved with its one child done while four migration slices lived only as a "not carved yet (gated on #765)" list — so the gate cleared but the epic shipped nothing and read as complete). So at epic close-out, walk the body: **every scope item must be either checked-and-cited (it shipped → name the child/artifact that delivered it) or carved into a tracked child** (an open slice, or a `blockedBy` edge to a real prereq). If any scope is still uncarved, you have only two honest moves — **carve it** (scaffold the slice) **or keep the epic open**; resolving over it is never one of them. `check:standards` enforces the body half: an unchecked `- [ ]` box in a resolved epic is an **error** (check it once shipped, or reopen), and forward-looking uncarved-slice language ("not carved", "carve once/after/later") is a **warn** (confirm the scope shipped or is a deferral that cites its tracking `#NNN`).

> Deletion is no longer the close-out step — `resolved` is. The only time you remove a `backlog/<id>.md` is to discard an item that was opened in error or fully superseded, not to record completed work.

## Keep the blocker DAG honest — re-evaluate `blockedBy` whenever you touch the backlog

`blockedBy` only earns its keep — deterministic readiness tiers (#249/#250), the dependency graph, "what's the critical path to ready work" — if the edges stay **current**. The failure mode is silent: prerequisites stated in prose ("after #NNN", "blocked on…", "needs X from #NNN", "once Y ships") never get lifted into the field, so items show as agent-ready (Tier A) when an unfinished dependency actually gates them. So **re-evaluate edges at every seam**, not as a one-off audit:

- **When you author or close-out an item (including leftover spin-offs):** before saving, scan the body for every prerequisite phrasing and lift each into `blockedBy: ["NNN", …]`. A genuine blocker is a **hard prerequisite** — the item *cannot start or complete* until the blocker resolves (it consumes an artifact/API/registry/protocol the blocker produces, or a `decision` the blocker ratifies). It is **not** mere "see also" (that's `crossRef`) or epic grouping (that's `parent`). Record real prerequisites even if the blocker is already `resolved` — the lineage is correct and a resolved blocker leaves the item Tier A. Be conservative: a false edge wrongly hides ready work, so when in doubt, leave it out and keep the prose note.
- **When you claim/resume an item:** sanity-check its own `blockedBy` against reality. A listed blocker that has since resolved is fine (leave the edge; the tier logic handles it). But if the body reveals a real prerequisite that was never declared, **add it now**; if a listed edge is spurious (not actually a prerequisite), **remove it**. A claim is the cheapest moment to correct the item you're already reading.
- **When you resolve an item:** you don't hand-edit dependents — the loader recomputes readiness from the edge set, so resolving an item automatically frees whatever listed it. Just confirm the items this work *created or touched* carry correct edges.
- **Discipline, not blast radius:** edit only the items you're already working; don't sweep the whole backlog mid-task. `check:standards` **errors** on an unresolvable, self-referential, or cyclic edge, so run it after edits — a green gate proves the DAG is still acyclic.

## The digest — every item's lead paragraph; keep it an accurate one-glance summary

Each item's **first paragraph is its digest** — the loader derives `item.summary` from it (no separate field, so it can never become a stale *copy* of the body; it *is* the body). It's what selection reads to rank without opening every item, and what the shortlist / index / graph tooltips surface. Because a fast, trustworthy digest is what makes selection quick, two gates keep it healthy — one mechanical, one human:

- **Mechanical (`check:standards`):** the lead paragraph is a **required field** — a missing/empty one is an **error**. A paragraph over `DIGEST_MAX_WORDS` (100) **warns** — keep the digest to ~2-3 sentences ("what + why"), not the full body; the detail lives below it under `## Build` / `## Acceptance`.
- **Human/agent (review-time) — the part tooling can't judge:** a derived digest can't drift as a *copy*, but its **content** drifts when you rewrite an item's scope and leave the opening paragraph describing the old plan. So **whenever you author, claim, work, close out, or review a story, re-read its lead paragraph and refresh it if it no longer matches the current item.** Authoring a new item (including a close-out spin-off) means *writing* a real digest, not letting the first sentence of detail stand in for one. Reviewing a candidate during selection means *glancing* at its digest and correcting it on the spot if it's stale — this is the cheapest moment, since you're already reading the item. Until this is second-nature across every skill, treat "review a story" as implicitly including "review its digest."

## Selecting the next item to work on

> Use when asked "what's next?", "pick the next backlog item", "what should I work on?" — or via the `next-backlog-item` skill. The goal is the item an **agent can implement now**, not just the most interesting one.

**Gather — run the deterministic ranker first; do not re-derive the tiers by hand.** The readiness
rubric (tier, batchable, leverage) is already computed **once** by the loader (`src/_data/backlog.js`)
and surfaced on the `/backlog/` **Prioritisation** tab. Re-globbing `backlog/*.md` and re-tiering each
item in prose is the slow, drift-prone path that the tab makes unnecessary — it was finding 2 batchable
items in minutes where the loader lists 23 instantly. So **step 1 is a single command**, not a file sweep:

```
npm run check:readiness -- --select          # human-readable ranked view (the same data the tab shows)
node scripts/check-readiness.mjs --json       # machine-readable: { selection: { counts, tierA, batchable, tierB } }
```

It prints, instant and **identical to the tab** (same loader → zero desync): the open counts, the
**Tier-A** items ordered (leverage desc → smaller-first → NNN), the **batchable**
subset (small Tier-A), and **Tier-B** decisions ranked by leverage. Take the ranked list from here —
**do not recompute `tier`/`batchable`/leverage**; the loader already did, deterministically. The CLI
already keeps only `open` items (it drops `active`/`resolved`/`parked`), so the only thing left to read
by hand is each *shortlisted* item's body, for the one judgment a field can't make (below).

**Concurrency is real — the snapshot goes stale.** Multiple agents work this backlog at once, so the `status` you read while gathering can be claimed by another agent before you finish presenting. Two guards, both cheap:
- **Ownership is `status`, never the working tree.** Concurrency is owned by the `status: open → active` transition (and the `reserve` session soft-holds, #083) — **not** by git. A racing agent is detected by the item reading `status: active`, *not* by uncommitted edits. **Never run `git status`/`git diff` to judge whether an item is taken or eligible, and never drop an item because it (or the tree) has uncommitted changes** — a perpetually-dirty working tree is the normal baseline here (the user owns pushing; commit your own pieces per *Rules* → *Commit each finished piece of work — never push*).
- **Re-read fresh right before you commit to it.** The instant before you present the single recommendation — and again the instant before you claim it (see *Working an item* → *Claim it on start*) — re-`cat` *that one item's* frontmatter from disk (not the bulk snapshot). If it now reads `status: active`, you lost the race: drop it and fall to the next Tier-A item. (Commit state is irrelevant — don't check it.)

**Refine the top of the ranked list — the CLI gives the structural tier; you add only the prose
judgment.** The `--select` ranking is the deterministic core of the rubric (type + resolved
prerequisites + size). The two signals a field *can't* decide — does the body actually read as a
buildable plan, or does it hide a design fork — are still yours, but you apply them **only to the
shortlist** (the top few candidates you're about to present), not to all 100+ items. The test is
*"could an agent open a PR for this today, without a design call?"*:

| Signal | Ready ↑ | Not ready ↓ |
|---|---|---|
| `kind` | `story` / `task` (a concrete build) | `decision` (open fork) |
| Body verbs | "Implement / Roll / Add / Align / Extract" + **named file paths** + acceptance criteria | "Decide whether / Open sub-decision / alternative held open / held open" |
| `relatedReport` | a **plan** or an IMPLEMENTED / self-contained-handoff report | an open exploration with unsettled options |
| Prerequisites | none, or "now that X is proven", or every item it references is `resolved` | references an unresolved item, "blocked on …", "surfaces the moment …" |

**The tiers (what the CLI's structural pass assigns — your prose refinement can only *demote*, never promote):**
- **A — agent-ready:** a non-decision build (`story`/`epic`/`task`), prereqs resolved — the loader's `tier: 'A'`. **Pick the next item from here.** (A buried fork the field couldn't see drops it to C — that demotion is your shortlist pre-flight, below.)
- **B — one nod away:** a `decision` that already states a recommendation — needs only the user to ratify, then it's a quick edit/build. Offer it; don't auto-run it.
- **C — needs design / not agent-ready:** open `decision` forks, anything blocked. Surface for discussion, never auto-pick.

**Order within Tier A** is already applied by `--select` (leverage desc → smaller-first → NNN — so chain-unblockers and small clear builds float up). Layer only the judgment the sort can't encode: prefer clearer acceptance criteria, and cluster around recently-shipped work for momentum.

**Dependency check.** An item that says "see the `<other>` item", "prerequisite", or "after X" is Tier A only if that prerequisite is `resolved`; otherwise it's blocked — surface the prerequisite instead.

**Output (don't start work yet).** Present a short ranked shortlist (top 3–5, grouped by tier, one-line rationale each), then the single recommended next item with its reasoning, then ask whether to start. Follow the planning-as-discussion style — recommend, don't rapid-fire multiple-choice.

**If the recommended item is a `decision` (Tier B/C), claim it, then present the decision to be made — not a build.** A fork is a call to make, not a build to start — but talking it through is itself claimable work, so **claim it first** (see *Working an item* → *Claim it on start*: re-read to win the race, flip `open → active` + `dateStarted`, rename the chat) so a concurrent session won't discuss the same fork. *Then* lay out the open fork / human judgment it needs, the realistic options with their tradeoffs, and your recommendation with reasoning (planning-as-discussion; expect follow-up). If the user redirects, release the claim (`active → open`) and claim the one they want. Once the call is made **and _explicitly ratified_** (see *the ratification gate* under *The decision item is the source of truth* — agreement on a sub-point is not ratification), close it out (`resolved` + the ruling) — its successor work then turns agent-ready, so continue or re-run selection. This is the same framing as *When nothing is agent-ready* below; the difference is only whether a ready item also exists beneath it.

**The item is the source of truth — every refinement gets baked into the item itself, never left only in chat; and never ask permission to do it.** This is the general **story rule** (it holds for any work item, sharpened here for decisions): the conversation *talks the work through*, but the item is where it *lives*. So the moment the discussion produces **anything that refines the item** — a fresh finding, a reframed fork, a dissolved or newly-added option, a changed recommendation, **or an illustrative example / clarification the user asked for** — **write it into the item in the same turn you say it**. This is **update by default, no permission step**: a clear refinement flows straight into the item — the same fix-on-the-spot rule as the *Fork-readiness pass*, now applied **live during the discussion**, not just at claim time. **But a refinement that flips a default, rewrites a contract, or changes a classification is a *new claim*, not transcription — attack it in the same turn you write it** (*Red-team the default → Red-team each update*). Banking unchecked reframes for a terminal sweep is what lets a late red-team reverse the whole session at once; an at-write-time inline attack reverses at most the one edit. High-leverage reframes (statute-codifying, contract-dependent) escalate to the skeptic sub-agent *at the edit*, not at resolve.
- **Never end a turn with "want me to add this to the item?" / "should I capture that?"** — capturing *is* the default action, not an opt-in you offer. Bake it in, then *report that you did* ("folded the example into Fork 2"). Offering it as an optional follow-up is the exact slip this rule kills: an answer that improves the item but lives only in chat is lost to the next session.
- **A user question is a refinement request.** If the user asks for an example, a sharper framing, a clarification, or a worked-through detail, the answer belongs *in the item* (e.g. an illustrative before/after sample per the *concrete-code* check) — give it in chat *and* bake it in, same turn.
- **For a `decision` specifically: never ask the user to ratify anything that lives only in chat.** The thing the decider ratifies is always the item's current text. If your chat message proposes an option, reframe, or recommendation the item doesn't yet contain, the item is **stale** and any ratification against it is invalid — write it in, *then* ask. When the reframe is big enough to be its own decision (e.g. it reopens a `resolved` upstream call), capture it *as* a fork/note and file the spin-off per *Closing out*; don't let it exist only as chat.
- **Mirror the item's default — never *voice* a recommendation the item doesn't already state (#1935).** The presenter-side of the source-of-truth rule, and the one most easily skipped: in the discussion your stance **is** the item's **bold default** — read it and present *that*, not a fresh stance you formed in chat. The moment you catch yourself recommending something the body doesn't say — a different option, or a *sequencing/sub-choice the item left as an aside* — **that is the edit-the-item signal, not a thing to say.** Promote or flip the fork and re-run its red-team *in the item*, then present the item's now-current default. A chat recommendation that diverges from the body always means one of exactly two things, both fixed in the item, never in chat: the item left a choice **un-prepared** (a residue — finish its prep, see *The prepared-fork shape → no live choice outside a fork*), or the prep went **stale** (re-prepare it — next rule). The decider must never see a stance the item doesn't carry; if `prepare` and the decision turn ever reach *different* conclusions, that is the defect, not a judgment to reconcile (a decider only ratifies the item's default — there is nothing to diverge *from* unless a choice was left un-prepared).
- **The only pause-to-ask case is a genuinely ambiguous conclusion** — the new element doesn't cleanly settle which way the call goes. Then surface the ambiguity, get the user's steer, *then* record it. Everything clear gets baked without asking.
- **Baking in is not appending — when a refinement changes the *leading answer*, sweep the whole story for what it now contradicts.** Adding a fresh finding is additive; a **reframe, a reversal, a dissolved fork, or a changed recommendation is subtractive** — it falsifies statements already written elsewhere in the item. Writing the new ruling into a *new section at the bottom* while the digest, the summary lead, the "recommended path at a glance" table, each fork's **bold default**, the axis-framing, and the verification notes still assert the *old* answer leaves the item **self-contradicting** — and a reader (or the decider, or the next session) hits the stale default *first*, before ever reaching the new section. So every leading-answer change triggers a **whole-item integrity pass in the same turn**: re-read the item top-to-bottom and reconcile *every* place the prior answer appears — flip or strike-through (`~~…~~ → **new**`) each superseded default, table cell, and summary line, and fix the digest/`summary` so the one-line story matches. The operative default must read the **new** way **everywhere it appears**; the old reasoning is kept only as explicitly-marked **superseded audit**, never as a live-looking default. The integrity bar: **a reader scanning the item in order must never encounter the old answer presented as current.** Don't wait to be told the table still says the opposite — catch it on the same turn you reframe. (Cheapest enforcement: after a reframe, `grep` the item for the old verb — "retain", "keep", the prior default's keyword — and confirm every hit is struck or recast.)
- **Resolving a `decision` requires _explicit ratification_ — never infer it from agreement.** Baking refinements into the item (the rule above) and *resolving* the item are two different acts, and the second has a hard gate: **flip `active → resolved` only on an explicit ratification utterance from the decision-owner** — "ratify", "approved as written", "lock it in", "resolve it", "ship it as ruled", or an equally unambiguous go. A user agreeing with a *sub-point* ("agreed", "yes that's right", "makes sense", a nod to one fork, or a clarifying question you answered) is **discussion, not ratification** — it refines the item, it does **not** close it. The slip this kills: reading "agreed" on one fork as a green light to resolve the whole decision. So: **(1)** write the call into the item as a **`## Proposed ruling — AWAITING RATIFICATION`** block (each row prefixed *proposed*, not *ratified*), keep the item `active`, and **state in chat that it's staged for their explicit go**; **(2)** present the staged ruling for a last look; **(3)** resolve *only* after the explicit utterance — then rewrite the block's "proposed" → the recorded ruling, stamp `dateResolved`, and run `resolve`. **Before that resolve, run the *Red-team the default* pass (below)** — attack the branch you're about to ratify; a stamp is not a check. **Pre-filed spin-off builds stay `blockedBy` the decision until that ratification** — never let them enter the backlog as ready Tier-A work off an un-ratified call. When in doubt whether a reply ratifies, **ask one direct confirming question** ("ratify this as written?") rather than assuming. **Once ratified, the call is immutable — never reshape a ratified decision in the same breath (#1886).** If a red-team or any finding lands *after* the ratification utterance, you do **not** re-open and amend the ratified forks: the decision stands as ratified, codify it faithfully, and route the late finding to a **new follow-up item** (a reconciliation `decision`, `blockedBy`/`parent` the original) — never a retro-edit of the decided item. A statute *cross-reference* added when codifying (citing a sibling anchor for lineage) is not an amendment; *changing the ruling's framing or discriminator* is, and is forbidden post-ratification. This is the structural reason the statute-overlap check belongs in *prep*: after the human has spent their ratification, the only honest moves are "codify as ruled" or "file a new item" — reworking the call is off the table.
- **The _only_ sanctioned auto-ratify: there is no other valid fork.** You may resolve a `decision` *without* the explicit utterance **iff** the fork-existence test (*Support all coherent; the fork-existence test*) collapses to **exactly one** coherent branch — every alternative is genuinely flawed / won't-work, so the "decision" is *applying* a forced outcome, not choosing. There's nothing to ratify because there was nothing to decide. The moment **≥2 branches are genuinely valid** (a real either/or — even one the user might reasonably pick differently), it is a real decision and resolving it needs the explicit utterance above; a recommendation, a confidence score, and a red-team that *amends* a branch all signal a live fork, **never** an auto-close. Invocation context does **not** widen this: entering decide-mode via `/decision` and the user saying "go" selects the *item to discuss* — it is **not** standing authorization to resolve+commit whatever ruling you then form (the #992 slip: "go"/"ready" read as ratification of an amended two-fork call). Hold the commit, the resolve, *and* the spin-off scaffolding behind the gate.

**Fork-readiness pass — before presenting a decision, make the item explain its fork concretely; fix it on the spot if it doesn't.** You've just claimed it and are reading it to present it — the cheapest moment to improve it (same fix-on-the-spot rule as *The digest*). A decision item earns a good ruling only if the fork is legible, so check these, in order, and *edit the item* (not just your chat message) where it falls short:
- **Verify the `preparedDate` stamp is truthful before you trust it — a stamp is a claim, not a guarantee.** Walk the *live* (still-open) forks: each must be in prepared-fork shape — named options + tradeoffs + a **bold** default, grounded in the linked `/research/` topic. If a fork is raw despite the stamp — flagged "needs a human call", named with a slash (`webpush`/`webpermissions`), or otherwise missing its options/default — the item is **mis-stamped**, not ready: finish its prep in-place (the checks below, or hand off to `/prepare`) **before** presenting it to ratify. Never present a half-prepared decision as ready-to-ratify just because `preparedDate` is set; a false stamp is exactly how an un-researched call reaches a decision turn dressed as a nod. (Worked example: [#009](/backlog/009-gap-13-webpermissions-project/) carried a `preparedDate` + a research report, yet its surviving naming/ownership fork was flagged "needs a human call" with no options — the stamp was false; the fix was to shape that fork to DoR before deciding.)
- **A stale prep is re-prepared *before* the first presentation — staleness is caught at claim, not patched mid-ratification (#1935).** `preparedDate` certifies the research was current *when stamped*; if the tree moved since (a cited `file:line` shifted, a precedent changed, a `blockedBy` resolved, a sibling decision landed), the bold default may no longer be factual. So the *first* thing you do after claiming — before presenting anything — is re-validate the live forks against the current tree. If a fork has gone stale, **rewrite it + re-run its red-team into the item now**, so the decider's *first read* is already a fresh, attacked, source-of-truth item. Never present stale prep and repair it during the discussion — that reintroduces the chat-runs-ahead-of-the-item divergence the source-of-truth rule forbids. (Same move as a discussion-born reframe, only triggered by staleness at claim rather than by the conversation.)
- **Prior-art research first — never form a stance cold.** Before you name options or recommend a default, do the research the decision rests on; an ungrounded stance is a guess, not a ruling. Match the research to the fork: a decision that **authors or designs something greenfield** (a new intent/block/plug/protocol/adapter, or any "no design exists yet" standard) **must** run [design-first.md](design-first.md) step 1 first — survey browser standards (MDN/WHATWG/W3C/WAI-ARIA APG) and the benchmark libraries (`references.json`: MD3, Carbon, Radix, Fluent, Apple HIG, plus the relevant ecosystem libs), reuse the platform's existing vocabulary, and **publish the findings as a `/research/` topic** — a `researchTopics.json` entry + a `src/_includes/research-descriptions/{id}.njk` write-up ([research-workflow.md](research-workflow.md)), keeping the session `reports/{date}-{slug}.md` as the artifact and linking it via `relatedReport`. Decision research belongs in the **`/research/` discovery surface** (where it's browsable *as research*) and is **referenced from** the decision item — not only mirrored off it through `relatedReport`. Cite the topic (and the surfaced options) in the item. (#64's survey is published as `/research/tree-select-hierarchical-selection/`.) A decision that only **ratifies existing code** still needs the concrete-refs check below but not a web survey; a decision over **already-researched** ground just links the prior report. Research routinely revises the stance and surfaces forks the item omitted — expect to rewrite the options after it, and do **not** present the decision until it's done.
- **Concrete code to look at — not just an authored snippet.** Whenever the fork concerns code that *already exists* (an impl to change, a call site to migrate, a contract to ratify against what's shipped), the body must point at the **real code with `file:line` references the decider can open** — e.g. `frontierui/blocks/droplist/AutoComplete.ts:116` (the actual `aria-controls` wiring), not just a hand-authored sample. Go read the relevant code at claim time and cite it; the decision is only as good as the real code the decider can inspect, and an authored snippet alone hides whether the fork matches what's actually there. Author a minimal illustrative sample *in addition* where it sharpens the choice (a before/after, a side-by-side the way #281 got HTML⇄JSX tables), but it never *substitutes* for the concrete references. For a purely greenfield fork (no code exists yet), the smallest candidate sample that makes the choice tangible is enough. Either way the example/refs belong *in the item*, not only in chat.
- **Verify the item's grounding claims before endorsing its default — don't inherit a prepared item's assertions as facts.** A prepared item's bold default rests on *claims* — a cited precedent ("mirrors the MaaS `serve()` precedent: handler + contract in WE"), a contract-vs-impl classification ("a pure reference handler → WE"), a "this is the same shape as X" analogy. **Trace each load-bearing claim to the real tree before you ratify against it**, because the recommended default inherits any error in them. Two checks that catch the common miss: (1) **precedent citations — open the cited code and confirm it actually says what the item claims.** (A real miss: #730's C1 default leaned on "MaaS `serve()` lives in WE"; tracing it, the MaaS serve handler lives in the *impl* layer `blocks/renderers/module-service/`, not a WE plane — the precedent argued the *opposite*, and C1 collapsed once traced.) (2) **placement/classification claims against the guiding principle** — a "reference handler / pure dispatcher → WE" classification is a *contract-vs-impl* call: apply the test (code that **defines** a contract or conformance → WE; code that **delivers** a capability — a registry-dispatching, artifact-producing handler — → FUI, per *impl-is-not-a-standard*). A running handler is impl; "it's a small pure reference" does not make it a standard. Catch a layer violation like keeping implementation in WE **on first read**, not after the user pushes back. Where a traced claim is wrong, **fix the fork in the item** (correct the default + its rationale) before presenting. (At *ratify* time this generalizes into the *Red-team the default* pass below — from tracing the cited claims to attacking the whole branch.)
- **Standard-layer grounding before mapping vocabulary onto a component — grep the standard layer, not just the impl tree.** A recommended default that maps a domain word (a **badge / pill / label / status / chip / tag**, or any UX noun) onto a component is making a *standard-layer* claim, and must be grounded there **before** the mapping is endorsed: grep `src/_data/intents/` and [platform-decisions.md](platform-decisions.md) for an intent or decomposition that **already owns that vocabulary**, and cite the owning intent in the item. The miss this catches: grounding that checks only the *impl tree* (does a `badge` component exist? what props does it take?) silently re-conflates vocabulary the **standard layer already split** — a badge/pill/label is not one thing. (Worked example: #1621's prepared default put taxonomy *pills* on the *status badge*, re-merging exactly what #1319 had split into **Status Indicator / Tag / Notification Marker**, because the grounding pass read the component and skipped `intents/`. The fix is one grep: a `tag`/taxonomy mapping cites the Tag intent, a state mapping cites Status Indicator — they are different intents, so they are different components.) A vocab→component mapping that **cannot cite an owning intent** is either ungrounded (do the grep) or a genuine gap (a new intent is itself the fork) — never a silent default.
- **Options stated with tradeoffs.** Each realistic branch is named with its tradeoff and (where you have one) a recommended default — so the reader sees what they're choosing between. Add any branch the body omits.
- **Short, concise, well-explained.** Tight digest (≤100 words, "what + why"), body that explains the fork without rambling — trim bloat, sharpen vague sentences. Concise *and* complete: don't drop the example or a real option to hit brevity.

Improving the item here is durable capture, not busywork: the next reader (or a fresh session) inherits a fork that's ready to decide instead of one that needs re-explaining.

**Re-run `check:standards` after a substantive body rewrite — and always before `resolve` — not after every trivial edit.** The gate validates an item's *structural / rendering* shape (link syntax, raw HTML, frontmatter, digest length, blocker-DAG, decision/epic conflation), **not** the correctness of the ruling — that's the discussion + *Red-team the default*'s job. The shape footguns it catches render **live on `/backlog/`**: a `[[wiki-link]]` (memory-files-only) prints literally, and raw interactive HTML (`<details>`, an unclosed `<select>`) can swallow the rest of the page — so a prose-heavy decision rewrite is exactly where they slip in. The whole-repo gate doesn't run per-edit (it's an on-demand scan). For per-edit feedback there is a **scoped per-item lint** (#845): **`npm run check:item -- <NNN>`** (or `node scripts/check-backlog-item.mjs <NNN-slug>`) runs *just* these rendering/structural checks on one `backlog/<id>.md` — the same `lintBacklogItemRendering` findings the gate emits, plus the frontmatter-colon and per-item `blockedBy` checks — and exits non-zero on an error. It's cheap enough to run on every update, or to wire as a `PostToolUse` hook on `backlog/*.md` so the footguns are caught the moment they're written:

```jsonc
// .claude/settings.json — opt-in: lint a backlog item on every Write/Edit to backlog/*.md
"hooks": {
  "PostToolUse": [{
    "matcher": "Write|Edit",
    "hooks": [{ "type": "command",
      "command": "f=\"$CLAUDE_TOOL_FILE_PATH\"; case \"$f\" in */backlog/[0-9]*.md) npm run --silent check:item -- \"$(basename \"$f\" .md)\" || true;; esac" }]
  }]
}
```

The whole-repo `check:standards` stays the authority for the cross-entity checks the single-file pass can't see (graduatedTo/relatedProject resolution, the `blockedBy` cycle walk, dup ids, decision/epic conflation) — run it after a real body rewrite and always before `resolve`, as above.

### A ratified decision is reversible — stay agile, don't ossify

As the constellation matures, **more and more decisions carry a `resolved` ruling — and a growing fraction of them will eventually be revisited or reversed. That is normal and healthy, not a failure of the original call.** A ruling was the best read of the evidence *at the time*; new findings, a shifted boundary, a downstream build that exposes a wrong assumption, or simply the user changing their mind are all legitimate grounds to reopen it. Treat the backlog of ratified decisions as **agile and revisable**, never as a frozen contract you must defend at all costs.

This cuts **both ways**, and holding both is the point:

- **Push back first — defend the standing ruling on its merits.** When a new request cuts against a `resolved` decision, don't silently flip. **Surface the prior decision, explain *why* it was ruled that way** (cite the item `#NNN`, its rationale, the principle it served), and say plainly that the new ask reverses it. The standing ruling has earned a reasoned defense; make the user choose against it with eyes open, not by accident. This is the same instinct as the *Red-team* pass, pointed at change rather than at the original default.
- **But stay genuinely open — change directly when the case holds.** Pushing back is *advocacy, not obstruction*. If the user's reasoning is sound, or they simply decide the tradeoff differently now, **reverse the call without friction** — don't re-litigate a settled-then-reopened point past the first honest defense, and don't treat your own prior recommendation as something to protect. Agility beats consistency: a coherent reversal today is worth more than a stale ruling defended out of inertia.

**Mechanically, a reversal is a normal decision turn, not an erasure.** Reopen the decision item (`resolved → active`, or file a successor `decision` that supersedes it with a `crossRef`/`blockedBy` to the original), record the *new* ruling **and a one-line "supersedes the #NNN ruling because …"** so the lineage stays honest, then re-run the ratification gate (explicit ratification + *Red-team*). Never just overwrite the old ruling as if it never existed — the audit trail of *why it changed* is itself valuable. Downstream items the original unblocked may need re-checking against the new call. (Heavier-weight categories already lean this way — monetization/tiering rulings are explicitly *soft-accepted & revisitable* — but the agility applies to **any** ratified decision; standards rulings just carry a higher bar for what justifies the reversal.)

**Correct flawed archive decisions lazily, on citation — the statute layer is the proactive audit surface (#2091).** When a *class* of authoring flaw is discovered across the resolved-decision archive (e.g. the two-confusion class above), do **not** mass-rewrite history: the archive is lineage, and the standing rule already says the statute — not the decision chain — is what gets cited. Audit **proactively** only where flaws leak forward: the `platform-decisions.md` anchors (one bounded pass per flaw class) and the **currently-prepared, un-ratified stock** (the items about to land on the human's desk). A flawed *resolved* decision is corrected **when something cites it** — the normal reversal mechanics above — never as a bulk sweep.

### Red-team the default — attack the recommended branch before resolving (#766)

The fork-readiness pass and the `✓ ready to ratify` stamp are a **completeness skim** ("options stated, fork legible, stamp truthful"); neither is an **attack**. That gap is real: `✓ ready to ratify` is the strongest-*sounding* signal but the weakest-*verified* one — it is just `preparedDate` being set, a self-assertion by whoever prepped — and the same session that authored the bold default then ratifies against it, with the same blind spot in both seats. #730's C1 default carried the stamp, a `relatedReport`, and concrete `file:line` refs and **still** shipped a default that put a running handler in the standards repo; every mechanical gate passed and a human caught it. So **before you flip a `decision` `active → resolved`** (the *ratification gate* above), run one adversarial pass on the branch you're about to ratify:

- **Attack the *classification* before the value — is this even the right *kind* of question? (the #1892 axis).** The skeptic's first job is not "is the default wrong?" but "is this framed as the right *kind*?" Both #1892 misses survived a skeptic that only attacked values: Fork 2 (a config dimension wrongly framed as a ratifiable fork) and Fork 1 (a residue classification run over a contract narrowed to fit the answer). So the skeptic must try to **re-route the question**: (i) is this `## Fork N` actually a [config dimension](#config-extends-platform-default) (two values of one knob → not a fork), a *support-both*, or *settled by precedent* — i.e. **not a ratifiable fork at all**? and (ii) if the classification turns on a contract (residue/which-layer), is the contract the **coherent-standard** one (#1826), or one **narrowed to make the test pass**? A skeptic that grants the framing and only haggles the default leaves the deeper error untouched.
- **Merit-basis attack — is the fork (and every tradeoff you kept) decided on *merit*, or on effort/cost/timing? (the #2010 axis).** The *not-a-prioritization* rule (below) and `check:health` G4 are the *prep-time* guards, but they are a keyword-shaped tell-scan and a self-assertion — a prepared fork can pass both and still reach ratification with a **cost discriminator wearing merit's clothes**, because the same session that framed it ratifies it. So the ratify-time skeptic runs the *not-a-prioritization* screen as a live attack, at two levels: **(i) the whole fork** — imagine both branches *free to build and instantly maintained*; does a **merit** difference (correctness · composability · a11y · UX · lock-in · consistency-with-a-sibling-precedent) remain? If none, the "fork" is a ruling-plus-backlog-ordering — **dissolve it** to *support-both + a separately-prioritized build*, don't ratify it as a fork. **(ii) every *surviving* per-branch tradeoff** — re-express each listed downside as a named merit axis **or** a cost (maintenance · drift · sequencing · "smaller diff" · "cheaper" · "fewer sites to touch" · "another artifact to keep in sync"); if it can *only* be phrased as a cost, **strip it and re-derive the default on merit**, because an effort con biases the on-merit call toward the cheaper branch. The tell that this axis was skipped: two branches that are **behaviourally identical** whose choice is justified by diff size — behavioural equivalence means a *principle* must decide, never cost. **Attack lands → rewrite the item's rationale to the real merit principle (or dissolve the fork) before ratifying**, exactly the fix-on-the-spot the *not-a-prioritization* rule demands — never ratify over an effort-framed default. (Worked example: [#2010](/backlog/2010-parity-wiring-seam-how-customtemplatetypes-activation-reache/) shipped prepared with Fork 1's (a)-over-(b) resting on "smallest diff / cheap / triples the edit" — (a) and (b) were behaviourally identical, so the ratify-time merit attack regrounded the call on **sibling-parity** (its sibling registry already rides the injector chain) + **open/closed extensibility**, the cost language stripped, before the fork was ratified.)
- **Argue the opposite of your own bold default** — state the *strongest* case for the main alternative, not a strawman. If you can't make the alternative sound reasonable, you haven't understood the fork.
- **Try to name the guiding principle the default violates** — run the **chosen branch** (not the item) against the relevant principles one line each: *impl-is-not-a-standard*, *npm-scope-mirrors-layer*, the A–E catalog (#607). This is the *verify-grounding-claims* check (above) generalized from "trace the cited claims" to "attack the whole branch."
- **Statute-overlap check — diff the rule you'd codify against the existing statutes, *in prep* (#1886).** A decision that will set `codifiedIn` is about to *write or refine a standing rule*; the attack that bites latest and hurts most is **a rule that silently collides with an anchor already in [platform-decisions.md](platform-decisions.md)** governing the same turf (the same attribute, mechanism, vocabulary, or layer) by a *different test*. Catch it where it's actionable: **draft the codified rule's core claim, grep `platform-decisions.md` for anchors on the same subject, and reconcile any overlap inside the item — cite the sibling anchor and state how they compose — *before* `preparedDate` is stamped.** The cost of finding this at *resolve* is the failure #1886 hit: a prepared decision codifying "a card's root is intrinsic-vs-extrinsic" reached ratification before anyone noticed [composition-preserves-a11y-contract](platform-decisions.md#composition-preserves-a11y-contract) already judged author-set `as=` by an *a11y-contract* test — so the human spent a ratification turn on a rule carrying an unreconciled statute conflict, which is exactly what prep exists to prevent. The skeptic sub-agent's prompt must include this axis ("does this contradict or duplicate an existing platform-decisions anchor?"), not only "is the default wrong on the merits."
- **Citation-scope check — when a default cites an anchor to *authorize* a branch, confirm the cited rule's authoring scope actually reaches the case (#1932).** Distinct from the *grounding-claim* check (does the cited code/precedent say what the item claims?) and the *statute-overlap* check (does the rule I'd codify collide with an existing anchor?): this attacks a default that **leans on a statute as authority** — "the default is (a) *because* anchor X says so." A rule written for one scope must not be stretched to govern a wider one. The tell: a value-level rule cited to settle an *engine-robustness* question, or a single-author / catalog-evolution rule cited to authorize *cross-author* extension. **For each authorizing citation, read the cited anchor's own scope and confirm it covers this case; if it governs a narrower turf, the citation is *supporting context*, not *authority* — re-derive the default on its own merits.** (#1913 hit this twice: Fork 2 cited the `intents-ux-only:385` *value-default* clause as if it forbade throwing on an unknown **id** — an engine-robustness question `:385` never addressed; Fork 3 stretched the #1318 *open-numbered* rule past its scope. Both defaults were right on merit, but the *authority* was over-extended — a miss that reads as settled when it isn't.) Fold this axis into the skeptic sub-agent's prompt alongside statute-overlap.
- **Outcome rule: attack fails → ratify; attack lands → amend the default, then re-attack** the amended branch. A landed attack is not a veto — it's a correction; re-run until the strongest attack fails.
- **A discussion-born flip clears the *same* gate the prepared default did — the attack guards the branch you're about to ratify, not just the one the item shipped with.** The prep ran design-first research and (ideally) a fork-existence pass; an alternative you *generate mid-discussion* and propose to ratify in its place enters with **none** of that scrutiny, under cover of "we discussed it." So when discussion supersedes the bold default, run this pass on the **new** branch before resolving. Treat **"more complete / more reliable / more robust"** as an explicit trigger phrase: completeness bias completes the *wrong layer* just as easily as the right one — re-run the per-fork classification (esp. *mechanism-vs-UX-policy* and *impl-is-not-a-standard*) against the supersession, because a flip that adds standard vocabulary for what is actually consumer policy (or impl) is the classic miss. (Worked example: #1407 Fork 2 — a mid-discussion proposal to promote scroll-spy's argmax into a first-class `viewport-presence` selection mode *sounded* more complete; the merit re-derivation showed the recurring/buggy part was UX policy the charter keeps with the consumer, and the prepared composition default held.)

**Weight it to the leverage, don't bureaucratize every nod.** The default is an **inline** reasoning pass the deciding agent does itself — one extra step, cheap, run on every decision. Reserve a **separate-skeptic sub-agent** (a throwaway agent prompted *only* to refute the default and name the principle it breaks) for **high-leverage forks** — wide-open calls, or a decision with a high `gates` count — where prep-and-ratify sharing one blind spot is most dangerous. Do **not** impose a two-agent prep/decide split on every mechanical ratification; the inline attack is the floor, the skeptic agent is the escalation.

**Red-team *each update*, not once at the end — a discussion that edits a decision is a stream of new claims, each owed an attack in the turn it is written (#1892).** The #1437 seat-shift below generalizes: just as the *first* attack on a prepared default must not be the decision turn, the *first* attack on a **discussion-born reframe** must not be the terminal resolve. When discussion supersedes something — a flipped fork, a rewritten contract, a changed classification — **writing that into the item is not transcription, it is a new claim**, so run the **inline attack on it before presenting it as the item's new state**, and escalate to the separate-skeptic sub-agent *at that point* when the new element is high-leverage (codifies a statute, or is a contract-dependent classification per *Contract-first* above). **Do not bank a sequence of unchecked reframes for one terminal sweep** — that is what makes a late refutation feel like it "cancels all the work": it reverses N unattacked edits at once instead of correcting each as it lands. Run incrementally and the terminal red-team becomes a *confirmation* (re-read the per-update verdicts), and any refutation reverses **one** edit, cheaply. (The #1892 failure: two "update the story" turns rewrote Fork 1 patch→residue and dissolved Fork 2 with no write-time attack; the end skeptic then refuted the lot — including a contract-gerrymander an at-write-time inline pass would have caught for one cheap step. The classification insights survived; only the un-incremented rewrites were wasted.)

**The skeptic sub-agent's primary seat is _prep_, not the decision turn (#1437).** A prepared fork must arrive at the decision *already attacked* — `/prepare` runs the skeptic sub-agent on every fork's default and folds the verdict in (a `Skeptic:` line under each `## Fork N`; see *The prepared-fork shape* below and the `prepare-decision-item` skill, pass 4). The reason is the failure mode #1437 exposed: when prep only *asserts* a default and tags "flag for the skeptic", the decision turn is the *first* time anyone attacks it — and an un-attacked default is as likely to be wobbled the **wrong** way (a bad discussion-born flip) as confirmed. Front-loading the attack means the prepared default is the *survivor* of a refutation, not a raw assertion. **At the decision turn the skeptic pass therefore becomes a _confirmation_** — re-read the prep's `Skeptic:` verdicts, and run a fresh skeptic sub-agent only when (a) the prep carries no `Skeptic:` line (prepared under the old skill — treat as un-attacked), or (b) the discussion surfaced a new branch/flip the prep skeptic never saw (the *discussion-born flip* rule above). The inline attack still runs on every ratification as the floor; what moves to prep is the heavy separate-skeptic escalation.

### Principle-conformance pre-flight — readiness is conformance, not just mechanics (#608)

`--select` (and `check:standards`) gate **mechanics**: tier + size + resolved `blockedBy`. An item can pass all three and still not be truly ready — it may build into a standard that doesn't exist yet, hide an unratified design fork, cite stale `file:line` refs, or quietly violate a guiding principle. The **forward conformance gate** (the #608 sweep, run retrospectively first by #607) folds that judgment into selection so an item must pass *principle conformance* before it counts agent-ready. It has three layers, in increasing cost:

1. **D3-readiness — automatic, in the loader.** An open `issue`/`idea` whose `relatedProject` is a `concept` project **with no shipped surface** (zero resolved items filed against it) is `projectPending`: the standard must exist first, so the loader **demotes it out of Tier A** (it falls to C). Precision mirrors the audit's D3 fix (#613): a `concept` label over *substantial shipped work* is status-drift (graduate the project — #617), **not** a pending project, so its dependents stay ready. `check:readiness --select` lists held items under *"Held — project pending"*; `check:standards` warns with the same set. Nothing to run — it's already in the ranking.
   - **CTA invariant (#1275) — every not-ready item must name its next action; a bare "not ready" is forbidden.** The demotions above (`projectPending`, `humanGate`) and the structural ones (`blockedBy`, oversized story, unsliced epic) all exist so that **no open item is ever just "not ready" with no stated way forward.** Each must resolve to one **call-to-action** the Prioritisation table renders as a pill: *build it* (Tier A) · *ratify / prepare it* (decision) · **slice it** (unsliced epic) · *split it* (story `size > 8`) · *unblock it* (`blockedBy #NNN`) · *graduate the project* (`project-pending`) · *clear a human gate* (`humanGate`). The loader derives `hasCta` as the exact union of those pills and **`check:standards` hard-errors on any open item where it's false** — so a card can never ship as a dead end for whoever picks it up. **An epic is sliced, not built**, so an unsliced epic's CTA is always **slice** — even when its `relatedProject` is still `concept` (`sliceable` is orthogonal to tier, like `splittable`: a `project-pending` epic such as [#1004](/backlog/1004-implement-web-charts-native-first-svg-renderer-custom-chart-/) is Tier C *and* "ready to slice", and the project-pending pill is suppressed on epics in favour of the slice cue). To clear a gate error, give the item the state that earns a pill — don't leave it stateless. This is the structural guarantee the per-reason demotions add up to; the reasons below are its branches.
   - **Human gate (#1137) — the sibling automatic demotion for human-only residuals.** When an item's *only* remaining work is something **no agent can do** — a credentialed deploy / secret (`never-push`), agent-training **feedback**, a one-time external **setup** (account, hosted resource), or a human **review/sign-off** — add a `humanGate: { kind, what }` to its frontmatter (`kind ∈ deploy|credential|feedback|review|setup`; `what` = the one-line human action, e.g. a runbook pointer). The loader treats it **exactly like `projectPending`**: it demotes the item out of Tier A (→ C, excluded from `batchable`), so the selector stops surfacing it as agent-ready. It is deliberately **not a `blockedBy` edge** — nothing in the backlog will ever resolve it (a *person* acts, then removes the gate), so a `blockedBy` to a phantom node would be a lie. `check:readiness --select` lists gated items under *"Held — awaiting a human action"*; the Prioritisation tab shows a kind-coloured pill; `check:standards` validates the shape (kind enum + non-empty `what`) and surfaces the held set. This is the canonical home for the "the agent built everything it can, the last step is yours" residual — reach for it instead of parking the item (which reads as *deferred*, not *waiting on you*) or inventing a fake dependency.
2. **Deterministic decision-governance + ref drift — `npm run check:health`.** Runs `scripts/audit-backlog-health.mjs` (read-only): G1 undeclared-prereq edges, G2 built-ahead-of-ruling, **G3 ungoverned-architecture** (a resolved build that graduated to a **new named standard entity** — `<kind>:<name>`, kind ∈ block·intent·protocol·project·plug·capability·adapter — with no governing decision in its lineage or prose), **G4 false-prepared-fork** (a `preparedDate` decision whose `## Fork` sections carry prioritization/effort tells — "premature", "no second consumer", "not broken … but", "more to build/maintain", "cheaper/expensive" — i.e. the fork-existence test was skipped at prep, so the stamp is false), D1 dead `file:line` refs, D2 dangling `#NNN`, D3 stale projects. Treat each hit as a **candidate**, not a verdict — the first judgment pass over-reports whenever a governing decision lives in prose or one epic-hop up (the #607 lesson; #612 hardened the lineage walk, but verify before acting). **A governance gate's subject is *governable architecture* — a new named standard entity — not routine impl (#1498).** G3 scopes to entity-graduations (`<kind>:<name>`) and ignores file-path / locus-prefixed graduations (`we:`/`fui:`/`plateau:scripts/...`): a file landing in an existing subsystem implements already-settled arch and rarely warrants a fresh decision. This is why G4/G5/G7 stay small — they key on a narrow subject; G3 was the lone over-broad gate (350 → 41 when it counted entities only, not impl). The reusable rule: when a late-activated gate over-fires, narrow its *subject* to the governable noun before reaching for a recency or volume cap. The excluded file-path pool is impl, not a violation — it is dropped from the count (not retained as a standing INFO block); the full graduated set remains queryable from the data if an occasional sweep is ever wanted. <span id="canonical-build-kind-predicate"></span>**Both G2 and G3 gate on the *one* canonical build-kind predicate (#1473): `isExecKind = (kind) => kind !== 'decision'`, exported once from [`scripts/check-standards-rules.mjs`](../../scripts/check-standards-rules.mjs) beside `BACKLOG_KINDS` and *imported* by the audit (and by `readiness/proposer.mjs` as `isBuildable`), never re-declared.** Every non-decision kind ships work — `story`/`task` are build leaves, `epic` is the umbrella — so all three count as exec; only `decision` is excluded. It is defined as `!== 'decision'` (not a positive list) so a *new* build kind is auto-covered, and the lone silent-death vector (`decision` itself being renamed) is pinned by [`scripts/__tests__/exec-kind.test.mjs`](../../scripts/__tests__/exec-kind.test.mjs). Re-declaring the rule inline is the drift class that left the exec gate on dead `idea|issue` vocabulary — and so G2/G3 silently never firing — from the kind-axis migration until #1473 rewired it. **G4 is the deterministic backstop for the "never trust the `preparedDate` stamp" rule:** at claim time a prepared decision's forks are *claims to re-verify*, not settled — re-run the fork-existence + not-prioritization test and collapse any false fork to the #088 shape (one invariant + a supported/deferred list) **before presenting**, exactly as at prep. A `preparedDate` exempts nothing. (Worked example: [#803](/backlog/803-per-component-a11y-panel-content-data-sourcing-for-the-web-d/) shipped prepared with three "forks"; the claim-time re-run collapsed all three — two forced invariants + one prioritization-in-fork's-clothing — to a single ruling, and G4 was added to catch the next false-stamp without a human push.) **Scoped mode (#957):** `npm run check:health -- --scope=<batch-slug>` attributes the flags by *owning item id* against the session's claimed ids (the #952 claim baseline now stamps owning ids), demoting concurrent/pre-existing findings to a non-failing note count and surfacing only your changeset's items — the id-axis mirror of `check:standards --scope` (#952). A flag with no owning id (D3, project-keyed) is unattributable and stays in view (fail-safe). The batch gate step **adopts** it so the gate-red diagnosis is deterministic, not a manual `grep errors + git status` triage.
3. **Judgment skim — the body pre-flight you already do.** Catalog A–E conformance (mandate-nothing / strict-where-it-matters / lock-in / decomposition-placement / decision-discipline — full catalog in [#607](/backlog/607-audit-all-resolved-backlog-items-against-the-guiding-princip/)) plus the buried-fork check (*Running a batch* → body skim). This is the part no script can decide.

**Remediate, don't escalate (the #608 flow).** When a flagged item is fixable by **pure agent work — no judgment call** (stale ref → real path, prose prereq → `blockedBy`, false/stale premise → corrected, missing placement note → cite the already-ratified ruling), **fix it in place** on the spot (the same fix-on-the-spot rule as the digest and fork-readiness pass). Only the **irreducible residual** — a genuine buried fork, or a principle violation needing a design call — escalates, and it escalates as a **ready-to-ratify nod** (options + recommended default), never a raw mess. The hard boundary: remediation **brings an item to DoR, it never decides or ratifies** — and it **never force-fixes a card into batchability by quietly making a design call** (e.g. inventing a `blockedBy` edge that contradicts an established convention — verify the convention before "lifting" a prose prereq). That quiet-design-call is exactly the failure the retrospective audit (#607) exists to catch.

**Per-fork classification pass — run these standard questions on *every* element of the decision, automatically.** Most sub-forks answer themselves once you classify the element against the architecture, so apply this fixed sequence to each fork before forming a stance (and record the classification in the item — it *is* much of the ruling). Resolve them in order; an earlier answer often settles later ones:

1. **Which layer is it?** Block (ships runnable code) / **Intent** (declarative UX "what" — a dimension) / **Protocol** (a conformance contract independent vendors implement) / **Capability** (a platform feature, tiered per impl). Use the cheat-sheet in [design-first.md](design-first.md) (§"which layer"). Litmus: *no runnable code* rules out Block; *the UX "what"* → Intent.
2. **Is it a Protocol, or just an Intent dimension?** Reach for a Protocol **only** when many vendors must interoperate or swap engines (positioning: CSS-anchor ⇄ Floating UI; keyboard handlers). A deterministic, fully-specified transform with **no swappable-vendor story is NOT a protocol** — making it one is lock-in for no interop gain (minimize-lock-in: a protocol is the single escapable lock, never reached for casually). If it's not a protocol, it's an Intent concern.
3. **Does it affect an Intent? Then expose the whole axis as configurable — don't bake one behavior.** An Intent-level concern is the declarative "what," so surface *every* legitimate value as a configurable dimension rather than collapsing to a fixed behavior with hidden ride-alongs. (E.g. cascade → `cascade` + `valueReporting` dimensions, not one on/off knob.)
4. **Fixed mechanic or configurable dimension?** Expose as a dimension **only if both branches are legitimate end-states**; otherwise bake it as a fixed mechanic (and say which invariants are fixed *because they hold under any value of the dimension*). **A dimension is *not* a fork — this answer is dispositive: pull it out of the `## Fork N` sections and the preview table entirely.** Per [`#config-extends-platform-default`](platform-decisions.md#config-extends-platform-default), a multi-legitimate-end-state concern is a **config dimension** — enum/contract → WE type-only, value → project config, **core stays default-less** — and its default is **not a ratifiable pick** but the **platform-default *flavor*** fixed by Q6 (most-permissive), with the restriction as the author's opt-in. Offering an `option` with a *ratified default* is **not** the same as classifying it as a dimension: the giveaway you mis-routed is a `## Fork N` whose two "branches" are just *two values of one knob* (decline **vs** throw; eager **vs** lazy). If Q4 says dimension, the only thing left is which value is the platform-default flavor — and Q6 already answers that, so there is nothing to ratify. (The #1892 Fork-2 miss: severity `report`/`throw` was framed as a ratifiable fork-with-a-bold-default when it is a config dimension whose flavor Q6 fixes to `report`.)
5. **Is it DI-injectable?** Distinguish **structural** dimensions (locked to local Trait Selection — model, editable, hierarchy) from **behavioral** ones (`focusStrategy`, collision, truncate) that may fall back to the **Ambient Intent / DI channel**. A behavioral dimension is resolvable via DI by default — don't hardcode it on the member.
6. **What's the default? The most flexible / most permissive value, always** — the restriction is the author's opt-in, never the default.
7. **Does it sit at a seam between two intents?** If selection-state, focus, etc. propagates *across* another intent's structure, name the composition seam and put the dimension on the intent that owns the structure it traverses, *gated to* the other intent's relevant value.

**Contract-first — derive the contract *before* running any classifier that tests it; never reverse-engineer it to fit the answer (the #1892 Fork-1 miss).** Several classifiers (Q1 *which-layer* via [#1282](platform-decisions.md), the [residue bar #1839](platform-decisions.md#plugged-only-residue-bar) — *"the clause-(ii) test is over the **contract**, not the bare capability"*) run a statute test **over a contract**, so their verdict is only as honest as the contract you feed them. The trap: pick the *narrow* contract that makes the test return the answer you wanted (minimal residue / no patch / don't-port). Defuse it by ordering the work: **(1) derive the contract independently from [#1826](platform-decisions.md) — *what coherent standard is this plug/feature proposing?* — without looking at which classification you'd prefer; (2) only then run the classifier over that contract.** If a *narrower* contract flips the verdict, that is a **red flag, not a shortcut** — it means you are about to standardize a half-feature (a "validation" nothing enforces; a "context" nothing propagates). The #1892 Fork-1 miss: the prep reached "not residue → no patch" only by narrowing the contract to "validation through the plug's own helper"; the coherent-standard contract ("a declared vocabulary enforced *regardless of call-site*") fails #1839 clause-(ii) ⇒ it **is** residue ⇒ patch. A coherent standard sets the contract; the classifier then reads it — never the reverse.

**Standing test before any of the above — is this even a decision? Support every coherent option; mandate nothing; exclude only what is actually broken.** WE is not in the business of dictating how a consumer/author works — it supports *any coherent way of building*. So before framing a "fork," ask whether its branches are genuinely mutually-exclusive end-states or just *different coherent approaches that can all be supported*. If they can coexist, **they are not a decision** — record them as "supported by default" and move on; do **not** spend the decider's judgment picking one. A fork is a *real* decision only in two cases: (a) **a forced invariant** — exactly one branch is correct and the alternative is *flawed / won't work* (e.g. a cache-identity rule that omits a byte-determining input silently serves drift) — which is a **ratify**, not a weigh; or (b) **a genuine choice** — two branches are both coherent but cannot coexist (a real either/or, e.g. ship an optional tool now vs. not). State the excluded branch as *the broken one* and say why; if you can't name a broken branch, there's no decision — support all of them. **Before accepting case (b)'s "cannot coexist," run the *composability probe*: try to build one branch as a *facade over a shared kernel* the other also exposes (the #756 factory pattern). If both branches can be facades over one kernel — e.g. a behavior kernel surfaced as *both* an attribute spelling and a persistent `we-`/CEM element — they coexist, so it's support-both, not a fork. "Cannot coexist" is a claim you *prove by failing to compose*, never one you assert.** This is the most-permissive-default (Q6) sharpened into a **fork-existence test**: most "open decisions" are really "support all coherent options + one forced invariant," and saying so is the ruling. (Worked example: [#088](/backlog/088-module-service-versioning/) collapsed five "forks" to one forced invariant + one genuine A/B.)

**Third archetype — the *validation-gate* decision (a go/no-go on a candidate), distinct from a merit fork.** Not every `kind: decision` is a fork. A card that asks **"do we commit to building candidate X, and on what trigger?"** — the shape most AI-generated idea cards take (the [#142](/backlog/142-ai-generated-dev-experience-feature-candidates/) dev-experience pool is the canonical batch) — has **no rival branch** to weigh, so the two-case fork test above doesn't fit it. It is still a **legitimate decision**: the human decides what earns a place on the roadmap, and "validate this AI-proposed candidate" is exactly that call. So the standing test has **three** outcomes, not two: *(1)* it's a real fork → `## Fork N` shape; *(2)* its branches coexist → "Supported by default"; *(3)* it's a **one-sided go/no-go on a candidate** → the **validation-gate shape** (below). **Do not** mis-route a validation gate into either of the first two — it is neither a fork (no excluded branch) nor a coexistence question (there's nothing to support-both). **Guard against the retired soft-park ([#1620](/backlog/1620-implement-the-backlog-hold-model-priority-low-demote-from-au/)):** a bare *"do we ever need this, on what trigger"* with no prior-art delta and no recommended verdict is **not** a prepared gate — it's the soft-park #1620 retired. Prep's whole job here is to *add the missing substance*: the prior-art delta (named incumbents + the WE semantic delta), a recommended **verdict** (go / no / not-yet), and a **concrete** un-gate trigger. The verdict is a **merit + readiness** call (is the candidate real, and is now the moment), never a cost/effort ranking — the *not-a-prioritization* rule below still binds. (Worked example: [#1631](/backlog/1631-shareable-full-context-repro-bundle-for-the-dev-browser/) and its 20 #142 siblings — each prepared as a validation gate with a verdict + trigger + merit-refuted skeptic, no `## Fork N`.)

**A *merit-conceded* not-yet is not a ratifiable decision — it dissolves to accepted-on-merit + ordering (#2092).** The not-yet verdict is sanctioned only while the **merit half is genuinely open**. Apply the partition test: *strip timing / named-substrate-readiness / demand from the card, and ask whether a merit/warrant unknown remains* — "is the delta even real," "does the moat evaporate without a substrate that may never carry the facet." **Unknown remains → it stays a validation-gate decision** (merit is *conditional*; e.g. [#1648](/backlog/1648-pii-and-sensitive-data-flow-map/), whose on-moat delta hinges on the declared state model carrying facets that may not exist). **No unknown remains** (merit *flatly conceded* in prose, only ordering left) → **it dissolves**: a named substrate → a `blockedBy` DAG edge; a 2nd-consumer/demand threshold → a graduation trigger; pure timing → a plain backlog priority. This is the *not-a-prioritization* rule (#1961) reaching the go/no/not-yet verdict, and it extends the two-confusion screen (below) from `## Fork N` sections to the **verdict** itself. **Two guards the ruling requires:** (1) dissolve **compresses** the human turn to a one-glance **batch-confirm** of the concession — it does not delete it; an AI prep author's prose merit-concession is *not* the human roadmap-validation the #142 pool exists to give, so `accepted-on-merit` still needs a (batched) human nod, never an auto-`resolve` on the author's say-so. (2) The conditional-vs-conceded call is made by the **fresh-context screen**, not the authoring session — every #142 moat is "derived from the declared model," so any card's merit is *frameable* as conditional; that dial-by-prose is exactly the blind spot the screen catches.

> **A fork is not a prioritization tool — it decides the *best option on merit*, never *what's worth doing now*. Cost/effort/sequencing is never a fork branch.** A fork asks which branch is *correct* (or which is *broken*); whether the best option is *affordable to build right now* is **prioritization**, and prioritization happens at a different moment entirely — at *what-do-we-work-on-next* time (backlog selection / burndown ordering), never as a branch inside the design decision. So "**do the better thing vs. the cheaper thing** because the better thing is expensive" is **not** a fork — it is prioritization wearing a fork's clothing. Strip the cost out of the decision: **rule for the best option** (the absolute end-state, decided on merit), then file its build as a *separately-prioritized* backlog item whose **priority — not its existence — is the open knob** (no "defer because expensive" framed as a design choice; the end-state is ratified, the *schedule* is just ordering). Effort/cost asymmetry **alone** never constitutes a fork: if both branches agree on the end-state and differ only on *when / whether-it's-worth-it-now*, there is no design decision — there is **a ruling plus a backlog ordering**. **This applies *inside* a legitimate fork too, not just to whole forks:** even when a fork is genuine, every per-branch *tradeoff* you list must be a **merit** tradeoff (correctness, composability, a11y, UX, lock-in) — **never effort**. Tells like "**more to build**", "**a second entry**", "**more entries**", "**more to maintain**" are cost/effort smuggled into the merit column; **strip them**. The effort of the winning branch lives on its *build item's `size`*, not in the decision's pros/cons — listing it as a con biases the on-merit call toward the cheaper branch, which is exactly the contamination this rule forbids. A branch's only valid downside is a way it is *worse on merit*; if its only downside is "costs more," it has **no** downside in the fork. **The tell-list is illustrative, not exhaustive — the real screen is the *nature* of each downside, not its wording.** Re-express every per-branch downside as either a named **merit** axis (correctness · a11y · UX · lock-in · composability) or a **cost** (maintenance · drift · sequencing · "another artifact to keep in sync"); if it can *only* be phrased as a cost, strip it. This catches effort that **dodges the keyword list**: "*reintroduces a drift surface the other avoids*", "*avoids the maintenance of B*", "*another manifest to keep in sync*" are all maintenance cost wearing merit's clothes — a branch whose selling point is *"avoids the cost of the alternative"* is making a prioritization argument, not a merit one. **The mechanical loop — run it until the call is clear:** imagine both branches are *free to build and instantly maintained*; does a **merit** difference remain? If none, dissolve to support-both + a separately-prioritized build; repeat until what's left is one of {forced invariant · genuine merit either/or · support-both}. That *is* "take merit vs effort until the decision is clear." (Worked example: [#465](/backlog/465-portable-cross-field-rule-expression-language-or-stay-shape-/) — the apparent "shape-only **vs.** portable cross-field compiler" fork *dissolved* once cost was recognized as a prioritization input, not a branch: the ruling is the best end-state — portable CEL cross-field as an optional, advertised capability — and *when* to build it is normal burndown ordering, leaving only the representation (CEL) as the genuine on-merit choice.)

**Standing bias across all of the above: separate and decouple.** When a fork is "one combined thing vs. two composed things," default to **two** — split a reusable axis into its own intent/protocol/plug that others *compose*, rather than absorbing it into a larger one. A concept that recurs without its neighbour (disclosure without a tree; positioning without a droplist) earns its own home. Couple only when the split has a concrete, named cost; the burden of proof is on combining, not on separating.

**Standing lens: an elemental-but-missing platform capability is plugged(proposed) + unplugged(safe-now), not one forced choice.** When a fork concerns a capability *elemental to web applications but absent from every platform spec* (distinct from below-the-Baseline-floor, which `#native-first-baseline` rules out of scope), do not collapse it to a single posture — that's the slip that throws out a real capability (e.g. #1807's first skeptic killed "validation as the default" and nearly lost validation entirely, until the dual axis rescued it as the *plugged* posture). Frame it as **unplugged** (safe-today usage of only what the platform ships — the product surface, #606) **and** **plugged** (the proposed standard as runnable code — the upstream candidate, carrying enforcement/polyfill). Decompose the capability and apply the test **per layer**: a layer present in a shippable browser → native (`#native-first-baseline`); a layer absent from every spec → the plug. Guardrail: the *contract* stays single-substrate — plugged/unplugged is a delivery+enforcement axis, not two contracts. Codified as the *plug-as-proposed-standard* corollary under [platform-decisions.md#native-first-baseline](platform-decisions.md#native-first-baseline); doctrine #1826, first application #1807.

This sequence is also why **research-first matters**: it supplies the platform vocabulary and the prior-art that each classification leans on. Walking it on every fork is what makes the "many questions answer themselves" effect real — skipping it is how a decision drifts into an ungrounded preference poll.

**The prepared-fork shape — a decision's Definition of Ready (do this *ahead* of the call).** The two passes above are *what to verify*; this is the *form a prepared decision takes on disk* — so the work can be done ahead of time (pure agent research + authoring, no human judgment yet) and the eventual decision turn is fast ratification rather than cold research. [#64](/backlog/064-tree-select-block/) is the canonical worked example: an item is *ready to decide* when a fresh reader can rule on it without re-doing the research, i.e. its body has all of:

- **A digest that declares the grounding** — names that no design exists yet, that the N forks below are grounded in a prior-art survey **published as a `/research/` topic** (referenced here; the session report is linked via `relatedReport`), and that each carries a recommended default in **bold**. The lead paragraph is the at-a-glance promise the item is prepared.
- **An axis-framing paragraph** that decomposes the concern into the orthogonal axes the research surfaced (tree-select: *disclosure / hierarchy / selection*), each pinned to **concrete `file:line` refs into the real tree** (`intents.json:1059`, `blocks.json:22-32`) — the *concrete-code* check made structural.
- **A "recommended path at a glance" preview table** under the framing paragraph — one row per fork (*recommended default · main alternative · confidence*). It lets the decider **ratify every row in one glance** or jump straight to the one they'd override; the **confidence** column (high / med-high / low-divergent) says where judgment is actually needed vs. where to nod — #64's Fork 4 (`Low — divergent across libraries`) flags itself as the only real call. Put confidence in the table, not also inline on each fork (one home, no dupe).
- **One `## Fork N` section per open fork** — opens with a one-line **fork-existence justification** (see next bullet) → crux-with-refs → options **`(a)`/`(b)`/`(c)`… named with their tradeoffs** → the **recommended default in bold** → rejected branches marked *Rejected* with the reason → any sub-decision → a closing **`Skeptic:` line** recording that prep's skeptic sub-agent attacked the default and what it changed (`REFUTED → flipped to …` / `SURVIVES — beat <attack>` / `SURVIVES-WITH-AMENDMENT → …`; see *Red-team the default* → *the skeptic's primary seat is prep*). A `## Fork N` with no `Skeptic:` line is **un-attacked, so not at DoR** — the prep close-out gate must reject the stamp, and at the decision turn a missing line means "treat as un-attacked, run the skeptic now" (the *confirmation-vs-fresh-attack* branch above). Wiring this into the readiness lint / `check:health` G4 (alongside the missing fork-existence-justification check) is a worthwhile follow-up, not yet implemented. (At resolution each gains a dated ruling, and the item **graduates to spin-off builds via a `blockedBy` chain** in composition order — #64 → #295 hierarchy → #296 block (disclosure pre-existed, from #008) — a prepared decision yields agent-ready builds, not code itself.)
  - **Fork-labeling convention — fixed, so every decision item reads the same way (and the glance table, red-team, and ruling can cross-reference unambiguously).** The **section heading is `## Fork N`** — numbered *Fork 1*, *Fork 2*, … (matches #1349, #1337, #064, #088). Options **inside** a fork are lowercase-lettered **`(a)`/`(b)`/`(c)`** and referenced as **"Fork N (a)"** (e.g. "Fork 2 (a)"). A second-level choice under one option is a **"Fork N sub-fork"**. **Never** `## Fork (a)`/`## Fork (b)` for the sections, and **never** combined letter-number labels like `A1`/`B2` — those collide with the heading numbering and break cross-references. The "recommended path at a glance" table's first column is the fork number (*Fork 1*, *Fork 2*); the red-team and *Recommended ruling* sections cite **Fork N (x)**, never a bespoke alias.
- **Every `## Fork N` must carry its fork-existence justification *at prep*, not only re-derive it at the call (#819).** The *fork-existence test* (above) is the standing gate, but until #819 it was only applied when a decision was *presented* — so a prep pass could stamp `preparedDate` over a "fork" whose branches actually coexist, and the gap was caught only at decision time by a skeptic or the user (the #811 miss: prep asserted "Fork 1 … a real fork" with **no** justification of which branch is flawed or why the branches can't coexist). Close it at the source: **before stamping `preparedDate`, each declared `## Fork N` must state, in one line, *why it is a fork* — either name the **flawed / excluded branch** (case (a), forced invariant) or say **why the coherent branches genuinely cannot coexist** (case (b), real either/or). A fork that can't produce that line is not a fork — dissolve it to the *"Supported by default (not decisions)"* list before stamping.** This is the *fork-existence test made a per-fork authoring requirement*, not just a presenter's skim — it sits alongside the *not-a-prioritization* rule (a "justification" that reduces to cost/effort is not a fork-existence reason; strip it). **Two excluded-branch justifications are valid only if *positively disproven*, never asserted:** (1) *"no consumer needs branch B"* — search the platform's **ratified goals** for the consumer *first* (the #1457 miss: a no-element block has no CEM, so the polyglot framework-flavor generator #463/#855 *was* the consumer, sitting in plain sight). **And for a mint specifically, "no *second in-house consumer* yet" is NOT a valid rejection reason (#2533): prior-art research establishing a fundamental, recurring, web-platform-aligned pattern is *sufficient* to justify a standard.** A standard exists to be reused across independent parties, and that reuse case is established by *prior art* — native primitives, studied algorithms (d3-scale as a standalone package, `SemanticZoom` as a shipped control, BPMN/git-graph swimlane layout) — **not** by waiting for a second in-house consumer to appear. Rejecting or parking a mint because only one board/app consumes it today is the struck second-consumer bar; re-rule on the pattern's prior art instead. (2) *"the branches cannot coexist"* — run the **composability probe** (try building B as a facade over A's kernel) and report it *failed*. An asserted "no consumer" or "cannot coexist" with no search and no failed composition is a skipped fork-existence test, not a justification. `check:health` G4 already flags prioritization/effort tells in a prepared fork; the readiness lint (below) flags a prepared fork whose justification line is **missing** outright.
- **Two-confusion screen — a *fresh-context* agent must clear every fork of the two recurring authoring confusions before `preparedDate` is stamped (#2091).** The two flaws that most often survive prep and reach the human — costing a deep review-and-correct per item — are **(1) standard-vs-implementation**: the fork asks the decider to rule on something *not observable across the impl boundary* (an impl detail dressed as a standards call, the #2030 axis), or places an implementation concern on the standard side of the WE↔FUI boundary (#1282); and **(2) merit-vs-prioritization**: strip cost/effort/timing from both branches and no merit difference remains — prioritization wearing a fork's clothes (the *not-a-prioritization* rule above; #1961, #2010). Both already have inline guards (the pass-4 skeptic's classification axis, the G4 tell-scan) — but those are a **same-session self-assertion** and a **keyword scan**, and both flaws are *framing* errors the authoring session is structurally blind to. So before stamping: spawn a **throwaway fresh-context agent that did not author the forks**, whose prompt asks exactly two questions per `## Fork N` (and per validation-gate verdict): *"(1) Does this fork rule on an implementation detail — would the ruling be invisible to a consumer across the boundary — or put an impl concern on the standard side? (2) Imagine both branches free to build and instantly maintained: does a merit difference (correctness · composability · a11y · UX · lock-in · precedent-consistency) remain?"* **For a validation-gate verdict (no rival branch), question 2 specializes to the #2092 partition:** strip timing/named-substrate/demand — does a genuine merit/warrant *unknown* remain, or is merit conceded and only ordering left? Merit conceded → `flagged(prio)` → **dissolve** the gate (accepted-on-merit + `blockedBy`/trigger, human turn compressed to a batch-confirm), don't leave it a ratify turn. Record the verdict as a one-line **`Screen:`** note beside each fork's `Skeptic:` line (`Screen: clear` / `Screen: flagged(impl|prio) → <fix applied>`). A flagged fork is **rewritten before the stamp** — re-layer it (impl detail → black-box default or FUI-side note; the observable contract stays) or dissolve it (ruling on merit + a separately-prioritized build item). A body with any fork missing its `Screen:` line is `○ needs prep`; the stamp is false. This is deliberately *narrower* than the pass-4 skeptic — two named confusions, every fork, no leverage exception — because the screen attacks the **framing** where the skeptic attacks the **default**, and a cheap fixed screen holds the floor the discretionary attack keeps missing.
- **A fork that picks a *default value / set / strategy* is usually not "which one" — it's "mandate vs. config-driven open default."** Before ratifying a single default (a reaction set, a render strategy, a collation), check whether it should generalize to the **Config-Extends-Platform-Default** shape: a **default-less core** + an **open registry / inherited-extensible setting** in the *platform config the project extends*, with the "sensible default" living in the platform **flavor**, not the standard. A locked set and a free-for-all then become *configurations* of that registry, not rival branches — and the fork shrinks to "what's the flavor default," often a one-line ratify. (Worked example: [#370](/backlog/370-candidate-standard-expressive-symbols-reactions-emoji-sticke/) Forks 3 & 6 — "pick the render strategy" / "pick the reaction set" both collapsed into open config-driven platform settings.)
- **At graduation, spin out one Technical Configurator card per *documented technical setting*.** A ratified decision that documents a technical (impl) choice — a render strategy, an update strategy, a transport — must produce a **Technical Configurator option** (plateau-app; add a domain via seed + provider entry) so a project can pick its inherited/extended platform value. The platform-config **schema** ships with the intent/block build; the **Configurator UI** is its own card. **Cross-ref an existing Configurator domain instead of duplicating it** (e.g. real-time transport → the webrealtime domain). Pairs with *Intent-UX-only, technical→Configurator*: a technical setting is never a UX dimension on the intent and never a WE mandate. (Worked example: [#370] — render-strategy + reaction update-strategy cards; sync-transport cross-ref'd to webrealtime.)
- **Make the call tangible — a real-life example, not just abstract fork prose.** A decider rules better on something concrete, and there are two forms, chosen by what the decision is about. **(1) Real-world occurrences — always.** Name the shipped products/systems where the pattern already appears (the *"Known occurrences"* the prior-art survey turned up): the YouTube/Netflix scrubber for a dual-track `progress`, Percy/Chromatic/reg-suit for a visual-diff surface, Microsoft's shipped `SemanticZoom` for representational zoom. This is exactly what justifies a *mint* under the corrected bar (recurring, web-platform-aligned prior art — not a second in-house consumer) and what a validation gate's incumbent table supplies. **(2) A rendered example — when the decision is about a *visual or UI surface*.** Don't leave a UI-shaped call as an ASCII sketch or prose: build a small, self-contained, *operable* mockup of what the surface could look like and embed it in the item, so the decider can *see* and click it rather than imagine it. The `prepare-decision-item` skill (pass 3) carries the embed mechanics (a standalone file under `src/assets/`, iframed from a directory URL). (Worked examples: [#2533](/backlog/2533-console-board-derived-ui-standards/) grounds every fork in named occurrences; [#2538](/backlog/2538-shape-the-annotated-visual-diff-surface-contract-then-decide-mint/) embeds an interactive visual-diff mockup the decider operates.)
- **Research reshapes the forks, it doesn't just fill the template** — in #64 the survey *added* Fork 4 (a multi-select cascade the item omitted) and dissolved a worry the original raised. A prep pass that only confirms the item's existing framing usually means the survey was too shallow.
- **Lead with the decision; quarantine the context — the decider must see *what they're choosing* in one screen, not hunt for it.** Run the *fork-existence test* (above) first and let it shrink the item: branches that can coexist go under a short **"Supported by default (not decisions)"** list; forced invariants are stated as a one-line **ratify**; only genuine either/or choices get a `## Fork N`. Then push all the framing tables, sequencing, and relationship notes **below** a clear `## Context` (or `---`) divider so they don't read as part of the call. A prepared decision that lists five "forks" where four are just "support all of these" is *verbose, not thorough* — it buries the real call. Crisp-and-separated beats complete-but-undifferentiated: the glance table + the `## Fork N` sections should contain **only** things the human actually decides. (Worked example: [#088](/backlog/088-module-service-versioning/) was rewritten from five bold-default "forks" to *one forced invariant + one A/B*, with the matrix/sequencing demoted to `## Context`.)

An item with all forks stated this way — each carrying options, a bold default, and a fork-specific `/research/` topic — is **prepared**: stamp **`preparedDate: "YYYY-MM-DD"`** into its frontmatter with `node scripts/backlog.mjs prepare-stamp <NNN>` **in the lane** (the #2219 (b) flow, shipped #2264 — the stamp is a backlog mutation, blocked from a primary cwd and landed via the one PR; don't hand-Edit it onto primary). While you prepare, a hard local `prepare-hold` keeps the fork out of every other session's selection + claim. `check:readiness --select` then ranks a prepared item **prepared-first within Tier B and tags it `✓ ready to ratify`** (vs `○ needs prep`), so decision-mode surfaces the ready-to-ratify forks ahead of the ones still needing a research pass. The research cost is paid once, ahead of the turn, not re-paid cold in every discussion.

**The prepared *validation-gate* shape — DoR for a go/no-go decision (no `## Fork N`, still prepared).** A validation-gate decision (third archetype, above) reaches Definition of Ready with a different body shape than a fork — it carries **no** `## Fork N` (correctly: there is no excluded branch to weigh) yet is genuinely prepared. Its DoR body has all of: a **`## Digest`** leading with the recommended **verdict** (go / no / not-yet) + a confidence (lead with prose, not a `**Label:**`); **`## What you're deciding`** stating the candidate concretely; **`## Why this isn't a classic fork (and is still a decision)`** naming it a one-sided gate; **`## Context & prior-art delta`** — a table of real named incumbents and the WE semantic delta (this is what turns a bare "do we need it" into a real call, and supplies the examples a decider needs); **`## Dependencies & lineage`**; and **`## Recommendation`** = the verdict + a **concrete** un-gate trigger (a named substrate shipping, an evidence threshold — never "on what trigger" left blank) + a **`Skeptic:`** line refuted on merit (the prep skeptic seat applies here too — attack the verdict, fold the result in). The `/research/` topic is optional for a validation gate when the prior-art delta is captured inline and shallow; spin a topic when the survey is deep enough to stand alone. (Canonical worked example: [#1631](/backlog/1631-shareable-full-context-repro-bundle-for-the-dev-browser/).)

**`preparedDate` means the research + authoring is done and every fork is at the Definition of Ready — *not* "this item gets ratified directly."** A fork reaches that bar three ways: it's **resolved**, it's stated in the prepared-fork shape above, or it's **delegated** — carved out to a child item that is *itself* prepared. **(A *validation-gate* decision reaches the bar via the validation-gate shape just above — verdict + concrete trigger + merit `Skeptic:` line — instead of `## Fork N` sections.)** **Those three are exhaustive. A fork left as a bare "needs a human call" / "confirm X" / "TBD" / slash-name (`webpush`-vs-`webpermissions`) with no options + tradeoffs + bold default is _not_ at DoR — it is the un-prepared part wearing a prepared stamp.** *(This disqualifier targets **merit forks** that dodge their options/default. It does **not** disqualify a yes/no question per se: a **validation-gate** decision asks a go/no-go question by design, and is at DoR once it carries a recommended verdict + concrete trigger + merit `Skeptic:` line. The disqualifier still bites a validation gate that's a **bare** question — no verdict, no trigger, no prior-art delta — which is the #1620 soft-park, not a prepared gate.)* The point of prep is to bring the *human-judgment* fork to options-and-a-default the decider ratifies or overrides; "left for a human" unshaped means prep stopped exactly at the boundary where its real work begins. Stamping `preparedDate` over such a fork is a false "ready" — fix it (shape the fork or delegate it) before stamping, and never trust the stamp without checking (see the *Fork-readiness pass*' first bullet). So a **parent/epic decision whose only open fork was spun out to a child is prepared once that child reaches prepared shape**: stamp the parent's `preparedDate`, and rewrite the carved-out fork from "Open decisions" to **"→ delegated to #NNN (prepared)"** so the flag is truthful rather than a bare stamp. Ratifying the child settles the parent's fork; the parent then `resolve`s. Don't leave such a parent tagged `○ needs prep` just because the call happens on the child — there is no separate research left to do on it. (Worked example: the design-ref corpus decision [#581](/backlog/581-design-ref-corpus-pipeline-shape-subject-classification-fork/) — Forks 1 & 2 resolved, Fork 3 delegated to the prepared [#394](/backlog/394-design-ref-corpus-first-run-scope-taxonomy-seed-grow-targets/) → stamped prepared, then resolved. It was originally conflated into the epic #382 and later split out — see the "never one item that is both `kind: decision` and `kind: epic`" rule under *Rules*.)

**No live choice may sit *outside* a `## Fork N` — a "residue / TBD / decide-at-ratification" aside in prose is an un-prepared fork in disguise, and the structural reason `prepare` and the decision turn can _diverge_ (#1935).** The trinity above (resolved / stated-as-fork / delegated) and the close-out walk both scan the **`## Fork N` headings** — so a real sub-choice written as *prose* slips past every check: "open residue for the ratification turn", "whether v1 ships X in full or Y first", "TBD", "to be decided later", a glance-table footnote, a "Net mechanism … open question:" trailer. That aside **is** a live fork. Leave it and the decider must originate a conclusion **cold** at the call — which is precisely how a prepared item and its own decision turn reach *different* answers. They should not be able to: a decider only *ratifies the item's default*, so divergence is never a judgment gap — it is the signal that **a choice was left un-prepared**. Close the hole at prep: at close-out **scan the whole body, not just the headings**, for any deferred-choice phrasing and resolve each — promote it to its own `## Fork N` (research it now: options + bold default + `Skeptic:` line), fold it into an existing fork's default, or drop it as not-actually-a-choice. A body carrying a dangling residue is `○ needs prep`, never `✓ ready to ratify`; the stamp is false. (Worked example: [#1935](/backlog/1935-how-to-determine-merge-risk-files-that-must-be-reserved-befo/) was stamped prepared with Fork 2's "ship C in full vs. A-only first" left as an *open-residue* prose line rather than a fork — the decision turn then formed a chat-only sequencing recommendation that diverged from the body's stated default of C, the exact failure this rule prevents.)

**Always give two links per offered item** so the user can open it either way:
- the **live page** — `[<id>](http://localhost:3000/backlog/<id>/)`. Browse on `:3000` (Vite), which **proxies `/backlog/` to the 11ty server on `:8080`** — that's the URL the user actually has open. (A *newly authored* item can `404` until the running 11ty server rebuilds — it's stale, not a wrong URL; a clean `eleventy` build always has it.)
- the **source file** — `[backlog/<id>.md](backlog/<id>.md)` (opens the markdown in the editor; always works regardless of server state).

Example offer line: *`jsx-directive-sugar` — add the deferred `<For>/<Show>/<Resource>` layer ([live](http://localhost:3000/backlog/070-jsx-directive-sugar/) · [md](backlog/070-jsx-directive-sugar.md))*.

### When nothing is agent-ready — surface the one highest-leverage blocker

If *Gather* + tiering leaves **no Tier-A item** (the ready pool is empty — everything left is Tier B/C, blocked, or needs a design call), **do not** return a long menu of open decisions. Instead pick **exactly one** item — the one whose resolution unblocks the most downstream work — and put it to the user as the single thing they need to decide. This is the **only** time selection recommends a non-Tier-A item, and it still returns just one.

- **Rank the blockers by downstream unblock count.** For each open `decision`/blocked item, count how many *other* open items depend on it — anything that names it ("see #NNN", "prerequisite", "after X", "blocked on …"), rolls under it (`parent`), or sits in its chain. **Most dependents wins.** Break ties by the *size* of the work it gates (more / larger downstream points first), then prefer the **smallest decision** (a one-nod Tier-B ratification over a wide-open fork) so the unblock is cheap.
- **Claim it first, then present it as a decision to make — not a build.** A decision discussion is itself claimable work, so before laying it out, claim it as its own turn (see *Working an item* → *Claim it on start* — re-read to win the race, flip `open → active` + `dateStarted`, then the chat-rename prompt) so a concurrent session won't talk through the same fork. *Then* state the item (with its live + md links), name **what's blocking it** (the open fork / human judgment needed), and **list concretely which items it unblocks** once resolved (their `#NNN`). Frame it as: *"resolve this and these N items become agent-ready."* Run the **fork-readiness pass** (above) before presenting — concrete example present, options stated with tradeoffs, short/concise/well-explained — and fix the item on the spot if it falls short. Follow planning-as-discussion (lay out the realistic options + your recommendation with reasoning; expect follow-up questions) — **do not** fire a multiple-choice; this is a design call, not a bounded pick among ready items. If the user redirects to a different item, release the claim (`active → open`) and claim that one instead. Once the call is made **and _explicitly ratified_** (see *the ratification gate* — never infer ratification from agreement on a sub-point), close out the decision item (*Closing out* — `active → resolved` + `dateResolved` + the ruling); its successor work is then agent-ready.
- **If even the blocker set is empty** (truly nothing open), say so plainly — the backlog is drained of actionable work — rather than inventing an item.

## Working an item — claim it, then keep it live

The backlog file is the **durable, resumable record** of in-flight work — treat it that way so a lost or interrupted session can pick up exactly where you left off:

> **Work in a lane, not the primary checkout — set it up FIRST (#2123).** Building an item *edits the tree*, so like every edit-action session the work runs in an **isolated lane clone**, never the shared primary checkout: `we:scripts/guard-lane.mjs` denies a primary `Edit` and `we:scripts/guard-bash.mjs` blocks a direct `main` push, so provision/enter the lane **before** you start editing (`node we:scripts/lane-pool.mjs status --json` → pick a clean lane → work there → land via PR) rather than discovering it when the guard fires after you've spent effort. **There is no primary exception any more.** `claim`/`release`/`resolve` (and every other `we:scripts/backlog.mjs` frontmatter splice — `scaffold`/`settle`/`retype`/`yield`/`prepare-stamp`) also run **in the lane clone** and land in the item's own PR — `we:scripts/guard-bash.mjs` denies every one of them unconditionally from a primary cwd (#2302), with no override, per the ratified #2219 direction ("everything rides the one lane→PR; nothing splices to the primary item file").

1. **Claim it on start — one command does the race-safe flip.** Run **`node scripts/backlog.mjs claim <NNN>`**: it re-reads from disk, refuses only if the item isn't still `status: open` (lost the race — another session already flipped it to `active`), and otherwise flips `open` → **`active`** + stamps `dateStarted` (today) in one surgical splice — then prints the chat-rename slug for you to copy. **It does not look at git/commit state** — uncommitted edits never block a claim (ownership is the `status` flip, not the tree). This *is* the claim; you don't hand-edit the frontmatter. (Manual fallback if you must: set `status: active`, add `dateStarted: "YYYY-MM-DD"`, save — *before* writing any code.) If the command errors, **stop — it's taken**; go back and pick another. Selection drops `active` (see *Gather*), so a fresh session won't re-pick work already underway. **This applies to a `decision` too:** discussing a fork is itself claimable work, so claim it (and rename the chat) *before* presenting the decision's substance — the only difference from a build is that the "work" is the discussion, not code. If the discussion never resolves, release it back to `open` (rule 3); when the call is made, close it out (`resolved` + the ruling) like any item. **The same claim re-read also emits the execution-routing verdict** (Opus-inline vs Sonnet sub-agent) — see [*Model routing*](#model-routing); it costs nothing extra because the freshness re-read it depends on is already happening here.
2. **Keep it in sync as you go.** Maintain a short **`## Progress`** section in the item's body and update it *as work happens*, not at the end — if the session dies mid-task, this section is **all the next session has** (recovering the prior chat is not reliable; the item body is the contract). Keep this exact shape so any session can resume in seconds:
   ```markdown
   ## Progress
   - **Status:** active — <one line: where things stand>
   - **Branch:** <git branch with the WIP>
   - **Done:** <what's landed + green>
   - **Next:** <the single concrete next step>
   - **Notes:** <key file paths, decisions, gotchas>
   ```
   (An in-session TodoWrite list is fine for live tracking, but it is **not** durable — only the item body survives the session.)

   **Keep one in-progress todo, phrased as a status — it's exposed live.** The item's `in_progress` TodoWrite entry is surfaced on the **/backlog Active-work tab** as that card's *currently-doing* line (read live from your session transcript; the board prefers it over an inferred last-action). So keep **exactly one** todo `in_progress` at a time and phrase it as a present-tense status a glancer can read (`Wiring the lane overlay`, `Gating + verifying the build`) — not a vague `step 3`. Let that todo **be** your progress channel: it doubles as the live status *and* keeps you from narrating the same thing in prose (the *track, don't narrate* habit). This is the live, at-a-glance view; the **durable** record still lives in `## Progress`.
3. **If abandoned unfinished,** run **`node scripts/backlog.mjs release <NNN>`** (`active` → `open`); leave the `## Progress` section as-is (it says where it stopped), so it returns to the pool instead of looking claimed-but-dead.
4. **Reclaiming a stranded claim.** An item left `active` by a crashed/forgotten session would otherwise be invisible forever (selection drops `active`). So when asked to work and the pool looks empty — or when explicitly told to continue one — treat an `active` item as **resumable**: read its `## Progress`, check out its branch, and continue from **Next**. Don't re-pick it as fresh; resume it.

When it's fully done, follow *Closing out a completed item* (mark `resolved` after the final-capture pass).

## Model routing — the Opus loop orchestrates, a Sonnet sub-agent executes {#model-routing}

**You run on Opus and never switch the loop's own model.** The judgment work stays on the Opus main loop — selection, the whole decision arc, slicing/splitting, preparing a fork, and every close-out review. What gets pushed down to a **Sonnet sub-agent** is *execution*: the mechanical middle of an item whose call is already made. The point is to spend Opus tokens only where judgment lives — and the bigger win is **context isolation**: a sub-agent starts fresh, so it does its file-reading in its *own* cheap context and hands back a summary, instead of bloating the Opus loop's expensive context with execution churn. From the human's seat nothing changes — same Opus session, same skills, same commands; routing is a step *inside* the skill, never a model the user picks.

**The routing verdict is emitted at claim, never self-rated mid-work.** A claim already re-reads the item fresh and re-evaluates its `blockedBy`/digest (*Working an item* → *Claim it on start*; *Keep the blocker DAG honest*) — fold one more output into that same read: **does this item's execution qualify for a Sonnet sub-agent?** Decide on signals already in front of you, *never* on a self-rated "is this hard" (a cheaper model can't judge when it's out of its depth, so that question is never asked of it):

- **Delegate to Sonnet** only when ALL hold: the card is *prepared/DoR or mechanically clear*; it **still holds against the current tree** (the claim re-read found no drift — no reopened fork, no reappeared blocker, locus correct); blast radius is **bounded** (single locus, touches no contract / shared gate / cross-repo seam); and `size` is small. The re-validated card *is* the sub-agent's self-contained brief — if it isn't complete enough to hand off, it wasn't ready. The sub-agent returns a **summary, not file dumps**.
- **Keep it on the Opus loop** when the claim re-read finds **drift** (a stale card that no longer matches the tree is *judgment again* — re-prepare or re-decide, never blind-execute), when the work itself is **judgment** (`decision`/slice/prepare), or when it's a **trivial one-liner** the Opus loop is already positioned for (a fresh sub-agent's context load would cost *more* than just doing it — delegation has a floor).

**The asymmetry sets the bias: when in doubt, escalate.** Over-delegating an easy item wastes a few tokens; under-escalating a judgment item writes a bad ruling into the backlog graph. So **Sonnet is the path an item must affirmatively qualify for** (prepared · sized · single-locus · drift-free), and anything ambiguous stays on Opus. Escalation back up is a **control-flow event, not a confidence score**: a Sonnet sub-agent that hits a real fork **stops and hands back** (it never decides the fork) — the hand-back is what returns the item to the Opus loop.

**Mechanism.** Where the loop would do the execute step, spawn `Agent(model: "sonnet")` with the re-validated card as the brief; on its return the Opus loop runs the close-out gate and reviews the diff against that card. Judgment turns are never spawned this way.

**Every spawn carries the return-hygiene contract (#1861, model-usage watch #1855).** A sub-agent's final message *is* its return value the Opus loop keeps — not a human-facing report — so prepend this contract to every execute/probe/research spawn (it is the floor that makes "returns a summary, not file dumps" enforceable rather than aspirational):

> **Return the conclusion the parent will keep, not a transcript.** (1) **Never fabricate specifics** — no invented version numbers, `file:line` refs, API names, flags, or counts; if you didn't *read* it this run, don't state it as fact (an honest "unknown / not verified" beats a plausible guess — a wrong ref silently misleads). (2) **Flag uncertainty explicitly** — mark inference vs. verified fact. (3) **Prefer a tight ranked list over prose**; omit raw file dumps and step-by-step narration. (4) **If returning a structured object, every field must be grounded** — leave it empty rather than guess.

The failure this kills is concrete: the first #1855 front-B sweep returned **fabricated Claude Code version numbers**. The parallel `/workflow` path applies the identical contract mechanically — `parallel-execute.workflow.js` defines it as a `RETURN_HYGIENE` constant prepended to every `agent()` prompt (the script has no filesystem read at runtime, so the prose above is mirrored there; keep the two in sync when either changes).

**Batch interaction — calibration re-learns, it doesn't break.** A serial `/batch` calibrates its point budget from a context-% reading mapped to points resolved (*Running a batch* → *Calibrating*). Delegating execution to Sonnet sub-agents moves the file-reading churn off the Opus loop, but the loop **still does claim → brief → summary → close-review per item, in sequence** — so its context still grows **roughly linearly in items**, just on a gentler slope (this is *unlike* the parallel `/workflow` path, where points resolve entirely in worktree contexts and the orchestrator sees only pack/land overhead — fully decoupled, which is why that path skips calibration). So delegated-serial calibration stays valid; what changes is that **`capacityPoints` re-converges *upward*** — the same context budget now buys more points, i.e. **bigger batches**. The self-calibrating EMA absorbs this on its own over a few sessions; the only transient is a short **under-estimate** while it catches up (safe — it stops early with budget to spare, never overruns). The one genuine caveat is **mixed mode**: if some items delegate and some run inline, loop-context-per-point is bimodal and a single `capacityPoints` blends two regimes (noisier, still a safe under-shoot) — making delegation the *standard* execute path collapses that back to one stable slope, and if the noise ever bites, calibrate against *loop-context cost* rather than raw points. So: **let delegated-serial `/batch` calibrate normally and expect `capacityPoints` to rise**; single-item `/next` has no calibration; the parallel `/workflow` lanes are the natural place to pin `model: "sonnet"` on their `agent()` calls.

## Running a batch — chain several small items, stop on a solid condition

> Use via the `batch-backlog-items` skill (`/batch`, `/batch-next`). A batch works several **agent-ready** items back-to-back **without stopping for approval between them** — to run a bit longer and progress faster — while keeping a real validation stop at every seam and a hard backstop that guarantees it ends. It **reuses the single-item arc unchanged** (*Selecting*, *Working an item*, *Closing out*); it only adds the loop and the stop rule below.

**The size of a batch is a POINTS BUDGET, not an item count.** The old "take the top 3 small items" cap left a session ~80% idle (a real 10-item batch consumed only ~20% of the window). So a batch now **packs as many points as possible up to a budget** — the calibrated `budgetPoints`, read from `.claude/skills/batch-backlog-items/capacity.json` (the affine model's largest points-load under a context ceiling minus a data-driven margin; see *Calibrating the budget*). The cost it sums is **`batchCost`** — a story's `size`, and **`2` for a task** (tasks carry no burndown points but still consume context, so they aren't free). The budget is the **deterministic backstop** that guarantees the batch ends (every item costs ≥ 2, so the sum strictly rises) — replacing the count cap. Because the count is now "whatever fills the budget," a single `size·5` story joins the chain when it fits the remaining points, where the old `≤3`-only gate would have left both the points and the slot on the table. **`capacityPoints` self-calibrates at close-out** (see *Calibrating the budget* below), so the target tracks what a session actually fits rather than a fixed guess.

**What a batch keeps (unchanged): per-item on-disk ownership.** Every item is still claimed individually — re-read to win the race → `status: open → active` + `dateStarted` *before* any code → `## Progress` kept in sync → `resolved` at close-out after the full gate. A stranded batch is therefore exactly as recoverable as a single item: each one is claimed and progress-tracked on disk, and selection still drops `active`, so nothing gets re-picked mid-flight.

**The one thing a batch drops: the claim-as-its-own-turn *stop*.** That stop exists only to keep the per-item chat-rename prompt from being buried — a batch renames the session **once**, up front (see the skill), so there is no per-item prompt to protect. Claim and work flow together. The on-disk active/ownership discipline above is **not** relaxed — only the per-item rename-stop is.

> **Claim as the *first keystroke* on an item — before any file read.** Because claim and work now flow together with no stop between them, it is easy to slide into exploring/editing code while the item is still `open` and only flip it at close-out. That leaves a real window where a concurrent agent sees it as unclaimed. So on starting each item, **flip `open → active` + `dateStarted` first**, then read code. (Re-read to win the race, then immediately claim — don't interleave exploration between the two.)

**Eligibility — take the suggested pack straight from the ranker; don't re-derive it.** The size+tier
gates are **deterministic**, so the loader derives them (`src/_data/backlog.js`) and
`npm run check:readiness -- --select` prints them directly (also `--json` → `selection` + `batch`),
identical to the `/backlog/` Prioritisation tab. Two derived sets matter:

- **`item.batchable`** — the candidate set the budget may pack: a Tier-A item small enough to chain —
  a **`task`**, or a **`story` of `size` ≤ 8**. This is what the packer walks. There is no separate
  ≤3 "core" tier — the budget packs **smallest-first** (the rank orders by effort), reaching a single
  `size·8` only when the remaining points fit, so genuinely small work still leads.

The CLI's **`Suggested batch — points budget`** block *is* the pack: it greedily walks the ranked Tier-A
list and lists the items whose cumulative `batchCost` fills the budget (`--json` → `batch.picked`/`spent`).
**That pack is your plan — read it, don't reconstruct it** (reconstructing it by hand from frontmatter is
the slow, under-counting pass this replaces). The rules below are documented so the derivation stays
auditable; keep them and the loader fields identical if either changes. The gates are structural only —
the body-fork pre-flight below is the non-structural guard a field can't decide, applied **only to the
items in the pack you're about to chain**.

Plus the Tier-A guards every packed item must still clear: `issue`/`idea`, concrete bounded build,
prereqs resolved, **no design fork**, **named file paths**, **clear acceptance criteria**. **Never
packed:** a **`story` of `size` ≥ 13** (the should-split band), any **`epic`**, and all Tier B/C items
(decisions, anything needing a design call) — surface those for discussion as usual. (A **blocked** item — Tier C *only* because a prerequisite is still open — is likewise never *packed up front*, but a **cascade batch** claims it the instant its predecessor resolves and frees it to Tier-A; see *Linear & semi-linear cascade*. The permanent exclusion is of work needing a *design call*, not of a clean build waiting on a sibling.) (If an item
has no `size`/`kind` yet, fall back to the blast-radius heuristic — a handful of files / one subsystem
— and treat it as eligible only if it's clearly small.)

> **Pre-flight each candidate's *body* for a buried fork — `size` measures effort, not decision-weight.** A `story·3` can still hide a real design call ("decide whether the served form stays self-contained or imports the helper") that the size field never shows. Before approving the plan, skim each body for **"decide whether / alternative held open / open sub-decision"**; a small item with a buried fork is a **stop risk** (stop rule 4), not a clean batch item — either resolve the fork in discussion first (Tier B) or drop it from the chain.

> **When the body-skim shows an item is mis-flagged, *fix the flag* — don't just skip it.** The batchable flag is a structural proxy (`type` + `size` + resolved prereqs); a body that reveals the item is really a **decision/fork**, **deferred-by-intent / gated on an unmet precondition / on-demand-only**, or **mis-sized** is *mis-flagged at the source*. Silently skipping it leaves it to re-surface and re-cost the same pre-flight in every future batch — so **correct the frontmatter in place** (a one-line splice, body untouched) as part of the batch's opening pre-flight, not as a separate chore: a clean `--select` pool is the batch's own input. The lever map (the same fields the loader keys on — `tier` ← `type`/blockers, `batchable` ← `tier A` + `story·≤3`/`task`):
> - a genuine fork, strategy, or triage call → `kind: decision` (moves to Tier B, "discuss, don't auto-build");
> - explicitly gated on an unmet precondition / on-demand-only → `status: parked` (drops out of the pool until reopened) — **and parking always carries a machine-readable reason (#1392, amended #1620):** the soft `deferred`/`external-infra`/`superseded` reasons are **retired** (parking is not a prioritisation escape). A park must reduce to a real structural state — a `parkedReason` of `platform-gated` (waits on a shipped browser feature) or `maturityGated` (building now yields a *worse* artifact — see [the hold model](#hold-model), requires a typed `maturityTrigger`), or — when the blocker is a single tracked item — a real `blockedBy` edge (which pills "blocked by #N" and needs no `parkedReason`), or a `humanGate` for a human-only residual, or — on an epic — a `childlessReason`. A *settled-but-low-value-now* hold is **not a park at all** — it is `priority: low` (see the hold model), which stays in the pool but out of auto-select. `check:standards` **errors** on a parked item with no derivable reason, and the reason renders as a `⏸ <reason>` pill on the tile / Prioritisation table / detail page (`reasonPill` in `we:src/_includes/backlog-badges.njk`, vocab in `we:src/_data/backlogMeta.js` `parkedReasonMeta`). A deliberate hold's WHY is first-class and surfaced, never buried in prose;
> - agent-ready but too big to batch (e.g. new-standard / new-intent authoring) → bump `size` to its honest points (`8`/`13`) so it stays Tier-A *single-item*, out of the eligible pool.
>
> Run the standards gate after the splices and report the reclassifications in the plan so the user can object — they are reversible (git) and internal (backlog metadata), but they *are* edits to shared state.

### The hold model — `priority: low` vs a park, two mechanisms for two jobs {#hold-model}

Ratified by **[#1620](/backlog/1620-backlog-hold-model-add-a-load-bearing-priority-field-decide-/)** (amends #1392). The thing that classifies a hold is **what it does to the ready pool** — and the operator's test separates them: *"low priority, not rejected; if I have context free and nothing else, I might pick it up"* keeps an item **pickable**; "wait until X clears" **removes** it. So a hold is one of these, never a soft "deferred":

| Mechanism | Pool effect | When |
|---|---|---|
| **`priority: low`** | dropped from the **auto-selected ready set**, but **stays visible** in a filler section (demote, don't hide) | settled & valid, just low-value-now — *"might pick it up if nothing else"* |
| **`maturityGated`** park (+ typed `maturityTrigger`) | **removed** until the typed trigger fires | building *now* yields a **worse artifact** (you'd guess the shape, tune against nothing, automate the unproven) |
| `platform-gated` park | removed | waits on a shipped browser feature |
| `blockedBy` / `humanGate` | removed | a real prerequisite item / a human-clearable residual |
| `kind: decision` open | decision lane | a genuine merit fork |

- **`priority: low` is a *demote-not-hide* bit, not a JIRA enum.** The only honored value is `low` (absent = normal — no `high`/`normal`, which import priority-rot and duplicate the tier). The readiness `--select` ranker **excludes** `priority: low` from the auto-selected ready set so it stops false-surfacing as Tier-A, but the loader still **lists** it in a separate *"deprioritised / filler"* group — visible, browsable, hand-pickable when nothing better exists. Because it stays *visible*, it can't silently vanish the way a soft park did: it is auditable filler, not a hide. Use it for *settled-but-low-value-now*, never for "gated on X" (that's a park) or "decide whether" (that's a `kind: decision`).
- **`maturityGated` is `platform-gated`-grade only because its trigger is concrete and external.** A `parkedReason: maturityGated` **requires** a machine-readable `maturityTrigger`, one of: `externalConsumers≥N` (a real 2nd impl / first external consumer exists), `realRuns≥N` of a named manual skill with a stability assertion (a soak that accrues over *time* through real runs — not a `humanGate`, which a human clears *on demand*), or a named `adoptionSignal` (a concrete integration milestone to test against). The gate **errors** on a `maturityGated` item with no typed trigger, and the trigger **must** name a counter or an external artifact's existence — **never a date or bare "later."** That guard is what keeps it from being `deferred` 2.0. Reserve it for the genuinely-unmapped soak/threshold/adoption residual: a true prerequisite item is still `blockedBy`, and a merit fork is still `kind: decision`.
- **"No consumer yet" is not a hold — when the contract is codified, a test fixture *is* the consumer. Build it.** This is the recurring "no consumer" defer-excuse, and it is **banned as a standalone reason to defer a build.** Three things make it bogus: (1) **`priority: low` does not mean "gated."** It means *settled-but-low-value-now, still pickable* — using it to encode "wait for a consumer to appear" is exactly the soft-`deferred` park #1620 retired, just wearing the filler bit. (2) **`maturityGated`'s bar is "building now yields a *worse* artifact"** (unproven shape, nothing to tune against) — which is **false the moment the mechanism's contract is codified** (a ratified decision pins its shape). A codified contract means you are *not* guessing; there is a correct shape to build to. (3) **`externalConsumers≥N` is satisfied by a real consumer's *existence*, never by its *absence*** — "no item is `blockedBy` me" does not license a park, it just means no *production* caller exists yet. **When the contract is codified, a vitest/unit fixture that encodes that contract is a valid consumer** — author it test-first and build the mechanism against it (this is exactly how its deterministic sibling builds: a fixture in → asserted output out). So the only legitimate states for a contract-codified mechanism are: **build it now (fixture = consumer)**, a real **`blockedBy`** edge to the *specific* item that must land first, or — only if building genuinely produces a worse artifact — a typed `maturityGated` park whose trigger names a concrete external counter. A prose "consumer-parked / awaiting-consumer" disposition (± `priority: low`, ± an `awaiting-consumer` tag) is **none of these** and is a mis-flag to correct at claim, not honour. (Worked example: #1827 — SSR injector-context hydration sat `priority: low` + "consumer-parked" prose for a mechanism whose contract was *already* codified by the resolved #1818; the FUI vitest fixture encoding #1818's non-deterministic cell was the consumer all along, so it was buildable from the day #1818 resolved.)

Vocab + enforcement: pill vocab in [`we:src/_data/backlogMeta.js`](../../src/_data/backlogMeta.js) (`priorityMeta` + `parkedReasonMeta`); the `--select` exclude + filler group in [`we:scripts/check-readiness.mjs`](../../scripts/check-readiness.mjs) + [`we:scripts/readiness/`](../../scripts/readiness/); the typed-trigger gate in [`we:scripts/check-standards-rules.mjs`](../../scripts/check-standards-rules.mjs); badges in [`we:src/_includes/backlog-badges.njk`](../../src/_includes/backlog-badges.njk).

> **A fork lives in a `kind: decision` item — never inline in an `idea`/`epic`/`story` body. Flip *or* carve, depending on whether the fork is the whole item or a part of it.** The lever above (fork → `kind: decision`) is the **flip** case: the item *is* the fork, so retype it in place. But when an item is **legitimately** a build idea/epic/story that *also* carries one or more forks in its body (a common epic shape: real capability framing **plus** "open design points" — e.g. [#192](/backlog/192-longitudinal-research-freshness-system/)), don't flip the whole item and don't ratify the fork in place. **Carve** instead: (1) scaffold a **separate `kind: decision` item** holding the fork(s) in the prepared-fork shape (options + bold default + concrete refs — cluster coupled forks into one item, #004-style); (2) make the original **`blockedBy`** that decision (the decision gates the build, so it blocks it); (3) **trim the fork out of the original's body**, leaving a one-line pointer to the decision item — never duplicate the fork in both places. Resolving the decision then unblocks the build (a slice / the epic's slices). This is the structural rule behind *Decisions Are Work Items*: the decision item is the plan of record, and a fork left inline in a non-decision body is mis-placed even when the item's `type` is otherwise correct. (Worked example: #192's four foundation forks carved to #441, which `blocks` #192.)

> **Keep the pre-flight to the body skim — don't re-derive what the selection already settled.** The `--select` projection is the source of truth for *ranking and readiness*; re-computing its inputs by hand is wasted turns at equal quality. Specifically: **(a) Don't re-verify already-resolved blockers.** An item only reaches the batchable pool with its `blockedBy` edges resolved — that's the projection's guarantee. The only blocker work here is the body-fork skim catching an *undeclared* prereq (corrected at claim, per *Keep the blocker DAG honest*), not re-`grep`ping that listed `resolved` blockers are still resolved. **(b) Never consult git to judge eligibility.** The working tree's commit state is irrelevant to whether an item is taken or workable — ownership is the `status: active` flag (+ `reserve` holds), not uncommitted edits, and a dirty tree is the normal baseline. Do **not** run `git status`/`git diff` during pre-flight, and **never drop an item because it (or the tree) has uncommitted changes** — that is not a drop-reason (see *The drop-reason classifier*). `claim` itself never looks at git. **(c) Don't hunt for a "better" cluster by default.** Reach one cluster deeper for a tighter same-subsystem alternative **only when the pack's body skim actually surfaces a fork or an "outgrew its estimate"** — that's when the exploration earns its cost. If the suggested pack skims clean, take it and go.

### The loop

1. **Plan the batch, approve once.** Run *Selecting* (passing **`--session=<batch-slug>`**, see *Cross-session reservation* below), then take the CLI's **`Suggested batch — points budget`** pack (the greedy fill of the ranked Tier-A list up to the budget) and present it as an **ordered batch plan** — the items it intends to chain, each with its live + md links and its `batchCost`, plus the running total against the budget. A single "go" (or one `AskUserQuestion`) authorizes the **whole batch**, not each item. **On that go, soft-hold the planned pack** — `node scripts/backlog.mjs reserve <NNN...> --session=<batch-slug>` — so a *second* concurrent batch packs around your items instead of re-analysing and starting them (see *Cross-session reservation*). `/batch <P>` overrides the budget (a points number, not an item count); `/batch-next <NNN-slug>` seeds the chain's first item (skip its selection, per *Selecting* step 0) and packs the rest by budget. **Cluster the order by subsystem/repo purely for efficiency** (this is an *ordering* hint, **not** a stop trigger — see the stop rule): group items that touch the same files/package together, and put items in a different repo (e.g. a frontierui-side change after a run of webeverything ones) **last**, so same-context work stays contiguous. A subsystem/repo boundary is **not** a place to stop — the batch runs through it to the budget. **The pack packs smallest-first but is free to reach a `size·8`** when it fits the remaining points — that's the point of budgeting over counting. If the batchable pool can't fill the budget, the plan is simply shorter (stop rule 3 fires early); don't pad it with `≥13`/`epic` work — those stay single-item (the *Other Tier-A* list), surfaced separately, not packed.
2. **Work each item through its full arc** — claim → work → close-out gate — reporting the **compact ledger** (below) instead of per-item prose. **The close-out gate is the item's *own locus* gate, not always this repo's** (see *Repo-locus*): look up `LOCI[item.locus]` and run its `gateCommand` in `repoPath`, commit only that item's files to `commitTarget`, and perform any `closeoutDiscipline` (exercise-app's GAP-tagging). A WE item gates with `npm run check:standards` here; a plateau-app item gates with `npm test` in `../plateau-app`. The pack's per-locus legend (printed by `--select`) tells you which.
3. **At each seam** (after an item's close-out, before claiming the next) **evaluate the stop rule.** If it says continue, **re-read the next item fresh from disk** to win the race (if it's now `status: active`, another session took it — drop to the next eligible; uncommitted edits are *not* a reason to drop).
4. **Top up the plan at a seam when the original pack runs dry but budget remains — and *loop* the top-up to exhaustion.** The pack was planned off the *opening* snapshot; resolving an item can **cascade-free** a previously-blocked item to Tier A — and when the batchable pool is a dependency *chain*, this top-up is the batch's **primary engine**, not an exception (see *Linear & semi-linear cascade*). So whenever you reach the end of the current pack with budget left, **re-run `npm run check:readiness -- --select --budget=<remaining>`** (`remaining` = budget − the ledger's resolved `cost`), pack its fresh suggestion, and **keep doing this until the budget is reached or the only items left are hard-tagged drop-reasons** (see *The drop-reason classifier* — a single re-pack that surfaces an item you *decline on judgement* does **not** end the loop; an item you can't tag `taken`/`blocked-in-fact`/`not-batchable`/`outgrew` must be claimed — note a cross-locus item is **never** declined for locus since #500, it's gated in its own locus). Momentum favours freed items (often in the subsystem you just touched). Pre-flight each topped-up item's body for a fork exactly as in the opening plan. **Never auto-absorb a close-out leftover/spin-off** this way: those are brand-new and unvetted (they may hide a fork — stop rule 4's whole point), so they are *captured and left* for a future batch, not claimed to spend budget. The budget is a **ceiling, not a quota**: it caps how far a batch may run and never manufactures work to hit the number — but stopping *below* it is only legitimate when the classifier says the in-locus pool is dry, not when the remaining work merely looks unappealing.

### Cross-session reservation — soft-hold the planned pack (deprioritize for other sessions)

Two independent `/batch` sessions both run `--select`, get the **same** ranked smallest-first pack, and
start the **same** items — only diverging when one *wins* the late `claim`. The seam re-read keeps that
*correct* (the loser drops the now-`active` item), but the loser already burned the up-front
analysis and any initial work. The reservation tier makes the overlap visible at **plan time** without
giving up late claiming (backlog **#083**):

- **At plan-approval, reserve the pack:** `node scripts/backlog.mjs reserve <NNN...> --session=<batch-slug>`
  (use the session label you renamed the chat to). This is **not** a status flip — the items stay `open`;
  the hard `claim` at work-start is still the only real lock.
- **`--select` deprioritizes — never excludes — foreign-held items.** Pass **`--session=<batch-slug>`** to
  *every* `--select` you run (the opening plan *and* each seam top-up) so your **own** holds don't sink
  your own chain; another session's holds appear at the bottom of the pool marked `⊘ held by <slug>`, so
  you pack around them but can still take one if it's all that's left (a small pool is never starved).
- **Release on the way out** (*Stopping*): `node scripts/backlog.mjs unreserve --session=<batch-slug>` —
  anything you reserved but never claimed flows straight back to full priority. `claim` already drops the
  hold on each item you actually worked, and a reservation older than the registry's `ttlMinutes` (default
  120) is ignored anyway, so a crashed session can never depress an item's priority forever.

Advisory by design: ignoring the hint only costs redundant analysis (self-correcting at the seam), never
corruption — so unlike a file lock it needs no enforcement. It is the *cross-session* complement to
`--parallel`'s *intra-session* partition. Cheapest stopgap if you'd rather not stamp reservations at all:
seed each concurrent session onto a different subsystem cluster via `/batch-next <NNN-slug>`.

### Linear & semi-linear cascade — batching a dependency chain

Parallel lanes and the serial pack both assume the batchable items are **independent** — already unblocked, so the only open question is order. But the highest-leverage autonomous runs are often a **dependency chain**: B is `blockedBy` A, C by B, and so on, so at plan time the projection lists **only the chain head** as Tier-A (everything behind it is Tier C, blocked). Treating that as a one-item batch wastes the run — resolving A *frees* B, which frees C. **Cascade batching** is the mode that works such a chain end-to-end: it **never claims a blocked item**, but it re-derives the eligible frontier at every seam, so each resolution pulls the next item into Tier-A and the batch keeps going. This is the answer to "this epic is a sequential chain — can we still batch it?": yes, in dependency order, one resolution feeding the next.

It is **not a new mechanism** — it's the seam **top-up** (loop step 4) promoted from *exception* to *primary engine*, and the rule that makes it safe is already in place: a seam re-pack absorbs **cascade-freed pre-existing items** (vetted work that was merely blocked) but **never close-out leftovers** (brand-new, unvetted — stop rule 4's domain). A chain is just the case where almost every continuation is cascade-freed.

Two shapes:

- **Linear** — a pure chain `A → B → C → D`, each the sole blocker of the next. Work strict dependency order; each close-out frees exactly one successor. The plan *is* the chain.
- **Semi-linear** — a DAG, not a line: resolving one item frees **several** dependents (fan-out), or a chain runs alongside independent items. The eligible **frontier grows** as stories resolve, so at each seam you pick the next item from the *current* frontier (smallest-first within budget, per `--select`), not from a fixed up-front order. A **storied epic's children** are the common source — resolve the foundational slice and a cluster of siblings goes Tier-A at once.

**Planning a cascade up front (one approval for the whole chain).** Only the head is eligible at plan time, so `--select`'s pack shows a single item — don't present *that* as the batch. Instead **trace the chain by hand from the head's dependents** (the items whose `blockedBy` lists the head, transitively) and present the *anticipated cascade* as the plan: the ordered chain, each item's `batchCost`, the cumulative total against the budget, and each link marked **`ready now`** vs **`unblocks when #PRED resolves`**. One "go" authorizes the whole cascade; the session label spans it (`batch-<date>-<head>-…-<tail>`).

**What still gates it (unchanged).** Every item runs the **full arc** (claim → work → close-out gate) and the **stop rule applies at every seam** — the cascade is not a license to skip validation. It stops exactly where any batch stops, and the chain adds two natural stops:
- A chain item that turns out to be a **decision/fork** (not a clean build) breaks the cascade at that link — **stop rule 4**. Its successors stay blocked behind it; surface the fork and stop. (So a chain of `kind: decision` items is *not* a cascade batch — those are Tier-B, discussed one at a time.)
- The **budget** still caps total cost and a **red gate** still halts before the next link — so a long chain can neither run away nor paper over a failure.

**Throughput order across the modes:** **`--parallel`** (reached via **`/workflow`**; independent, concurrent) > **serial pack** (the **`/batch`** default; independent, sequential) > **cascade** (dependent, sequential). Cascade can't have the parallel speedup — the items are chained by construction — so it trades that for the thing that actually matters here: running the whole chain unattended in one session instead of as a string of one-item handoffs.

#### Two execute modes, explicitly operator-selected — not auto-routed {#two-modes-explicit-selection}

`/batch` (serial) and `/workflow` (parallel) are **two execute modes of one skill**, kept as **separate explicit entries** — the operator chooses which to run per batch, with `--serial`/`--parallel` as per-invocation overrides. The serial inline path stays calibrated, context-bounded and Sonnet-delegable; the parallel orchestrator path runs worktree subagents on a token/agent budget and degenerates to serial when no disjoint lane exists. **Do not unify on a single auto-routing entry that always runs the probe** (#1862, rejecting that option): it would push *every* batch — including degenerate-serial ones — through the orchestrator and pay probe + integration-worktree overhead for no benefit, and would forfeit the inline-serial path. Explicit selection also keeps the calibration boundary per-command-clear (`/batch` calibrates, `/workflow` does not) rather than prose-gated. The parallel orchestrator and its safety contract shipped under #1143/#1147.

### The stop rule — solid by construction

Built so the batch is driven by **one number — the points budget** — with three hard exits that can only stop it *earlier*; nothing soft, nothing judgement-based, can stop it. Evaluate at each seam. **Stop the batch if ANY is true (and ONLY these):**

1. **Gate red *from this batch's own work* (safety stop).** The item just finished isn't fully green — relevant tests, `npm run check:standards`, and build must all pass (the close-out gate). On red: leave that item `active`, do **not** start another, report the failure. Never batch past a red gate *you caused* — this is the per-item validation that makes "running longer" safe. **But scope the stop to your own changeset.** `check:standards` is whole-repo, so a *concurrent* batch's in-flight files (an untracked new item, an unregistered report) can turn the gate red without any of it being yours — and freezing on that defeats the entire concurrent-batch design (per-item `claim` + reservations exist precisely so two batches run on separate stuff at once; see *Cross-session reservation* and [[feedback_gate_red_stop_scoped_to_own_work]]). So on a red gate, **diagnose first** — and the fastest, deterministic diagnosis is to **run the WE gates scoped to this batch**: `npm run check:standards -- --scope=<batch-slug>` (file-keyed attribution, #952/#949) and `npm run check:health -- --scope=<batch-slug>` (id-keyed attribution, #957). Both demote every *concurrent/pre-existing* finding to a non-failing note and leave only your changeset's findings blocking — so "is this red mine?" is answered by the scoped exit/flag count, not a manual read. (Fallback when no claim baseline is recorded, or to double-check: read the error lines and `git status`.) Does any error name a file in *this batch's* changeset (the item you just worked, or anything you staged/committed)? **Yes → real stop** (rule 1 fires — never compound your own breakage). **No — every error is in concurrent/external files (untracked, not yours)** → it is **not** your stop: log one line that the gate is red from external work, do **not** touch those files (you'd clobber the other session), and **continue** with your independent remaining items. The safety intent is "don't build on *my* broken foundation," not "freeze on any repo-wide red." (A genuinely shared-file conflict — the external red touches a file you also need — is `blocked-in-fact` for that item, not a batch-wide stop.)
2. **Points budget reached (deterministic backstop).** The resolved items' summed `batchCost` has reached the budget (no remaining-budget room for the next eligible item). Every item costs ≥ 2, so the sum strictly rises and this **always** fires — the batch always terminates, which is what makes the remaining, softer conditions safe to be approximate. (This replaced the old fixed count cap.)
3. **No eligible next item — *after a seam re-pack*, and every surfaced item carries a hard drop-reason.** The planned pack is exhausted **and** a fresh `--select --budget=<remaining>` surfaces nothing **claimable**. "Nothing claimable" is **not** "nothing I feel like taking" — it means every item the re-pack surfaces is provably ineligible against the **closed drop-reason set** below. This is the budget-left-on-the-table stop: correct *only* when the classifier is satisfied — never pad the batch with unvetted close-out leftovers, and never declare it because the remaining work *looks* unappealing.
4. **New fork surfaced, or the item outgrew its estimate *mid-work*.** The completed item raised an open design question, **or** an item you had **already claimed and begun** turned out materially bigger than its `size` (its real scope sprawls across far more files than declared, or needs a subsystem/artifact that doesn't exist). Capture it (per *Closing out* → leftovers), then stop. **The "outgrew" trigger fires only *after* you start the item — never on a pre-claim guess.** A `story·8` that is genuinely ~8 points of work has **not** outgrown anything; "looks big / risky / load-bearing / cross-cutting" is the budget's call to make (does it fit `remaining`?), **not** yours. If it fits, claim it.
**There is no "context seam" stop, and no mid-batch context check.** The four conditions above are the *only* stops. The **points budget (rule 2) is the sole soft driver** — and it *already is* the context limit, because *Calibrating the budget* sets it from real context-meter readings at past close-outs (the affine `budgetPoints`). So the budget is context, denominated in points. There is nothing left for a self-judgment stop to add:

- **Never stop on your own judgement.** You **cannot read your own context meter**, and a *felt* sense of "heavy context" (long transcript, many large reads, deep investigation, repeated test runs) is **not a measurement** — it reliably mis-fires (a batch that *felt* full has measured at ~22% used, i.e. ~3× the budget still in hand). "I've read a lot" / "this is getting long" / "I'm at a subsystem boundary" are **not** stops. A repo/subsystem boundary is a plan-time **ordering** hint (cluster same-repo items, put a cross-repo item last) — **not** a stop trigger.
- **Never interrupt the batch to ask for a context reading** to decide whether to continue. Asking "should I keep going / what's your context %?" mid-run is the exact anti-pattern this rule kills — it re-introduces the self-judgment stop through the back door. The user authorized the *whole* budget with one "go"; honour it. (The context-% question belongs at **close-out only**, for *Calibrating the budget* — never as a continue/stop gate.)
- **Keep claiming the next eligible item until rule 1/2/3/4 fires.** Absent a red gate, a surfaced fork, an empty pool, or the budget being reached, the next action is *always* to claim the next item. Trust the budget, not the gut — premature, judgement-driven stops are the failure this rule exists to kill.

#### The drop-reason classifier — a declined item is not a stop

Rule 3 is the one a tired batch abuses: it *feels* done long before the budget, so the seam re-pack gets waved off ("nothing good left") while eligible work is still on the table. The classifier makes the skip **auditable** instead of a hunch. **Every Tier-A item the re-pack surfaces that you do *not* claim must carry exactly one hard drop-reason from this closed set:**

| Drop-reason | Test (objective) |
|---|---|
| `taken` | `node scripts/backlog.mjs claim <NNN>` **refused** it because it is already `status: active` — another session owns it. The tool decided on the *status*, not you. (It does **not** look at git/commit state.) |
| `blocked-in-fact` | A dependency or artifact the item *needs* is **verified absent** — you checked and it isn't there (a function/file/served URL the body names doesn't exist), not "I assume it's blocked." Often means a `blockedBy` edge is missing → add it (*Keep the blocker DAG honest*). Also covers a cross-locus item whose **repo isn't checked out** in this session, so its `LOCI` gate literally cannot run — **and a verification item whose proof needs a real runtime no harness can drive yet** (real SW/Background Fetch, a VS Code Extension Development Host, a device sensor): its harness is the missing artifact, so scaffold the harness card + add the `blockedBy` edge rather than claim an un-run verification. See *testing.md* → "every verification must be agent-runnable." |
| `not-batchable` | It's a `decision`, a `story·≥13`, or an `epic` — excluded by the eligibility gate by definition (→ the *Other Tier-A* / Tier-B lists, never the pack). |
| `outgrew` | You **claimed and began** it and hit stop-rule-4 sprawl. (Pre-claim, this reason is not available.) |

*(There is no `out-of-locus` drop-reason since #500: a batch is locus-agnostic and gates each item in its own locus — see *Repo-locus*. A cross-repo item is claimed and closed like any other, not skipped.)*

**There is NO `dirty`/uncommitted drop-reason.** A modified or untracked working tree is the normal baseline (the user owns committing); `claim` never inspects git, so uncommitted edits never block or disqualify an item. Skipping work because the tree is dirty is exactly the kind of false stop this classifier exists to kill.

**An eligible `task`/`story·≤8` carrying *none* of these tags is not a valid skip — you must claim it.** "Big", "risky", "load-bearing", "needs a focused session", "I'd rather a fresh agent did it", "different subsystem", "I've done a lot already", "the tree is dirty" are **not** drop-reasons; they are precisely the judgment stops this rule kills. **At stop, the carry-forward must tag every unworked Tier-A item with its drop-reason** (`#461 — outgrew`, `#368 — taken`, …). If you reach the stop and an item has no tag — or its tag is a paraphrase of "felt big" — you stopped early; go back and claim it. The re-pack **loops**: keep re-running `--select --budget=<remaining>` and claiming until the only items left are hard-tagged or the budget is reached.

#### Repo-locus — a batch is locus-agnostic, gating each item in its own locus (#498/#500)

Every item carries a `locus` — its **gate home**, the repo whose gate can *honestly* close it. The honesty rule is unchanged: **resolving an item on a gate that never exercised it is dishonest.** What changed (#498 ruling, built by #500) is *how* a batch honours that rule. A batch is no longer single-locus: **one session packs and closes items of *any* locus, running each item's own locus gate at its close-out seam** — so the ~44 batchable items across all four loci are reachable in one run, instead of a WE batch dropping the ~32 that live elsewhere. Locus is about *which gate validates the work*, **not** where the backlog item is filed (the WE backlog tracks the whole constellation): an item filed as "FUI slice" whose code legitimately lands in WE and gates here (e.g. a generator that generalizes WE's own loader) declares `locus: webeverything` — judge by the gate, not the label.

**The locus signal.** Every item's `locus` is derived in the loader (`src/_data/backlog.js`): an explicit `locus:` frontmatter field when authored, else inferred (an `exercise-app` *descendant* of the flagship epic #314 — structural, never the noisy `exercise-app` tag; or a precise build-tag like `frontier-ui` / `technical-configurator`), else `webeverything` (the default). `check:standards` errors on an invalid authored locus and warns on a *batchable* item whose cross-repo locus was only inferred (a nudge to declare it — inference biases toward `webeverything`, since a wrongly-inferred cross-repo locus would route close-out to the wrong gate). An explicit `locus:` always overrides the inference.

**The `LOCI` registry (`check-standards-rules.mjs`)** is the declarative source of truth — one record per locus, config not loop logic:

| Field | Meaning |
|---|---|
| `repoPath` | Dir to run the gate in, relative to the WE root (`.` = this repo; `../frontierui`, `../plateau-app` for siblings). |
| `gateCommand` | The close-out gate that must be **green** before an item of this locus is `resolve`d (WE/FUI: `npm run check:standards`; plateau-app: `npm test`; exercise-app: `check:standards && check:app-conformance`). |
| `devServerProbe` | The canonical dev port to **detect-or-skip** for a render check (WE `3000`, FUI `6000`, plateau-app `4000`) — never spin up or kill a server ([don't-kill-dev-server]). |
| `commitTarget` | The repo the per-item commit lands in. Commits are **per-repo** — never `git add -A` across repos. |
| `closeoutDiscipline?` | An extra, non-skippable close-out rule beyond the gate. Set only for **exercise-app**: platform-first build, else tag a GAP (the `/exercise-app` rule). |

**How a cross-locus batch runs.** `npm run check:readiness -- --select` packs batchable items of every locus together (smallest-first, ranked) — the old single-locus filter and its "Other locus" list are **gone**; each packed item is flagged with its locus (`⌂ <locus>`), and the pack prints a per-locus gate legend so the batch is self-documenting. Then, at **each item's close-out seam**, the loop looks up that item's `LOCI[locus]` record and:

1. runs `gateCommand` **in** `repoPath` — green at every seam, exactly as the single-locus batch required, just in the item's own repo;
2. detects the `devServerProbe` port for any render check (skips if none is running — never starts/kills one);
3. commits **only that item's files** to `commitTarget` (per-repo; never `git add -A`);
4. if `closeoutDiscipline` is set (exercise-app), performs it as a **required** extra step before `resolve`.

Because each item is gated in its own locus, **`out-of-locus` is no longer a drop-reason** — there is nothing to drop; a cross-repo item is claimed, built, and gated like any other. The only residual case: a locus whose repo isn't checked out in this session (its gate literally can't run) folds into `blocked-in-fact` (a needed gate verified absent), not a locus-specific skip.

### Reporting — a compact ledger, not prose per item

After each item, update a single running ledger — one line per item, scannable:

```
Batch · cost 5/50 · gate ✓ · pts 3 · net −2/+1 · next: #155
✓ #154 cross-list-empty-tabstop — story·3 — fixed empty-list tabstop; tests+standards green
✓ #151 cross-list-empty-fixtures — task — added shared fixtures; green (+#162 leftover)
▶ #155 registered-behaviors-coverage — task — working
```

The header tracks the batch against its budget: **`cost <spent>/<budget>`** sums the resolved items' `batchCost` (the budget driver — a story's `size`, a task = 2), so you can see how close the budget backstop (stop rule 2) is. Each line carries the item's `kind`/`size` so the chain's effort is visible. The header's **`pts`** sums resolved story/unstoried-epic *burndown* points (tasks contribute none — see *Agile sizing*), which is exactly what the `/backlog/` Burndown moves; it is intentionally ≤ `cost` (tasks add cost but no burndown points). The header also shows **net flow** `−<resolved>/+<opened>` — a batch both resolves items *and* opens new ones (leftovers captured at close-out), and capture can outpace drain; surfacing the net keeps that honest. Note each captured leftover inline on its parent item's line (`+#NNN`). Expand to full close-out detail only on a **red gate** (show the failure) or when asked. Keep the body lean — the ledger *is* the report.

### Stopping — the standard closure (fixed, terse shape)

When the batch stops, **release any soft holds you still own** — `node scripts/backlog.mjs unreserve --session=<batch-slug>` (returns un-worked reserved items to full priority; see *Cross-session reservation*) — then close with the **standard closure block below — same shape every time, no bespoke essay**. The whole close-out is: the final ledger, a one-line stop reason, the carry-forward lines, the next-step line, and the single calibration line. Don't expand any of it into prose unless a red gate needs its failure shown, or the user asks.

```
Batch closed · <slug> · <N> resolved · cost <X>/<budget> · stop: <rule>
✓ #NNN slug — one-line outcome (gate ✓)        ← one per resolved item
~ #NNN slug — reshaped: <one-line why>          ← only if any (resize / premise-fix / outgrew)
→ #NNN slug — <drop-reason>: <one-line why>     ← one per unworked carry-forward item, tagged
Next: /batch-next <NNN-slug>   (only if eligible work remains; else: the fork/gate to address)
Context %? — for calibrate; skip if you can't read it.
```

Every carry-forward line carries one **drop-reason** (`taken` / `blocked-in-fact` / `not-batchable` / `outgrew`) — an untagged or "felt big" line means you stopped early (see *The drop-reason classifier*). The **Next** line is a fresh-agent handoff in its own fenced block when the budget filled with work still queued (context resets); on a **red gate** or **surfaced fork** it instead points at the `active` item to fix or the decision to discuss — not a fresh batch.

### Calibrating the budget — fold each session back into the estimate

The budget is only as good as `capacityPoints`, and a fixed guess goes stale (the seed was extrapolated from one batch). So **at the end of a batch session, record what actually happened** and let it correct the estimate. This is the feedback loop the count cap never had — with no count cap, the budget *must* learn what a session really fits.

> **Calibration is SERIAL `/batch` only — never `/workflow` (parallel).** The whole mechanism maps a context-% reading → points resolved, which holds *only when the main loop does the work*. A `/workflow` batch resolves its points in **subagent contexts** (the worktree lanes), so the orchestrator's context-% measures only pack/plan/land overhead and is **decoupled** from throughput — a reading there is meaningless and pollutes the mean. So under `/workflow` (parallel): **skip calibration entirely** — no context-% ask, no `calibrate` call. Its ceiling is the token/agent budget, not context. Everything below applies to the serial `/batch` path.

Run, once, at close-out (the **closing-session** skill does this automatically when a batch ran; do it by hand otherwise). The agent can't read the context meter, so the closure's **last line is always the one standard ask — `Context %? — for calibrate; skip if you can't read it.`** — verbatim, nothing longer. Rules baked into that line: never the `AskUserQuestion` popup (it's data, not a decision); never re-ask or block; if no reading comes back (the user may not see the meter either), **skip calibration** and move on. Then run:

```
node scripts/backlog.mjs calibrate --points=<cost-points resolved> --context-pct=<context used at close> --stop-reason=<which stop fired>
```

- **`--points`** = the resolved items' summed `batchCost` — the ledger's `cost <spent>` figure (a story's `size`, a task = 2). Count only items that actually `resolved` this session.
- **`--context-pct`** = the share of the context window consumed at close (the editor's context meter, 1–100), from the **user's** reply to the standard ask above — use it verbatim. The agent cannot read the meter; **never estimate or guess**, and if no reading comes back, **skip calibration** rather than invent a number (a fabricated value skews every future batch's budget). A rough reading from the user is fine.
- **`--stop-reason`** (optional but recommended) = which stop rule ended the batch — `budget`, `context`, `empty-pool`, `fork`, `gate`, `manual`, `outgrew`. **Since #1505 this is audit metadata only: every batch trains the estimate regardless of stop reason.** (The old design trained only on the rare `budget`/`context` stop and excluded work-bound stops — but work-bound is the *common* case, so the estimate starved on one sample. The affine model below measures the fixed overhead explicitly, so a work-bound batch is a clean data point, not a downward-biased one.) Pass the real reason anyway so the `samples` array stays a useful audit trail.

It folds the `(points, context%)` pair into the **pooled affine context-cost model** (#1505): a Deming/errors-in-variables fit of `context% = overhead + cost·points` over **every** sample in the retained 12-sample window. The intercept is the fixed per-batch overhead (real work in every batch), measured rather than misattributed to per-point cost. The next budget is the largest points-load that keeps predicted context under `contextCeiling` minus a data-driven margin `marginK · residualStd` (a tight fit spends nearly the whole ceiling; a noisy one holds more back) — stored as `budgetPoints` (with `fit`/`capacityPoints` for transparency), replacing the old `capacityPoints × targetFraction`. The 12-sample window ages out old regimes (a new model); RLS-with-forgetting is the planned successor (#1516). Don't hand-edit `fit`/`capacityPoints`/`budgetPoints` — let the command move them; the raw `samples` array stays for audit (each tagged with its `stopReason`).

## Splitting a large story — slice only when safe, never at quality's expense

> Use via the `split-backlog-item` skill (`/split`, alias `/slice`). Splitting takes either a **`story` of `size` > 8** (`13` — too big to batch, the *should-split* band) **or an unsliced `epic`** (already decided to decompose but with no children yet) and tries to break it into **smaller, agent-ready, independently-deliverable slices** that flow straight into `/batch`. The output is **always a report**; the on-disk split itself is **gated on approval** and **only happens when it's provably safe**. The governing instinct is conservative: a *needless* split fragments one coherent deliverable into pieces that only make sense together (quality loss, more review overhead, no gain); a *missed* split just leaves a big item to single-pass later. So **when a clean seam isn't obvious, don't split** — record it as "could not split" with the action that *would* make it splittable, and move on.

**Candidate set.** Two kinds. **(a) Oversized stories** — open (`status` ≠ `resolved`) items with `kind: story` and `size` > 8 (`13`, the *should-split* band): too big to batch, at risk of bundling several deliverables. **(b) Unsliced epics** — open `kind: epic` items with **no children** (no other item names them as `parent`): an epic is *already decided* to decompose, so it skips the should-we-split question (rubric (1) below) and goes straight to proposing slices, seeded from its own body's stated decomposition. The ranked tiers from `npm run check:readiness -- --select` (`--json` → `selection.tierA`/`selection.tierB`) cover the unblocked stories; **blocked (Tier C) stories and epics aren't enumerated there**, so a one-pass frontmatter scan of `backlog/*.md` (`kind: story` + `size` ≥ 13, **or** `kind: epic` with no child referencing its `NNN` as `parent`, all not `resolved`) is the complete list. `/split <NNN>` (or its alias `/slice <NNN>`) focuses one item; bare `/split` reports across the whole candidate set.

> **Slice granularity — stories vs sub-epics.** An unsliced epic's slices are *normally* stories/tasks (the leaf, batchable level). But some epics are a **roadmap / "epic of epics"**: every natural child is *itself* a multi-story scope with its own decomposition (a registry of N independently-fundable standards/subsystems, each "an impl epic" in its own right — e.g. [#1008](/backlog/1008-triage-roadmap-implement-the-12-designed-not-built-standards/), 12 designed-not-built standards). Forcing such an epic to slice straight into `≤3` stories is wrong — it would mean authoring ~N×4 stories at once, most un-investigable (no impl surface yet), the explosion the conservative instinct refuses. **For a roadmap epic, a slice is a *sub-epic*** — carved one level down, each then a future `/slice` candidate in its own right (slicing is recursive, one level at a time: roadmap epic → sub-epics → stories). The tell that you're in this mode, not just under-investigating: each natural child is *clearly* epic-sized (a whole standard / subsystem build), **not** a single story band — if a child fits in `size ≤ 5` it's an ordinary story slice, not a sub-epic. This is the materialization pattern's "discrete homes" step, not a contradiction of it: a sub-epic seeded with its row's **resolved design lineage** is *not* a "detail-less shell" (the thing a roadmap body rightly defers) — the lineage *is* the detail.

### The work-investigation pass — read the real work before proposing any slice

> **A slice boundary is a claim about the code, so you must look at the code before drawing it.** The rubric below demands each slice carry *named file paths* and a *re-estimated size* (conditions 2–3); those are honest only if they came from **investigating the actual work**, not from re-reading the item's narrative. Proposing slices "straight from the body" is guessing where the seams are — the same failure as forming a decision stance cold. So before applying the rubric to a candidate, do the investigation, with the **same rigour the decision path already mandates** (*Selecting the next item* → *Fork-readiness pass*: "go read the relevant code… cite it `file:line`", "never form a stance cold"):

1. **Read the real surface the item touches.** Grep/Explore the subsystem(s) the work lives in — the files, call sites, registries, fixtures it would change or extend. Find where the natural seams *actually* fall in the tree, not where the body's framing implies they fall. (For an unsliced epic, its body's stated decomposition is the *seed*, not the answer — verify each proposed slice against the real code before trusting it.)
2. **Ground every proposed slice in what you read.** Each slice's **named file paths must be `file:line`-citable** from the investigation (the same bar as a decision's concrete-refs check), and its re-estimated `size` must reflect the work you actually saw, not the body's guess. A slice you can't point at real code for is not a slice — it's a hypothesis.
3. **If the work can't be investigated to that depth, that's a result, not a blocker to wave past.** When the surface doesn't exist yet, or the scope is genuinely unknown until a foundational piece lands, the candidate is **could-not-split**; action: *land/investigate foundational slice A first — its artifact then exposes the seams for the rest*. Never paper over an un-investigable item with slices proposed on a guess — that manufactures fake agent-ready work the batch will choke on.

This pass is the operational form of the skill's conservative instinct ("when a clean seam isn't obvious, don't split"): it forces you to go *find* the seam in the code before judging it clean, rather than inferring cleanliness from the prose.

### The split-safety rubric — split only if ALL five hold

> For an **unsliced epic** (candidate kind **b**), condition (1) is already settled *at the parent level* — the epic **is** the decision to decompose — so don't re-litigate whether to split at all. Every slice must still satisfy (2)–(5), and (1) still applies **within** each slice: a slice may not bury its own unresolved fork (if one does, that slice is *could-not-split* pending its decision, even though the epic as a whole is sliceable).
>
> **Sub-epic slices (roadmap / "epic of epics").** When the slices are *sub-epics* (see *Slice granularity* above), conditions (3) and (5) take an **epic-appropriate form** — a sub-epic is not a leaf, so it neither lands `≤3`/`task` nor ships its own demo:
> - **(3′) Each sub-epic is a coherent, independently-ownable scope carrying real seed substance** — a whole standard / subsystem with a *resolved* design lineage (or a clearly-bounded reconcile-scope over partial existing impl), cited from the parent's rows, **not** authored from thin air. A sub-epic with no lineage / no resolved design is an empty shell → don't carve it. The `file:line`-grounding bar of the normal investigation pass relaxes for a *pure-build* sub-epic whose impl surface doesn't exist yet (that's expected — the seams get drawn when the sub-epic is itself `/slice`d after its contract is scoped); but the row's standard/spec must be real and locatable (registry entry, design item, njk).
> - **(5′) Each sub-epic is a legitimate standalone scope, refined in place** — it leaves the *backlog* in a valid state (a real home for future work) rather than a demoable *product* state; the demoable-state test re-applies one level down when the sub-epic is sliced into stories.
> - **(1) still bites hardest here:** a row that is **design-gated** (its build waits on an open decision/design item — carve the build off *that* item, not the roadmap) or hides a **true GAP** (no design *and* no impl — needs a design pass first) is **could-not-split-here**; its unblocking action is *resolve/land the upstream design item, then carve the sub-epic*. So a roadmap epic typically yields a **partial** sub-epic split: carve the fork-free rows, defer the gated ones with their action.

1. **Volume, not uncertainty.** The size comes from the *amount* of independent work, not from an unresolved design fork. **You cannot split away a decision** — if the body holds an open call ("decide whether…", an alternative held open, a `kind: decision` smell), slicing just scatters the same unresolved fork across children. → *could not split*; action: **resolve the decision first** (surface it Tier-B), then re-evaluate.
2. **≥2 nameable slices, each with a real home.** Each slice is either its **own `story`** (standalone, independently valuable) or a **`task` under the parent** (bounded sub-work — guard, wiring, coverage). If you can't name a clean seam where one slice ends and the next begins, it's atomic — don't force it.
3. **Slices land small (agent-ready & batchable).** Each slice re-estimates to `size` ≤ 5 (or is a `task`) with **no buried fork and named file paths grounded in the investigation pass** (`file:line`-citable, not authored from the body) — i.e. it satisfies the Tier-A/batchable gate (see *Running a batch* → *Eligibility*). A "split" that yields another `size·8` lump hasn't split anything; a slice whose named files can't be pointed at in the real tree hasn't been investigated.
4. **Clean DAG with real independence.** The slices form an **acyclic** dependency graph (edges via `blockedBy`) in which **≥2 can proceed independently**, *or* the chain at least unlocks **incremental delivery** (each slice ships valid on its own). A forced linear chain where every slice blocks the next and nothing is usable until the last gains nothing over leaving it whole. → if the only decomposition is a rigid chain, *could not split* unless incremental delivery is genuinely valuable.
5. **No coherence / quality loss — every slice leaves a valid, demoable state.** Per the Definition of Done (AGENTS.md), each shipped feature needs a fixture-driven demo. A slice that leaves the standard in a **partial/broken intermediate state** (half a protocol, half an algorithm, a registry with no consumer) violates this → atomic. Sometimes a **shared fixture authored up front** is what makes the slices independently demoable — record that as the unblocking action.

If any condition fails, the item is **could-not-split** and the report names the **specific action that would unblock a future split**: *resolve decision X*; *land foundational slice A as a standalone story first — its artifact then exposes the seams for the rest*; *author shared fixture F so the slices demo independently*; *re-scope as an unstoried epic and revisit once Y ships*.

**Record the verdict so the board stops re-flagging it — `unsplittableReason`.** An oversized story (`size` > 8) carries a deterministic **split** badge purely from its size; once a `/split` run rules it could-not-split, *clear that flag* by setting **`unsplittableReason`** in its frontmatter (the story-side mirror of an epic's `childlessReason`). It removes the story from the `splittable` set (no split badge, excluded from the "Splittable only" chip + the "· N to split" tally) and shows a muted **"atomic · \<reason\>"** pill instead — *analyzed, not a to-do*. Values name the failing rubric condition + its unblocking action: **`foundational`** (the surface doesn't exist yet — this story *is* the foundational slice; re-run `/split` once it ships), **`atomic`** (genuinely one coherent deliverable, no clean seam), **`fixture`** (needs a shared fixture authored first). A story blocked from splitting by **a buried fork is *not* an `unsplittableReason`** — `undecided` is retired here too (`check:standards` errors on it): carve the fork into its own `kind: decision` item, point the story's `blockedBy` at it, and re-run `/split` once it resolves. `check:standards` validates the value and that it sits only on an open oversized story (`src/_data/backlogMeta.js → unsplittableReasonMeta` is the badge vocabulary). Keep the body note too — the field clears the flag, the note carries the detail. **Don't dishonestly shrink `size`** to dodge the scan; record the real reason instead.

### The report — always produced

Write `reports/<YYYY-MM-DD>-backlog-split-analysis.md` (single-item runs may append/update the same dated report). Two tables:

- **Could split** — one row per candidate: `#NNN` · title · proposed slices (each with its re-estimated `size`/`kind` and a one-line scope) · the slice DAG (which `blockedBy` which) · which slices are independently batchable.
- **Could not split** — one row per candidate: `#NNN` · title · **which rubric condition failed** · the **unblocking action** that would make it splittable later.

Register the report's open questions back in the backlog per *Where an open question goes* — a "could not split, pending decision X" is itself a Tier-B decision worth tracking. The report is the deliverable even when **zero** items are split.

### Executing a split — only after approval, mechanically

Present the proposed slices + DAG for the candidate(s) and get **one "go"** before mutating anything (mirrors the batch plan-approval beat). Then, per item:

1. **Convert the original `NNN` in place → a *storied* epic.** Splice its frontmatter `kind: story` → `epic` and **remove `size`** (children now carry the points — a storied epic must have no size, see *Agile sizing*); refresh its **digest** to an umbrella framing ("Umbrella for …; sliced into #A/#B/#C"). Keep `status: open` and **keep the `NNN`** — never renumber (see *Rules*). *Edge case:* if the original already has a `parent` epic, prefer **not** nesting — keep it a `story` re-sized to its core slice and add the rest as **siblings under the same parent** instead of making a nested epic. *Already an epic (candidate kind b):* there's no `story` → `epic` conversion to do — but still **drop any residual `size`** (a mis-converted epic may have kept its old story points; once it gains sized children, `check:standards` errors that a sized epic double-counts), refresh the digest to umbrella framing only if it isn't already, and proceed straight to step 2, scaffolding the slices as its children.
2. **Scaffold each slice** with `node scripts/backlog.mjs scaffold --kind=story|task [--size=…] --title="…" --parent=<NNN> [--blocked-by=<NNN>,…] --digest="…"` — `--parent` rolls it under the epic, `--blocked-by` lays the DAG edges. Write a real digest per slice (it's the loader's `summary`). **For a sub-epic slice** (roadmap / "epic of epics" mode), scaffold the child as `--kind=epic` with **no `--size`** (an epic carries no points — its own future slices will), and seed its **body with the row's design lineage** so it isn't an empty shell (it's a real future `/slice` candidate, not a leaf). The parent stays an epic either way (kind **b** — no story→epic conversion).
3. **Gate.** Run `npm run check:standards` — it errors on a storied epic that kept a size, an unresolvable `parent`/`blockedBy`, a cyclic edge, or a missing digest, so green proves the split is structurally sound. Confirm the backlog count rose by the number of slices (proves each new file parsed; see *Closing out* step 2 for the hot-reload footgun). Re-evaluate `blockedBy` per *Keep the blocker DAG honest*.

A split **opens** items and (for a story) converts one — it never `resolve`s the original (its scope now lives in the children) and never deletes anything. Report the result as a net-flow line (`+<slices opened>`, original story → epic, or an already-epic left in place) and point `/batch` at the freshly-batchable slices.

## NNN collision — the standard resolution {#nnn-collision}

Two parallel lane sessions can allocate the **same** backlog `NNN` for a *new* item: each computes `max(existing)+1` from its own base view and neither can see the other's not-yet-committed file, so when both land they collide on one number (the **id-storm**). Because `NNN` is immutable for life (see *Rules* → "Never renumber an existing item"), the sanctioned resolution is **"newer yields"**: the *later* of the two colliding **new** items is re-filed to a fresh id — a **refile** (write the new file + delete the old), never a `git mv` (guard-bash blocks a shell renumber) — and **every inbound reference is rewritten** (`#NNN` short-refs, `/backlog/NNN[/-]` URLs, `parent:`/`blockedBy:` edges), so no link dangles. A number **already on the merge base is never yielded** — a base id appearing twice is a real *edit* conflict git already flagged, not an allocation race.

**Demoted to a dormant backstop under JIT numbering (#2291).** With #2288 landed, a *new* item is born hash-keyed (never an `NNN`) and only the drain assigns the real number, serially — so the allocation race this pipeline heals is now structurally unrepresentable pre-land; if the "ids must be unique" gate ever fires post-JIT it signals a JIT-allocation bug, not a routine race (see the tripwire framing on #2248's item). The pipeline below is KEPT as defense-in-depth (nothing thrown away), except the one fire point that ran on every producer land for no reachable benefit — pruned outright (#2291):

**The pipeline (one mechanism, several fire points):**

1. **Producer, pre-commit** — a local `yield` catches a collision the producer can already see before it ever pushes.
2. **The renumber core** — `scripts/backlog/renumber-collisions.mjs` (pure `planRenumber`) + its CLI `scripts/backlog-renumber-collisions.mjs` (#2071): decide the yielder, allocate the free id, rewrite refs. The shared helper `scripts/lib/nnn-collision-heal.mjs` (#2222) wraps it for the merge-time case, allocating a **free GAP id below the `max+1` frontier** (so a concurrent scaffold, which takes `max+1`, can't immediately re-collide).
3. ~~`pr-land.mjs` — pre-check heal (#2222).~~ **Pruned (#2291).** pr-land no longer merges by default (#2290 — the drain is the sole writer), so healing the lane tip here, before a PR the drain would separately merge (and separately precheck-heal at fire point 5), was dead wiring: it could only ever detect "none" once new items stopped carrying an `NNN` pre-land.
4. **`pr-land.mjs` — post-land heal (#2071/#2213).** Retained: the `--fallback-git` break-glass direct-merge path is still a real write to `main`, so it still re-checks post-merge `main` and heals any residual collision, non-destructively, never failing the land.
5. **`/merge` + `/drain` (`scripts/merge-ai-prs.mjs`) — the lander (#2222/#2276), retained as the drain's own dormant backstop (#2291).** Before merging, a certified-but-not-landable PR is healed: a collision-**red** tip is renumbered + rebuilt on its own, and a **stale-green + BEHIND** tip is renumbered **inside the rebase-drop rebuild** (`scripts/lib/rebase-drop-manifest.mjs` with `healCollision`) — one rebuilt tip both drops the transient `.lane-manifest.json` **and** clears the id dup, so it passes `check:standards` and merges. This closes the serial-route deadlock where the post-merge heal could never run because branch protection blocked the merge on the very check the dup made red.
6. **The backstop** — guard-bash's renumber block (no illegal shell renumber) + `scripts/check-standards.mjs` "ids must be unique" gate, now framed as a **tripwire** (#2248/#2291): under JIT numbering this should never fire, and a fire is an allocation-bug alert, not a routine heal-trigger.

**Where it does NOT auto-fire (resolve by hand):** a base-inherited duplicate (a real edit conflict — *never* auto-yielded); a hand-authored file that never rides a lane PR (no transport runs the heal — settle its number before committing); and the rare **dense-range** case where no gap exists below `max`, where the allocator falls back to the `max+1` frontier (correct, but a concurrent scaffold could still race it — the post-land/gate backstop catches that).

## Rules

- **Land each finished piece via a ready-to-merge PR — never commit to `main`.** (#2183/#2190, implementing #2123.) Every edit-action — a solo `/next` build, a serial `/batch` item, a `/slice` write, `/workflow`, `/pr` — runs in an **isolated lane CLONE** (`node scripts/lane-pool.mjs` — own HEAD, guard-immune) and lands via a **ready-to-merge PR**, never a direct commit to `main`. `main` only ever advances when a drain (`/merge` or `/drain`) merges that PR; then local `main` pulls. The core simplification: a **stable live-preview `main`** that changes only on merge, never churned mid-edit.
  - **The arc per item, in its lane clone:** reset the clone to `origin/main` → `claim` → work → green gate → `resolve` → **commit that item's changes** (one commit per item; **stage only the files that piece touched** — explicit paths, never `git add -A`) → `node scripts/pr-land.mjs --ref=lane/<slug>-<NNN> --no-wait` (opens the PR, does *not* merge) → label it **`ready-to-merge`**. Then STOP. **Never `git push` to `main`, never merge your own PR inline** — a separate optional drain lands it (you are correct with no drain running).
  - **The claim/ownership signal.** `claim` writes the central `claims.json` in the primary (a **local, uncommitted** signal `readiness --select` reads offline, Rule #105) so the *same checkout* won't re-offer the item; the `open→active→resolved` backlog flip rides the item's PR (so `main` stays clean). Cross-checkout double-claim across separate primaries is a documented residual (#2190) — the reserve/soft-hold (`backlog.mjs reserve`) is the interim cross-session courtesy.
  - **A lane clone is single-session, so the shared-checkout stash dance is no longer needed** — your clone is yours; stage your explicit paths and commit. End commit messages with the standard `Co-Authored-By` trailer. (Historic: before #2123/#2183 solo work committed directly to shared `main` and the guard blocked pushing; that stance is superseded — see #104.)
  - **Rebuilding a lane onto an advanced `main` — reset + cherry-pick, NEVER `git rebase`.** When `main` moves under an open lane PR (branch protection's strict-up-to-date wall, or a resolvable id collision to `yield` past), rebuild the lane with **`git reset --hard origin/main` then `git cherry-pick <your-commit>`** — deterministic, and immune to the footgun that a lane clone's local branch is itself named `main`: `git rebase origin/main` there resets HEAD to `origin/main` and **silently drops your commit** (observed 2026-07-04, which then blanked a PR — see the next rule). Better still, **prefer letting the drain's `rebase-drop` (#2198) rebuild a behind PR** over doing it by hand at all; reserve `git rebase` for genuine interactive history-editing, never routine "get up to date."
  - **Never force-push a `lane/*` ref without verifying HEAD still carries your work.** The force-push is the destructive step: confirm **`git diff origin/main..HEAD` is non-empty** (your commit SHA shows in `git log`) *before* pushing. A force-push whose head is **empty-diff vs `origin/main` blanks the PR** — GitHub sees no diff from base and auto-closes it (exactly how the dropped-commit rebase above lost a PR). **`--force-with-lease` does NOT protect against this** — the lease guards against a *concurrent remote* change to the ref, not against pushing the *wrong local state*. (Recovery, if it happens: the commit is still in the clone's `git reflog` — `git cat-file -t <sha>`, then `reset --hard origin/main` + `cherry-pick <sha>` + re-push + `gh pr reopen`.)
- **Review before adding (dedup).** Always scan the existing backlog first — list the titles and grep related terms (`grep -rilE "<topic>" backlog/`). If an item already covers the idea, **extend it** rather than adding a near-duplicate sibling. Watch for *parallel* tracks that look similar but are distinct (e.g. the `<component>` adapter items vs. the JSX adapter items) — cross-reference instead of merging.
- The `id` is the filename stem `<NNN>-<slug>` — unique by construction. The `NNN` number (`item.num`) is the stable routing key and short ref (`#042`); the `slug` is reword-able text. `kind` and `status` come from the enums above (validated, incl. the `NNN-` prefix + number-collision checks).
- **Never renumber an existing item.** The `NNN` is immutable for life — don't rename `backlog/<NNN>-…` to a different number to tidy the sequence, close a gap, or dodge a collision. It breaks `#NNN` refs and `/backlog/<NNN>/` URLs and, under concurrent agents, cascades into id collisions. On a collision the *newer* item takes a different free number; the number already on disk is never touched. (Slug rewording is fine — see *Renaming an item*.)
- **A decision and an epic are separate items — never make one serve as both.** Since #466/#487 collapsed the schema to a single `kind`, an item is *literally* either `kind: decision` or `kind: epic` (the old `kind: decision` + `kind: epic` conflation is now structurally impossible — the single axis can't hold both). The rule it enforced still stands, because the two roles have *opposite* lifecycles: a **decision resolves the moment it's ratified**, but an **epic must stay `open` until its last slice lands** (and `check:standards` **errors** on a `resolved` epic with an open child). When a program needs *both* a design call **and** a build umbrella, author **two items**: a **`kind: decision`** item holding the fork(s) → it `resolve`s on ratification; and a separate **`kind: epic`** umbrella → it stays `open` until its build slices resolve. Link them (`parent`, or the epic `blockedBy` the decision while it's open). This is the *Decisions Are Work Items* rule applied at the umbrella level: same reason a fork never lives inline in a build body (see *the fork-flip-or-carve rule* in *The drop-reason classifier*). Worked example (pre-migration framing): [#382](/backlog/382-design-reference-screenshot-corpus-collect-dedup-codify/) split → decision [#581](/backlog/581-design-ref-corpus-pipeline-shape-subject-classification-fork/) + the open epic #382, 2026-06-14. **The split is now structurally impossible, but the *implied* conflation still slips in: an epic whose body narrates an open fork ("decide whether…", "TBD which direction"), or that encodes a pending design call as `childlessReason: undecided`, is burying a decision just as surely.** Carve it the same way — a separate `kind: decision` the epic `blockedBy`-depends on while open. **`childlessReason: undecided` is retired entirely (`check:standards` errors on it)** — an epic never holds its decision inline as a reason value; the decision is its own item, the dependency is a `blockedBy` edge, and "slices not yet scoped" is the **unsliced** state (no `childlessReason` at all → the *slice* badge), not a reason. At authoring time, write the epic to *depend on* its decision item from the start rather than carving the fork out later. Worked examples (both already split correctly): the blocks-side [#1245](/backlog/1245-reference-runtime-blocks-router-navigation-are-duplicated-an/) → decision [#1246](/backlog/1246-canonical-home-for-the-reference-runtime-stay-subset-blocks-/), and the plugs-side [#1250](/backlog/1250-re-reconcile-fui-plugs-up-to-we-contract-anchored-add-a-real/) → decision [#1270](/backlog/1270-reconcile-fui-plugs-up-to-we-contract-anchored-not-repoint-d/), 2026-06-20.
- **No `title`/`summary` in frontmatter** — title = the body's `# H1`, summary = its first paragraph (or, for a pointer item, both come from the report). The loader derives them; the validator checks the *derived* values exist, so every item needs either a body or a `relatedReport`.
- **Body-link hygiene — a backlog body renders at `/backlog/<id>/`, so write links a *reader* can follow.** `[[wiki-link]]` is **memory-files-only** (the `~/.claude/.../memory/` convention); it has **no place in a backlog/standards body** — markdown renders it as literal text and the slug has no page. To reference another item, use its rendered URL `[#NNN](/backlog/NNN-slug/)`, never `[[…]]` or a repo-relative `…/NNN-slug.md` link (which 404s on the site). Don't put `localhost`/absolute-`/Users/` links in a body. Repo-relative refs to `reports/` and `docs/agent/` `.md` files (and `src/…` source paths) are the sanctioned agent-facing convention and are fine. `check:standards` errors on a body `[[…]]` and warns on the dead-link forms (lint 6d-sexies).
- `relatedReport` must exist on disk; `relatedProject` must resolve in `projects.json`; `crossRef` needs both `url` and `label`.
- `reports/` is **not** in the 11ty build, so `relatedReport` shows as a path label, not a link. Real site links go via `crossRef` / `relatedProject`.
- `npm run check:standards` validates all of the above — it reads `backlog/*.md` through `src/_data/backlog.js`, the same loader the site uses.
