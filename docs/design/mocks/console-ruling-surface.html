F1

Review-unit vs build-unit

forced-invariant fork

Is the thing a human launches/reviews the same granularity the agent builds? This fixes the runner contract and what a lane holds.

recommended
(a) Decouple: review-unit = cluster, build-unit = item

The human reviews + clears the whole blockedBy/parent closure as one thing; “Launch” decomposes into per-item builds in dependency order, each its own agent-run + PR. Runner keeps pulling one next-to-build item, WIP=1.

excluded
(b) The cluster is the atomic build unit

One agent-run builds the whole closure in a single turn. Breaks WIP=1; collapses per-item review + per-item proof into one un-reviewable, un-bounceable mega-diff.

Why (a)

Review granularity (cluster) and build granularity (item) are different by design, and the existing runner already builds the item, not the cluster — (a) keeps the ratified WIP=1 invariant while still giving a whole-cluster review. Decouple-and-compose is the standing bias.

Counter-argumentSURVIVES + amended

Skeptic pushed “atomic cluster-build ships a coherent feature in one shot.” Refuted on merit: coherence is a review property (satisfied by cluster-level review), not a build property; “fewer lanes” is a throughput claim, not merit. Amendment folded: struck a false “one lane = one item = one PR” identity — a lane may hold several sequential single-item builds. Lane packaging + wave scheduling deferred to #2557/#2560.

Downstreamrunner contract — “Launch a cluster” = enqueue its ready frontier as ordered per-item builds.
F2

Where confidence comes from

representation fork

The launch gate reads “confidence ≥ threshold,” but no such field exists today. Without a trustworthy source the gate is decoration.

recommended
(a) A stored, provenance-tagged field

Agent seeds it at prep · human may override · the proof backend refines it. Records by ∈ agent|human|derived as the trust tier, with nearest-wins precedence human > derived > agent — a proof recompute never silently overwrites a human override.

excluded
(b) Purely derived, never stored

The gate computes confidence live each render. Can’t carry a human override or a stable prep-time estimate — the human-curation thesis becomes unenforceable and the number is unauditable.

Why (a)

Only a stored, provenance-tagged field with a precedence rule lets all three sources compose without drift, keeps the gate’s input auditable + human-authoritative, and degrades honestly: before #2562 exists it carries by: agent (visibly a self-estimate), rendered at its true trust tier, not false certainty.

Counter-argumentSURVIVES + amended

Skeptic added the missing precedence rule (nearest-wins) and struck a timing crutch; statute-reconciled to #single-authoring-sot + #config-extends-platform-default.

Downstreamlaunch gate + the ascending-confidence review sort read the resolved value; #2562 plugs into the derived layer.
F3

Structured spec fields vs prose

either/or fork

The proof rows R1..Rn need a data model. Are requirements structured (machine-checkable) or prose the agent re-interprets? This is the schema the proof backend + review modal consume.

recommended
(a) Frozen structured requirement rows

Human-reviewed structured rows are the machine-checkable contract, seeded from prose (the backlog stays markdown-authored — not a hand-typed second home). Each row has a stable id the proof backend keys on.

excluded
(b) Requirements stay prose

The agent re-interprets acceptance prose per build. Not machine-diffable — you can’t map a proof row to a requirement, and drift is invisible.

Why (a)

Realizes the same-day #2564 “the spec is a schema/executable contract, not prose” at the item layer, while honoring #faithful-derivation (prose→rows is a lossy seed) and #single-authoring-sot (no second authoring home). Adds a round-trip invariant.

Counter-argumentSURVIVES + amended

Reframed against the same-day #2564 schema-not-prose direction; corrected the citation to #faithful-derivation-exclude-not-fabricate as governing; added the round-trip invariant. Seam flag: #2564 has its own open forks — this reconciles the framing, doesn’t rule them.

Downstreamproof schema — #2562 + the review modal key on the frozen row ids; rides #2558’s seam.

forced-invariant fork

The build-time constitution the agent must honor — one artifact, or an index into cluster rules — and how it’s injected into the agent’s context.

recommended
(a) A machine-readable index of inherited statute anchors

No new authoring home: an index over platform-decisions.md + the WE standards + AGENTS.md, injected as a build-time derived compact projection (regenerated → drift-free).

Fork 1

DAG frontier render rule

fork · what folds

At 500 items the board can draw ~30–50 nodes legibly. The frontier rule picks which. (Rendering everything flat is the broken branch — the board must fold.)

recommended
(a) State-based frontier

A node renders expanded iff it’s active (in-flight lane), buildQueued (next-cleared), an open gate (a prepared/blocked decision), or within one hop of one. Everything else folds to a summary band with an explicit “+N folded”.

excluded
(b) k-hop graph-distance frontier

Render every node within k edges of any active node. Floods the board with inactive neighborhoods while burying a distant but gated node you must act on. Distance ≠ attention.

Why (a)

Maps the board to what you’re actually working; a cheap O(1) predicate over already-derived fields (no per-render BFS); degrades honestly (nothing active → summary bands, the correct “nothing to see” signal).

// state-based frontier — a pure predicate over the loader's derived graph
const isActiveNode = (it) => it.status==='active' || it.buildQueued
  || it.tier==='B' /* ready gate */ || (it.kind==='decision' && it.openBlockers.length>0);
Counter-argumentSURVIVES + amended

Skeptic caught a load-bearing dead-code predicate: the first-draft tier==='B' && openBlockers>0 is unsatisfiable (tier B already means cleared blockers) — it would drop every prepared decision from the frontier. Fixed in the sample above; the ruling (attention beats distance) is unchanged. The build must ship the corrected predicate.

Downstreamsets what board slices C1/C5 draw. The LOD/SemanticZoom intent mint stays #2533 F3’s call.
Fork 2

Cleared-set ordering

forced-invariant ratify

Once a set is cleared for build, in what order does it run? (Framed as FIFO vs score vs pinned — but grounding dissolves that: those three are already composed in the shipped sort, not rivals.)

recommended
(a) Inherit the ratified #2526 order, filtered

The cleared set runs in the composite tier → score → rank → FIFO → num order, restricted to buildQueued. “Reorder the cleared view” = edit tier/rank. One ordering source.

excluded
(b) A separate cleared-only order

Clearing stamps a distinct signal (clear-time FIFO / a clearedRank) that can diverge from the score order — re-opening the exact two-knobs footgun #2526 closed.

Why (a)

A second cleared-only order would be a third ordering surface that can contradict the first (scored #1 but dragged to the back — which wins?). buildQueued stays a safety gate, never an ordering axis. The recommended path is verbatim the shipped nextToBuild.

Counter-argumentSURVIVES + amended

Reframed from a “weigh” to a forced-invariant ratify (the rival second field is one nobody proposed). The fresh screen independently called this “the strongest, most legitimate fork.” The ruling is still owed because the board build needs the explicit “no cleared-only field” cite.

Downstreamschema — the board must not add a cleared-only order field.
Fork 3

Wave vs whole-cluster launch

fork · launch mechanics

Does one launch action commit the whole cluster (engine self-serializes), or does the operator drive each wave by hand? (A mandatory human gate between every item is the broken branch.)

recommended
(a) Whole-cluster launch: plan + pour the frontier

One action clears the cluster and pours its ready frontier (wave 0) off the derived, editable wave plan (research R9 — show the waves and let the human edit). WIP is a policy value lanes ≤ N.

excluded
(b) Item-by-item, no cluster plan

The operator assembles + launches each item by hand. Discards the derived-wave structure and re-serializes throughput on human effort — the lane board’s core value never materializes.

Why (a)

The wave derivation already exists (scheduleWaves / readyAfter). Kiro’s mistake was deriving waves but not letting the human edit — we take the waves and the edit.

Counter-argumentSURVIVES + amended

Skeptic caught a seam overreach: the first draft ruled “wave k+1 auto-dispatches,” which decides later-wave consent timing — #2561 F1’s per-item call — and contradicts the design’s own “consent stays per-item.” Narrowed to rule only launch granularity; punts auto-vs-confirm to #2561 F1.

Downstreamfeeds #2560’s lease acquisition at each wave dispatch; aligns with #2561 F1 (unit), doesn’t duplicate it.
Fork 4

Repo / program dimension on lanes

fork · namespace

Are lanes scoped per-program, or one global fleet across the constellation? (The physical pool is already global, keyed by repo, with fungible clones.)

recommended
(a) One global fleet, program = a projection

A single fleet; a “program board” is a filter/view over it (program chip + per-program share). Scope leases are global — a lease keeps conflicting work out of any lane, cross-program.

excluded
(b) Per-program lane pools

Each program owns its own lane set. Hides cross-program contention (two programs touch the same file and silently collide at the drain) and fights the repo-keyed pool a program spans.

Why (a)

Contention is physical (same files → drain conflict), and files don’t respect program boundaries. Only a global lease namespace makes cross-program collisions visible — the lane board’s core novelty. You still get the per-program board: it’s a filter, not a partition.

Counter-argumentSURVIVES

Skeptic tried “a per-program board is what the user opens, so lanes should be per-program.” Held by separating view from namespace: (a) fully supports a per-program board as a filter while keeping the lease namespace global. The default is overdetermined by the repo-keyed substrate — (b) invents a key the pool lacks.

Downstreamthe #2560 lease namespace is global; holds the multi-backlog north-star open.

1

Verb vocabulary — the Prepare / Re-prepare / Revive family

Three near-synonyms for “make it runnable again,” and Re-prepare maps to 3 different causes (confidence-low / decision-pending / unprepared).

Recommendation

Keep three cause-specific verbs, as the taxonomy already assigns: Prepare (unprepared, off-lane) · Re-prepare (gated-in-queue) · Revive (parked). One verb per cause beats one overloaded verb — the label states which action the state needs.

2

Verb affordance — icon+label vs label-only

The ratified icon grammar says verbs are always icon+label; the mock went label-only after the icon sweep.

Recommendation

Reconcile to icon+label — the ratified rule wins; the label-only mock state was a regression, not a new decision.

3

Attention emphasis — running purple outline vs you-act amber edge

Do the running-card full purple outline and the you-act amber left edge compete at scan distance?

Recommendation

Scope each signal. Amber left edge = needs-you (the one attention signal). Running = a run-state, not an attention state — demote the purple full outline to a quieter fill/accent so it never competes with amber for “act here.” Honors say-it-once.

4

Header glyph chip (“lanes 2⟳·1‖·1⚠”)

It restates the readable stat-chip row in a cryptic glyph form a few hundred px away.

Recommendation

Drop it — redundant with the readable stat row; the design doc already flagged it as a follow-up removal.

5

AGENT / NOBODY badges on board cards

Render explicit who-acts badges, or let form carry it?

Recommendation

No badges. The amber edge (you-act) + the inert dashed/grayed form (nobody/agent) already encode who-acts — say-it-once. Badges would be a second encoding of the same fact.

6

fan-out vs forked encoding

One status with a state chip (“may split ×3” → “split ×3”), or two distinct states?

Recommendation

Two distinct states — matches the shipped taxonomy (UC-B4 ∥ fan-out-may-fork vs UC-B5 forked-running). “May fork” and “forked” are genuinely different lifecycle points, not one state with a chip.

7

Icon differentiation polish (group)

Filled-vs-stroke coherence in the amber family · a you-act corner-slot · unify “gated”’s three encodings · scales-glyph at 14px · run/drain silhouette similarity · dark-theme idle-circle contrast.

Recommendation

Adopt the polish set as a batch — all refinements within the ratified icon grammar; no rule reopened. Land as one pass at port time.

8

Two-operator presence model

seam · #2533

The optimistic verb-lock + “claimed by ‹name›” chip — presence/lock semantics before the concurrency slice builds.

Recommendation

Adopt optimistic verb-lock + a “claimed by ‹name›” chip as the default presence model, but confirm the collaborative-claim semantics against #2533 (where the question is parked) before the concurrency slice (#2555 C6) builds.
